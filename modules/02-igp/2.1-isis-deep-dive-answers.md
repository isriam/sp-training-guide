# 2.1 — IS-IS Deep Dive — Answer Key

## Question 1
> A new router joins the network but its IS-IS adjacency stays in `Init`. The link is up and you can ping the neighbor. What are the three most likely causes?

### Answer

The adjacency is stuck in `Init` — meaning this router sees the neighbor's IIH (Hello) packets, but the neighbor isn't listing this router's system ID in its IIH. The three-way handshake (RFC 5303) can't complete because the remote side doesn't acknowledge us. Ping works because IP connectivity exists at L3, but IS-IS runs at L2 and has its own adjacency requirements independent of IP reachability.

**1. MTU mismatch on the link.** IS-IS PDUs are not fragmented (they ride directly on L2, no IP layer to fragment). If one side has a 9000-byte MTU and the other has 1500, large IIH or LSP PDUs from the high-MTU side get silently dropped. The low-MTU side never sees the neighbor's Hellos consistently, so it never transitions past `Init`. Diagnose with `show isis interface` on both sides — compare MTU values. Fix by matching MTU or adjusting IS-IS padding (`hello-padding disable` only helps after adjacency forms; during initial bring-up, padding tests MTU compatibility).

```
IOS-XR:  show isis interface GigabitEthernet0/0/0/0
Junos:   show isis interface ge-0/0/0.0 detail
```

**2. Authentication mismatch.** If one side has IS-IS authentication configured (MD5 or keychain) and the other doesn't — or they have different keys — the receiving router silently discards IIH packets that fail authentication. The sending router sees its own IIH going out but never gets acknowledged. Check `show isis interface` for authentication configuration, and use `debug isis adj-packets` (IOS-XR) or `monitor traffic interface` (Junos) to confirm packets are arriving but being dropped.

**3. Level mismatch.** If the new router is configured as `is-type level-1-only` but the neighbor is `level-2-only`, they won't form an adjacency because there's no common level. IS-IS requires at least one matching level between neighbors. This is subtle because the IIH packets are still exchanged (L1 IIH on one side, L2 IIH on the other), but neither side sees a valid neighbor at its configured level. Verify with `show isis interface` — check `Circuit Type` on both ends.

**Bonus consideration:** If this is a broadcast segment (not P2P), also check that the network type matches. A P2P-configured interface won't form adjacency with a broadcast-configured interface — the IIH formats differ (P2P IIH vs LAN IIH).

## Question 2
> You're designing a national backbone with 200 routers. Would you use L1/L2 hierarchy or flat L2? Justify your answer with convergence and scalability trade-offs.

### Answer

**Flat L2 for the core and aggregation layer, with L1/L2 hierarchy only if you have a large access tier that benefits from route summarization.**

For a 200-router national backbone, flat L2 is the right default. Here's why:

**Convergence argument for flat L2:** Every router has the complete L2 topology database. SPF computation covers the entire domain, which means every router independently computes optimal paths to every destination. There's no dependency on ABR-style summarization or default route injection. When a link fails, every router recomputes and converges simultaneously. With L1/L2 hierarchy, L1 routers depend on L1/L2 border routers for inter-area reachability — adding a dependency chain and potentially suboptimal routing (L1 routers may not know the true cost to destinations in other areas).

**Scalability argument:** 200 nodes is well within IS-IS L2 flat scaling limits. Modern implementations handle 1,000+ nodes in a single level without issues. The LSDB for 200 routers is modest — each router generates one LSP fragment (maybe two if heavily connected), so you're looking at ~200-400 LSPs. SPF computation over this topology takes single-digit milliseconds on modern hardware. There's no scalability pressure to add hierarchy.

**When L1/L2 hierarchy makes sense at this scale:** If 150 of those 200 routers are access/PE devices behind a smaller aggregation layer, you could put the access PEs in L1 areas to contain their link-state churn. The 50 aggregation and core routers run L2, and the L1/L2 borders summarize access prefixes into L2. This reduces LSDB size on core routers and limits the blast radius of access-layer flaps. But this is an optimization, not a requirement.

**Recommendation:** Start flat L2. Monitor LSDB size and SPF frequency. If access-layer instability starts causing unnecessary SPF runs on core routers, introduce L1 areas at the access edge. Don't pre-optimize for a problem you don't have yet.

## Question 3
> During a maintenance window, you set the overload bit on a P router. A customer reports that traffic to a prefix directly connected to that router is now unreachable. Why?

### Answer

This is a classic overload bit gotcha that catches even experienced engineers.

**The overload (OL) bit tells other routers: "Don't use me as a transit node."** When a router sets OL in its LSP header, other routers run SPF and treat all transit links through that router as having infinite metric — effectively excluding it from any shortest-path-first transit calculations. Traffic that would normally traverse this router takes alternate paths.

**However, directly connected prefixes should still be reachable.** The OL bit only affects transit — other routers should still be able to route to prefixes that originate from the overloaded router itself. The IS-IS specification (RFC 1195, clarified by RFC 3787) says that when OL is set, other routers must still consider the overloaded router's attached prefixes (advertised in TLV 135/TLV 236) — they just won't use the router as a waypoint to reach destinations beyond it.

**So why is the prefix unreachable?** The most likely cause is that the implementation is setting the metric of all attached prefixes to `max-metric` (2^24 - 2 = 16777214) when the overload bit is set, which some implementations do by default. If the customer's traffic has to traverse multiple hops to reach this prefix, and the path through the overloaded router now has an astronomical metric, an alternate path with a lower total metric may not exist — making the prefix effectively unreachable.

On IOS-XR, `set-overload-bit` with the `on-startup` option has additional modifiers:
```
set-overload-bit on-startup wait-for-bgp
  advertise {connected | prefix}    ← controls which prefixes remain reachable
```

On Junos:
```
protocols isis {
    overload {
        advertise-high-metrics;    ← sets metrics to max instead of full OL
        allow-route-leaking;       ← still allows inter-level route leaking
    }
}
```

**The fix:** Use `advertise connected` (IOS-XR) or check that your implementation doesn't apply max-metric to attached prefixes when OL is set. Alternatively, use `advertise-high-metrics` on Junos instead of full overload — this sets transit metrics to maximum but keeps directly connected prefix metrics at their configured values, ensuring reachability while still draining transit traffic.
