# 2.4 — IS-IS vs OSPF: The Real-World Decision Framework — Answer Key

## Question 1
> You're designing a greenfield 200-node SP backbone with plans for SRv6 in 18 months. A senior engineer argues for OSPF because "the team knows it." How do you make the case for IS-IS? What specific technical advantages matter for this use case?

### Answer

Respect the experience, but make the technical case clearly. This is a greenfield build — the one time you get to choose without migration costs. "The team knows OSPF" is a training problem, not an architecture decision.

**The SRv6 argument is decisive.** SRv6 encodes forwarding instructions in IPv6 extension headers using SRv6 SIDs (128-bit locators). IS-IS carries SRv6 information natively via TLV extensions (RFC 9352) — SRv6 Locator TLV, SRv6 End SID sub-TLV, SRv6 SID Structure sub-TLV. These are mature, widely implemented across IOS-XR, Junos, and SR-OS, and are the basis of every vendor's SRv6 reference architecture. OSPFv3 SRv6 extensions exist (RFC 9513) but are less mature, less tested in multi-vendor environments, and require running OSPFv3 (not OSPFv2) since SRv6 is inherently IPv6. If the team "knows OSPF" they likely know OSPFv2, not OSPFv3 — so the familiarity argument partially collapses.

**Extensibility for future technologies.** IS-IS's TLV architecture means any new capability is an additive TLV — no protocol version change, no new LSA types, no backward compatibility breaks. Flex-Algo, SRv6 Network Programming, Application-Specific Networking — all arrived as TLV extensions. OSPF's fixed LSA structure requires opaque LSA workarounds or entirely new LSA types, with more complex backward compatibility testing.

**Operational simplicity at 200 nodes.** IS-IS L2-only flat domain for a 200-node backbone is one level, one LSDB, no area types to configure. No stub/NSSA/totally-stubby decisions. No Area 0 backbone dependency. No virtual links if your topology doesn't naturally provide Area 0 contiguity. Compare to OSPF: you'd need multi-area design (200 nodes in Area 0 is feasible but pushes limits), ABR summarization planning, area type selection for each non-backbone area.

**Multi-vendor consistency.** IS-IS behavior is remarkably consistent across IOS-XR, Junos, SR-OS, and EOS. OSPF has more vendor-specific behavior variations — NSSA Type 7→5 translation, forwarding address handling, opaque LSA flooding scope, and OSPFv3 AF support vary across implementations. For a 200-node backbone that's likely multi-vendor, IS-IS reduces interop risk.

**The training investment.** An experienced OSPF engineer learns IS-IS in 2-3 weeks. The protocol concepts are identical (link-state, SPF, flooding). The differences are addressing (NET vs Router ID), hierarchy (L1/L2 vs areas), and extensibility (TLV vs LSA). This is a one-time cost. The cost of choosing OSPF and then migrating to IS-IS in 18 months when SRv6 deployment begins is orders of magnitude higher — 200 nodes of ships-in-the-night migration versus 2 weeks of training.

## Question 2
> A regional ISP (50 nodes) runs OSPF with 3 areas. They want to add Segment Routing. Should they migrate to IS-IS or add SR to their existing OSPF? What factors drive this decision?

### Answer

**This one genuinely depends. Both paths are technically viable.**

**Case for adding SR to existing OSPF:**

SR-MPLS works with OSPF. Prefix SIDs are advertised via Opaque LSAs (Type 10, area scope). Adjacency SIDs, Flex-Algo, and TI-LFA all work with OSPF. At 50 nodes with 3 areas, the OSPF deployment is small enough that the opaque LSA overhead is negligible.

Key factors favoring "stay OSPF":
- **50 nodes is small.** No scalability pressure from OSPF at this size.
- **3 areas means existing summarization.** If the area design is clean and working, why disrupt it?
- **Migration cost is real.** Ships-in-the-night across 50 nodes is 2-4 weeks of change windows, validation, and risk. If the ISP has a small team and limited maintenance windows, this cost matters.
- **SR-MPLS only (no SRv6 plans).** If they're deploying SR-MPLS but have no SRv6 roadmap, OSPF SR support is fully adequate.

**Case for migrating to IS-IS:**

- **SRv6 is on the roadmap.** If they plan to evolve to SRv6 within 2-3 years, migrating to IS-IS now avoids a second migration later. One disruption instead of two.
- **Multi-topology needed.** If IPv4 and IPv6 topologies need to differ (common during dual-stack deployment), IS-IS MT is cleaner than running separate OSPF/OSPFv3 processes.
- **Multi-vendor expansion.** If the ISP plans to add vendors (e.g., currently Cisco-only but evaluating Juniper or Nokia for new POPs), IS-IS interop is more predictable.
- **Team skill investment.** Regional ISPs grow. Engineers who learn IS-IS now are prepared for larger-scale deployments.

**Decision framework for this specific case:**

| Factor | Weight | OSPF+SR | Migrate to IS-IS |
|--------|--------|---------|-------------------|
| SRv6 planned? | High | ✗ | ✓ |
| Multi-vendor? | Medium | ✗ | ✓ |
| Small team / limited windows? | Medium | ✓ | ✗ |
| Working OSPF design? | Medium | ✓ | ✗ |
| SR-MPLS only? | Medium | ✓ | — |

**My recommendation:** If SR-MPLS only and no SRv6 plans, add SR to OSPF. If SRv6 is anywhere on the 3-year roadmap, migrate to IS-IS now and deploy SR on IS-IS from the start. Don't pay the migration tax twice.

## Question 3
> During an OSPF-to-IS-IS migration, you notice that 5 out of 200 routers still show OSPF routes as active for their loopback prefixes, even after lowering IS-IS distance to 105. What are the most likely causes?

### Answer

IS-IS distance is 105, OSPF internal is 110. IS-IS should win on every router. Five holdouts means something is overriding the normal preference on those specific nodes.

**Cause 1: Manual distance/preference override on those 5 routers.**

Someone (possibly from a previous maintenance window, a template error, or a one-off troubleshooting session) set a manual OSPF distance lower than 105 on those routers.

```
IOS-XR: show run router ospf | include distance
        ← look for: distance 100 (or anything < 105)
Junos:  show configuration protocols ospf | match preference
        ← look for: preference 8 (or anything < IS-IS preference)
```

This is the #1 cause and exactly what the War Story in this section describes. A config audit across all 200 nodes — `grep` for `distance` or `preference` overrides — would have caught this before Phase 3.

**Cause 2: IS-IS not advertising those 5 routers' loopback prefixes.**

If the loopback interfaces on those 5 routers aren't configured under `router isis` (IOS-XR) or `protocols isis` (Junos), IS-IS has no route for those prefixes. OSPF wins by default because it's the only protocol with a route.

```
IOS-XR: show isis interface brief    ← is Loopback0 listed?
        show isis route <loopback-prefix>
Junos:  show isis interface          ← is lo0.0 listed?
        show route <loopback-prefix> protocol isis
```

Fix: Add the loopback interface to the IS-IS configuration on those 5 routers.

**Cause 3: IS-IS adjacency not formed on those routers.**

If IS-IS adjacencies aren't up on those 5 routers (authentication mismatch, level mismatch, interface not enabled), IS-IS has no reachability to or from them. Check `show isis adjacency` on those routers — if they have zero IS-IS neighbors, that's the problem.

**Cause 4: Connected route or static route with lower distance.**

If those loopback prefixes are also present as connected routes (they always are for the local router) or static routes with a lower distance, those would be preferred over both IS-IS and OSPF. This wouldn't normally cause OSPF to show as "active" though — connected routes would show instead. But if there's a redistributed connected route in OSPF with a modified distance, the interaction could produce unexpected results.

**Investigation approach:**
1. On one of the 5 routers: `show route <its-loopback> detail` — what protocol is active and why?
2. Check IS-IS specifically: `show isis route <prefix>` — does IS-IS even have the route?
3. Check for config anomalies: `show run | include distance|preference` on all 5
4. Compare those 5 routers' configs against a known-good router — diff them

## Question 4
> You're planning a ships-in-the-night migration for a 100-node network. What is your go/no-go checklist before executing Phase 3 (traffic cutover to IS-IS)?

### Answer

Phase 3 is the point of no return — you're shifting production traffic from OSPF to IS-IS. Everything before this is reversible with zero impact. After this, a problem means customer-facing impact. The checklist must be exhaustive.

### Go/No-Go Checklist

**Topology & Adjacency Validation:**
- [ ] IS-IS adjacency count matches expected count on every node (compare against OSPF neighbor count per node)
- [ ] All adjacencies are `Up` with stable uptime (no recent flaps in last 24 hours)
- [ ] IS-IS LSDB node count matches total network size (100 LSPs in L2 database on every router)
- [ ] No IS-IS overload bit set on any node (`show isis database | include OL`)

**Route Parity:**
- [ ] Every prefix reachable via OSPF is also reachable via IS-IS (automated script: compare `show ospf route` vs `show isis route` on every node)
- [ ] Loopback prefixes for all 100 nodes present in IS-IS routing table on every node
- [ ] BGP next-hops are resolvable via IS-IS (critical for MPLS VPN traffic)
- [ ] Metric sanity: IS-IS metrics produce the same traffic engineering intent as OSPF costs (preferred paths match)

**MPLS/SR Validation:**
- [ ] If running SR: all prefix-SIDs present in IS-IS LSDB (`show isis segment-routing label table`)
- [ ] If running LDP: LDP sessions are bound to IS-IS (not OSPF) for label distribution
- [ ] MPLS forwarding verified: traceroute with MPLS label shows expected label stack via IS-IS paths
- [ ] TI-LFA coverage at 100% (or acceptable threshold) under IS-IS

**BFD & Convergence:**
- [ ] BFD sessions configured and `Up` on all IS-IS interfaces
- [ ] Convergence test performed on at least 2 non-critical links: shut/no-shut, measure traffic loss < target
- [ ] SPF timers configured consistently across all nodes

**Configuration Consistency:**
- [ ] `distance`/`preference` setting for IS-IS is identical on ALL 100 nodes (no per-node overrides)
- [ ] No router has manual OSPF distance lower than IS-IS distance (the War Story scenario)
- [ ] All interfaces that are in OSPF are also in IS-IS (no missing interfaces)
- [ ] Authentication (if used) is consistent across all IS-IS interfaces

**Monitoring & Rollback:**
- [ ] Monitoring/alerting updated to track IS-IS adjacency state, SPF runs, and LSDB size
- [ ] Rollback plan documented: raise IS-IS distance back above OSPF distance (single command per node)
- [ ] Rollback tested on at least one node in lab or non-critical POP
- [ ] NOC briefed on migration window, expected behavior, and escalation path
- [ ] Traffic baseline captured (interface utilization, latency, packet loss) for before/after comparison

**Execution Plan:**
- [ ] Migration order defined: start with smallest POP, expand to larger POPs
- [ ] All routers in a traffic path migrate simultaneously (no asymmetric protocol selection)
- [ ] Maintenance window approved and communicated
- [ ] At least 2 engineers available during cutover (one executing, one monitoring)

**No-Go Criteria (any of these = abort):**
- IS-IS adjacency count doesn't match OSPF neighbor count on any node
- Any prefix reachable via OSPF but not via IS-IS
- BFD not operational on any core link
- Unresolved configuration inconsistencies across nodes
- Active network incident in progress
