# 2.3 — IGP Convergence Tuning — Answer Key

## Question 1
> A router has BFD at 100ms × 3, SPF initial-wait 50ms, and FIB programming takes 200ms. What's the minimum convergence time for a link failure without FRR? With TI-LFA?

### Answer

**Without FRR: ~550ms minimum.**

Walk the convergence pipeline:

| Stage | Time | Explanation |
|-------|------|-------------|
| **1. Detection (BFD)** | 300ms | 100ms interval × 3 multiplier = 300ms worst case (miss 3 consecutive packets) |
| **2. LSP/LSA generation** | ~50ms | Router detects failure via BFD, generates new LSP with updated topology. With initial-wait of 50ms, first LSP goes out in ≤50ms |
| **3. Flooding** | ~10ms | LSP floods to neighbors. On a well-connected backbone, full flood takes 10-100ms depending on diameter. Use ~10ms for adjacent routers |
| **4. SPF computation** | 50ms | SPF initial-wait is 50ms. First SPF runs after 50ms delay from LSP receipt |
| **5. RIB update** | ~10ms | Software operation, fast on modern platforms |
| **6. FIB programming** | 200ms | Given in the problem |
| **Total** | **~620ms** | Sum of all stages |

More precisely: stages 2-3 can partially overlap (pipelining), so a realistic minimum is **~550ms**. The dominant contributors are BFD detection (300ms) and FIB programming (200ms).

**With TI-LFA: ~300ms.**

TI-LFA pre-computes backup paths and pre-installs them in the FIB. When BFD detects the failure, the data plane switches to the backup path immediately — no SPF, no RIB update, no FIB programming needed.

| Stage | Time | Explanation |
|-------|------|-------------|
| **1. Detection (BFD)** | 300ms | Same — BFD still needs to detect the failure |
| **2. Data plane switchover** | ~0ms | Backup next-hop is already in FIB; linecard switches on BFD trigger |
| **Total** | **~300ms** | Just detection time |

The SPF/RIB/FIB pipeline still runs in the background to compute the post-convergence optimal path, but traffic is already flowing on the backup path within ~300ms. The background convergence replaces the backup path with the new optimal path within another ~300-500ms.

**Key insight:** With FRR, the convergence time is dominated entirely by failure detection. If you drop BFD to 50ms × 3 = 150ms, your TI-LFA convergence drops to ~150ms. That's approaching the physical limit of what's detectable.

## Question 2
> You're deploying a new backbone with 50 P-routers and 200 PE-routers across 8 POPs. Design a tiered convergence strategy (core vs. access). What BFD timers, SPF settings, and FRR mechanisms do you use at each tier?

### Answer

**The principle: aggressive on the core where stability is proven, conservative on the access where link quality varies.**

### Tier 1: Core (P-to-P links, 50 P-routers)

These are high-bandwidth inter-POP links on carrier-grade hardware with hardware BFD offload. Failures here affect thousands of customers. Speed is everything.

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| **BFD interval** | 50ms × 3 = 150ms detection | Hardware BFD offload on modern linecards eliminates false positives. Core links are stable. |
| **SPF initial-wait** | 50ms | Fastest first SPF. Core topology changes are rare but critical. |
| **SPF secondary-wait** | 100ms | Standard exponential backoff. |
| **SPF max-wait** | 5000ms | Bounds backoff ceiling. |
| **LSP gen initial** | 50ms | Match SPF aggressiveness. |
| **FRR mechanism** | TI-LFA (SR-MPLS) | 100% prefix coverage, pre-installed backup paths. No LFA coverage gaps. |
| **Prefix prioritization** | Critical: loopbacks, BGP next-hops | Ensure BGP next-hops converge first via `spf prefix-priority critical` |

**Core convergence target: <200ms with TI-LFA.**

### Tier 2: Aggregation (PE-to-P links, PE uplinks to core)

These links connect PE routers to the core. Hardware varies more — some PEs may be on mid-range platforms without hardware BFD.

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| **BFD interval** | 100ms × 3 = 300ms detection | Slightly conservative — PE platforms may have software BFD with higher jitter. |
| **SPF initial-wait** | 50ms | Same as core — SPF is cheap. |
| **SPF secondary/max** | 200ms / 5000ms | Standard. |
| **FRR mechanism** | TI-LFA (SR-MPLS) | Same as core. If PE has dual uplinks to core (which it should), TI-LFA covers both link and node failure. |

**Aggregation convergence target: <500ms with TI-LFA.**

### Tier 3: Access (PE-to-CE or PE-to-access-switch)

These links face customer equipment. Link quality is less predictable. False BFD positives here cause unnecessary customer-affecting failovers.

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| **BFD interval** | 300ms × 3 = 900ms detection | Conservative. Access links may traverse last-mile transport (Ethernet demarcation, microwave) where jitter is real. |
| **SPF initial-wait** | 50ms | Still fast — no reason to slow SPF. |
| **FRR mechanism** | LFA or no FRR | PE-to-CE links typically don't benefit from FRR — there's only one CE. If dual-homed CE, use PIC (Prefix Independent Convergence) via BGP instead. |

**Access convergence target: <1.5s (BFD) or <5s (IGP hello fallback if BFD not supported on CE).**

### Global settings (all tiers):

```
IOS-XR:
  router isis CORE
    set-overload-bit on-startup wait-for-bgp
    address-family ipv4 unicast
      spf-interval maximum-wait 5000 initial-wait 50 secondary-wait 100
      spf prefix-priority critical tag 1000

Junos:
  protocols isis {
      overload timeout 300;
      spf-delay 50;
  }
```

## Question 3
> After enabling TI-LFA, `show isis fast-reroute` shows only 73% prefix coverage. All nodes have SR prefix-SIDs. What are the possible causes and how do you investigate?

### Answer

73% coverage means TI-LFA can't compute a backup path for 27% of prefixes. Since all nodes have prefix-SIDs (eliminating the most common cause), the issue is topological or configurational.

**Cause 1: TI-LFA not enabled on all interfaces.**

TI-LFA must be explicitly enabled per-interface on most platforms. If you only enabled it on some interfaces, prefixes reachable only through non-protected interfaces won't have backup paths.

```
IOS-XR: show isis interface brief    ← check FRR column for each interface
        show run router isis | include fast-reroute
Junos:  show isis interface detail   ← check "Node/Link protection" per interface
```

Fix: Enable TI-LFA on every IS-IS interface:
```
IOS-XR: fast-reroute per-prefix ti-lfa   ← under each interface
Junos:  node-link-protection             ← under each interface
```

**Cause 2: Topology doesn't provide alternate paths.**

TI-LFA needs at least one alternate path to the destination that doesn't traverse the protected link/node. In hub-and-spoke topologies (single-homed PEs), there literally is no backup path — TI-LFA can't invent a path that doesn't exist. 27% uncovered prefixes could correspond exactly to single-homed leaf nodes.

Investigate:
```
IOS-XR: show isis fast-reroute <prefix> detail
        ← shows why TI-LFA failed for specific prefixes
Junos:  show isis backup coverage
        show isis backup spf results <prefix>
```

If uncovered prefixes are all behind single-homed PEs, that's expected behavior — not a bug.

**Cause 3: Missing SIDs or insufficient label stack depth.**

Different SRGBs across nodes are operationally undesirable but don't inherently break TI-LFA — the IGP advertises each node's SRGB, so the head-end can compute correct labels. More common TI-LFA failures: missing prefix-SIDs on backup path nodes, platform label stack depth limits preventing deep backup segment lists, or insufficient topology diversity (no alternate path exists around the protected element).

```
IOS-XR: show isis segment-routing label table
Junos:  show isis source-packet-routing srgb
```

Fix: Standardize SRGB across all nodes (e.g., `start-label 16000 index-range 8000` everywhere).

**Cause 4: Flex-Algo or topology constraints.**

If any Flex-Algo configurations or admin-groups (affinities) are in use, TI-LFA backup path computation respects those constraints — potentially eliminating valid backup paths.

**Investigation workflow:**
1. List uncovered prefixes: `show isis fast-reroute summary`
2. Pick a specific uncovered prefix and check why: `show isis fast-reroute <prefix> detail`
3. Check if the uncovered prefixes share a pattern (all behind same node? all in same topology region?)
4. Verify SRGB consistency, interface-level TI-LFA enablement, and topology redundancy
