# 2.2 — OSPF in SP Networks — Answer Key

## Question 1
> You're migrating a 200-router metro network from OSPF to IS-IS. The network uses NSSA areas for external route injection at access PEs. What's your migration strategy that avoids downtime? What's the IS-IS equivalent of NSSA?

### Answer

**Migration strategy: Ships in the Night.**

Run IS-IS alongside OSPF simultaneously, then shift traffic gradually. This is the only approach that avoids downtime for a network this size.

**Phase 1 — Enable IS-IS everywhere, passive.** Configure IS-IS on all 200 routers with interfaces added but at a higher administrative distance (IOS-XR: IS-IS default distance 115 > OSPF 110, so IS-IS naturally loses — no change needed) or higher preference (Junos: IS-IS L2 preference 18 > OSPF internal 10, so IS-IS also naturally loses). Validate that all IS-IS adjacencies form, LSDB is complete, and all loopbacks are reachable via IS-IS.

**Phase 2 — Validate IS-IS parity.** Compare IS-IS and OSPF routing tables on every node. Every prefix reachable via OSPF must also be reachable via IS-IS with comparable metrics. Check BFD sessions are up on IS-IS interfaces. Verify SR SIDs (if deploying SR) are programmed correctly.

**Phase 3 — Shift traffic.** Lower IS-IS distance to 105 (IOS-XR: `distance 105` under `router isis`) or preference to 9 (Junos: `preference 9` under `protocols isis`). Do this per-region, not globally — start with a small POP, validate, then expand. All routers in a given traffic path must shift simultaneously to avoid asymmetric routing.

**Phase 4 — Remove OSPF.** After 2+ weeks of stable IS-IS operation under production load, remove OSPF configuration. Do this in reverse order — leaf/access first, core last — so you always have a fallback path.

**IS-IS equivalent of NSSA:** There is no direct equivalent. IS-IS doesn't have area types like OSPF's stub/NSSA/totally-stubby. IS-IS handles external routes differently — external routes are redistributed into IS-IS and advertised as L2 routes with an "external" metric type (internal vs. external bit in TLV 135). All L2 routers see all routes; there's no mechanism to filter external routes from specific areas.

For the NSSA use case (external route injection at access PEs without flooding externals everywhere), the IS-IS approach is:
1. Redistribute the external routes into IS-IS at the access PEs (they become L2 external routes)
2. Use route policy/filtering at redistribution points to control which externals are injected
3. If you need to limit flooding scope, use L1/L2 hierarchy — put access PEs in L1 areas, redistribute only summarized routes into L2

The practical outcome is simpler: IS-IS doesn't need NSSA because its TLV extensibility and two-level hierarchy handle the use case without special area types.

## Question 2
> A regional SP has 3 POPs connected in a triangle. Each POP has 10-15 access routers behind an ABR pair. Total network is ~80 routers. Would you recommend OSPF or IS-IS? Justify your answer considering operational complexity, IPv6 requirements, and future SR deployment.

### Answer

**IS-IS. But this is one of the closer calls you'll encounter.**

At 80 routers, OSPF can absolutely handle the scale. The 3-POP triangle with ABR pairs maps cleanly to OSPF multi-area design (Area 0 for the 6 ABRs and inter-POP links, Areas 10/20/30 for each POP's access layer). OSPF would work fine today.

But "works fine today" isn't the question. The question includes IPv6 and future SR.

**IPv6 requirements tilt toward IS-IS.** With OSPF, you need OSPFv3 as a separate process for IPv6 — or OSPFv3 with Address Families (RFC 5838) to carry both IPv4 and IPv6 in one process. Either way, you're adding complexity. With IS-IS, IPv6 support is a TLV (TLV 236 for IPv6 Reachability). Single process, single configuration, single LSDB. If you need different IPv4 and IPv6 topologies, IS-IS multi-topology (MT) handles that natively.

**Future SR deployment strongly favors IS-IS.** While SR-MPLS works with both protocols, IS-IS has cleaner integration — prefix SIDs are carried in sub-TLVs of existing reachability TLVs, Flex-Algo support is native, and SRv6 support via TLV extensions (RFC 9352) is more mature than the OSPFv3 equivalent. Every vendor's SR reference design starts with IS-IS. You'll spend less time in TAC.

**Operational complexity argument:** The "team knows OSPF" argument is valid but short-sighted. IS-IS configuration is actually simpler — no area types (stub/NSSA/totally-stubby), no DR/BDR on P2P links (same as OSPF, but IS-IS has no NBMA mode to confuse anyone), no MTU adjacency sensitivity. The learning curve is 2-3 weeks for an experienced OSPF engineer. The cost of NOT migrating — maintaining two IGP skill sets when you eventually need IS-IS for SR — is higher.

**Design for this network on IS-IS:**
- Flat L2 across all 80 routers (well within single-level limits)
- SR-MPLS enabled from day one (even if not used immediately — costs nothing)
- BFD on all inter-router links
- TI-LFA for sub-50ms convergence on the triangle core

No hierarchy needed at 80 nodes. No area design to maintain. No NSSA/stub decisions. Simpler.

## Question 3
> Your OSPF network has Area 0 with 6 P-routers and 4 ABRs. One ABR reports 3x the expected LSA count in Area 0. The other 3 ABRs show normal counts. SPF is running every 2 seconds on the affected ABR. What's your triage approach?

### Answer

This ABR is seeing LSAs that the other three aren't — which means it's either receiving rogue LSAs from a non-Area 0 source, or it's generating excessive LSAs itself.

**Step 1: Identify the excess LSAs.**
```
IOS-XR: show ospf database summary        ← compare totals per LSA type
        show ospf database | count          ← raw count
Junos:  show ospf database summary
```
Compare LSA type counts between the affected ABR and a healthy one. The delta tells you which LSA type is inflated. Most likely candidates:
- **Type 3 (Summary LSAs):** ABR is leaking routes it shouldn't — possibly from a non-backbone area that isn't properly configured as stub/NSSA
- **Type 5 (External LSAs):** Redistribution running on this ABR that isn't present on others
- **Type 1 (Router LSAs):** Another router is flooding excessive LSAs into Area 0 through this ABR

**Step 2: Find the originating router.**
```
IOS-XR: show ospf database router adv-router <suspicious-RID>
        show ospf database external adv-router <suspicious-RID>
Junos:  show ospf database advertising-router <suspicious-RID> detail
```
Look at the advertising router ID on the excess LSAs. If they all come from one router, that's your culprit.

**Step 3: Check for stale/phantom LSAs.**
```
IOS-XR: show ospf database | include MaxAge
Junos:  show ospf database | match "MaxAge|0x80"
```
If you see LSAs at MaxAge (3600 seconds) that aren't being purged, there's a database synchronization issue — possibly a router that left the network but its LSAs weren't properly flushed.

**Step 4: Check this ABR's area membership.**
Verify this ABR isn't accidentally connected to an area that the other ABRs don't see. A misconfigured interface in a wrong area could inject an entire area's LSAs into Area 0. Check `show ospf interface` — every interface should be in the expected area.

**Step 5: Check for mutual redistribution.**
If this ABR has route redistribution configured (e.g., BGP→OSPF or static→OSPF) that the others don't, it would generate Type 5 LSAs that the other ABRs wouldn't originate. Check `show run router ospf` for any `redistribute` statements.

**The SPF running every 2 seconds is a symptom, not the root cause.** Each new/changed LSA triggers SPF recomputation. Fix the LSA source, and SPF stabilizes. If you need immediate relief while troubleshooting, increase SPF max-wait temporarily: `timers throttle spf 1000 2000 10000`.
