## Day-Zero Platform Requirements

Before you plan a single SID, know your platform's prerequisites. Missing these causes silent forwarding failures that look like bugs.

### NCS5500 — hw-module profile Requirement

On NCS5500 (and NCS5700 series), SRv6 forwarding requires a hardware profile that must be configured **before** enabling SRv6. It requires a line card reload:

```
hw-module profile segment-routing srv6
```

Without this profile, the hardware will not program SRv6 forwarding entries correctly. This is a **day-zero configuration requirement** — not something you add after the fact. If you deploy SRv6 on an NCS5500 without this profile and wonder why forwarding looks wrong, this is why.

After adding the hw-module profile, **a full LC reload is required**. Plan for a maintenance window.

The same profile gate applies to enabling uSID mode. The reload is non-negotiable — there is no workaround.

## SID Allocation Strategy

The first question in any SRv6 deployment is not "what equipment do we buy" — it's "how do we hand out SIDs without collisions." Get this wrong upfront and you'll pay for it every time you add a node, expand a region, or merge with another network.

### The SID Structure Recap

Every SRv6 SID is a 128-bit IPv6 address structured as `Locator | Function | Argument`. The locator is the network-routable piece — it's what IGP advertises and what gives you forwarding reach toward the node that owns this SID. The function is the instruction encoded in the lower bits of the SID — it tells the owning node what behavior to execute. The argument is optional per-packet context some behaviors consume.

For allocation purposes, you control two things: **how you carve up the locator space across nodes**, and **how each node assigns function codes within its locator**.

### Block Assignment Per Region/Node

Best practice is a three-level hierarchy: **SRv6 Domain Block → Regional Block → Per-Node Locator**.

Pick an IPv6 prefix to own as your SRv6 SID block. For intra-domain SRv6 (SIDs are never advertised beyond your AS), most SPs use Unique Local Addresses (ULAs, `fc00::/7`) — operators like SoftBank and Iliad/Free run production networks this way. For inter-domain SRv6 (cross-AS SID advertisement via BGP CT), you need globally routable space from your PA or PI allocation. RFC 9602 also assigned `5f00::/16` as the designated SRv6 SID range, and Cisco reserves `fcbb:bb00::/32` for lab/documentation. A common allocation is a /32 or /40 as the SRv6 domain block, subdivided from there.

**Intra-domain example (ULA — most common):**
```
SRv6 Domain Block:  fc00:0::/32       (your SP's SRv6 address space)

Regional Subdivision:
  North America:    fc00:0:0000::/40
  Europe:           fc00:0:0100::/40
  Asia-Pacific:     fc00:0:0200::/40

Per-Node Locators (example, North America):
  NYC-PE-01:        fc00:0:0001::/48
  NYC-PE-02:        fc00:0:0002::/48
  CHI-PE-01:        fc00:0:0010::/48
  LAX-PE-01:        fc00:0:0020::/48
```

**Inter-domain / globally routable example (RFC 9602):**
```
SRv6 Domain Block:  5f00:db8::/32     (using RFC 9602 5f00::/16 space)

Per-Node Locators:
  NYC-PE-01:        5f00:db8:0001::/48
  LAX-PE-01:        5f00:db8:0020::/48
```

The /48 per-node locator is industry standard. It gives you 16 bits for functions (65,536 function codes per node) and optionally 16 bits for arguments. Some implementations use tighter locators — /56 or /64 — when address space is constrained. Don't go tighter than /56 unless you have a very specific reason; the function space gets cramped and you'll hit limits when adding VRFs.

### uSID Block Planning

If you're deploying micro-SIDs (the uSID compression scheme, now standardized in RFC 9252 and RFC 9354 — check IETF Datatracker for the current status of related drafts), you need a separate **uSID block** — a prefix within which function values are compressed from 16 bytes down to 2 bytes per SID.

RFC 9602 assigned `5f00::/16` as the SRv6 SID range, which covers uSID deployments. Cisco's lab and documentation convention uses `fcbb:bb00::/32`. In practice, most deployed networks use a /16 or /32 carved from their ULA or PA space as the uSID block. There is no single "industry standard" block — consistency within your own domain is the requirement.

```
uSID Block:         fc00::/16        (example — operator-chosen, ULA is common)
Node uSID locators: fc00:0001::/48   (per-node uSID locator within the block)
```

The F3216 uSID format (32-bit block + 16-bit node function fields) is the dominant IOS-XR format. A /32 node locator in F3216 accommodates **6 micro-SIDs** per 128-bit address (32-bit block + 6×16-bit slots = 128 bits total) — meaning a 6-hop path fits in a single IPv6 destination address with no SRH required.

The key operational constraint: **uSID locators must not overlap with full SRv6 locators**. Track them separately in your registry. Mixing them silently breaks forwarding in ways that are painful to debug.

### The SID Registry

This is not a nice-to-have. It is the thing that prevents a 3am collision incident when someone adds a new PoP without checking whether that /48 is already claimed by the Singapore PE they deployed two years ago and forgot to document.

Your SID registry needs at minimum:

| Field | Example |
|-------|---------|
| Locator prefix | `fc00:0:0001::/48` |
| Node hostname | `nyc-pe-01` |
| Region | North America |
| Role | PE (L3VPN headend) |
| IGP | IS-IS L2, instance CORE |
| Allocated date | 2025-01-15 |
| Status | Active |
| Function assignments | e000=END, e001=END.X, d400=END.DT4, d600=END.DT6 |

If you don't have proper IPAM, maintain this in a spreadsheet or a Git repo with proper access controls. The point is: **every locator assignment is written down before the node is provisioned**, not after. Retroactive documentation is how you end up with a mystery /48 doing live forwarding that nobody can explain.

### Function Code Allocation

Within each node's locator space, function assignments should also be standardized across your domain. Everyone uses the same function codes for the same behaviors:

```
e000    END           — Basic segment endpoint
e001    END.X         — Endpoint with cross-connect (adj-SID equivalent)
d400    END.DT4       — Decap + IPv4 VRF lookup
d600    END.DT6       — Decap + IPv6 VRF lookup
d401    END.DT46      — Decap + IPv4/IPv6 VRF lookup
b000    END.B6.ENCAPS — Encap + SRv6 Policy insertion (headend behavior)
```

Non-standard function codes are fine for platform-specific behaviors, but document them. Operational coherence across your NOC requires that when someone looks at an SID and sees `:d400:`, they know immediately what that node will do with a packet carrying that SID as destination.

## Locator Planning

### /48 vs /64 Block Sizing

The size of your locator prefix is a tradeoff between route table efficiency and function address space.

| Locator Size | Function Bits | Argument Bits | Typical Use |
|---|---|---|---|
| /48 | 16 bits (65,536 codes) | 16 bits | Standard PE, P nodes |
| /56 | 8 bits (256 codes) | 24 bits | Constrained address space |
| /64 | 16 bits (65,536 codes) | 48 bits | uSID-only or constrained locator deployments |

> **Note:** The SID structure is `Locator | Function | Arguments` — all three fields must sum to 128 bits. A /48 locator leaves 80 bits for function+argument. Common split: 16 bits function, 16 bits argument, 48 bits zeroed (a /48 END SID typically looks like `fc00:0:0001:e000::`). Implementations vary — verify against your platform's SID format documentation.

**Use /48 for standard deployments.** 16 bits of function space (65,536 codes per node) is more than any production deployment will ever need. It's clean, it matches what most documentation assumes, and it keeps your locator math simple.

The only reason to go smaller is if you're constrained on IPv6 prefix space — which shouldn't happen in a greenfield SRv6 deployment if you've properly planned your SRv6 domain block. If you find yourself arguing about going to /56 because your domain block is too small, the real problem is that you allocated too small a domain block.

### Summarization at Domain Boundaries

At the edge of your SRv6 domain — the border where your network peers with other ASes, or where you have an SRv6-unaware region — your locators need to be summarizable.

This is why the regional block structure matters. Your North America block `/40` can be summarized into a single advertisement at domain boundaries. Internal per-node /48 locators don't leak beyond your region's edge if you're disciplined about your block allocation.

If your locators don't summarize — if you've assigned them without regard for regional hierarchy — you end up leaking hundreds of /48 prefixes into your BGP table at domain boundaries. That's a management nightmare and a scalability problem.

For inter-domain SRv6 (BGP CT, per-CE SID advertisement), summarization requirements become stricter. Plan accordingly.

### Renumbering Implications

Nobody wants to renumber SRv6 locators. If you've used SIDs in YANG models, provisioning systems, monitoring dashboards, and SR-TE policy configs, changing a locator means changing every one of those references — and the risk of a missed reference causing a silent forwarding failure is real.

**Do not rush your SRv6 address plan.** Get it right at day zero. The cost of renumbering scales with deployment size and operational maturity. A mistake that costs 2 hours to fix in a 5-node lab costs 2 weeks in a 500-node production network.

One practical safeguard: **never assign locators to node hostnames — assign them to node roles or node IDs**. If your node gets renamed or its role changes, the SID persists. The SID is the stable identifier; the hostname is the friendly label.

## SR-TE Policy Lifecycle

SR-TE policies in SRv6 are the mechanism for traffic engineering — they define an explicit path through the network encoded as an SID list. Understanding their lifecycle is essential for day-2 operations.

### Creation

Policies can be configured statically (hardcoded SID list), signaled via PCEP (PCE computes the path, signals the headend), or auto-instantiated by the headend based on ODN (On-Demand Next-Hop) requests from BGP.

Static policy creation on IOS-XR:
```
segment-routing
 traffic-eng
  policy CORP-VOICE-NYC-LAX
   color 100 end-point ipv6 fc00:0:0020::   ! LAX-PE-01 locator
   candidate-paths
    preference 100
     explicit segment-list NYC-LAX-VOICE
      !
  segment-list NYC-LAX-VOICE
   index 10 address ipv6 fc00:0:0010:e000:: ! CHI-PE-01 END SID
   index 20 address ipv6 fc00:0:0020:e000:: ! LAX-PE-01 END SID
```

Static policy on Junos (22.x+, `protocols spring-traffic-engineering`):
```
protocols {
    spring-traffic-engineering {
        srv6 {
            candidate-path "NYC-LAX-VOICE" {
                color 100;
                endpoint fc00:0:0020::;   # LAX-PE-01
                preference 100;
                segment-list {
                    srv6 fc00:0:0010:e000::;  # CHI-PE-01 END
                    srv6 fc00:0:0020:e000::;  # LAX-PE-01 END
                }
            }
        }
    }
}
```

> **Junos SR-TE CLI warning:** Junos SRv6 SR-TE config hierarchy changed significantly between major releases (20.x used `routing-options source-packet-routing`, 22.x moved to `protocols spring-traffic-engineering`). The above reflects 22.x behavior. Always verify the hierarchy against the release-specific Feature Guide before deploying. CLI paths that look valid may silently commit but not activate on older releases.

### Monitoring Steady-State

Once a policy is up, you need to know it's doing what you think it's doing. The key monitoring points:

**IOS-XR:**
```
show segment-routing traffic-eng policy name CORP-VOICE-NYC-LAX
show segment-routing traffic-eng forwarding policy name CORP-VOICE-NYC-LAX
show segment-routing traffic-eng policy all detail
```

The first command shows policy state — whether the preferred candidate path is active, what the SID list is, and the color/end-point tuple. The second shows the forwarding table entry — what the ASIC actually has programmed. These should match. If they don't, you have a policy/forwarding sync issue.

**Junos:**
```
show spring-traffic-engineering lsp
show spring-traffic-engineering lsp detail
show spring-traffic-engineering segment-list
```

> **Note on Junos CLI:** The correct command family is `show spring-traffic-engineering` (not `show spring-te`, which is an alias that may not be available in all releases). For SRv6-specific state use `show srv6`; for SR-TE policy state use `show spring-traffic-engineering`. These are different subsystems.

**State to watch:**

| State | Meaning | Action |
|---|---|---|
| Policy UP, path ACTIVE | Healthy | None |
| Policy UP, path INACTIVE | SID list invalid, IGP unreachable | Check SID reachability |
| Policy DOWN | End-point unreachable | Check BGP/IGP to end-point |
| Policy UP, no traffic | Steering rule missing | Check steering config / BGP CT / ODN |

### Policy Modification

Changing a live SR-TE policy's SID list requires care. The headend needs to atomically switch from the old path to the new one without dropping traffic in-flight on the old path.

IOS-XR supports **make-before-break** for SR-TE — you can add a new candidate path at higher preference while the old path is still active. The headend installs the new path, verifies forwarding, then tears down the old path. Verify the new path is truly active before removing the old preference:

```
! Add new higher-preference path
segment-routing
 traffic-eng
  policy CORP-VOICE-NYC-LAX
   candidate-paths
    preference 200
     explicit segment-list NYC-LAX-VOICE-VIA-SFO
    !
! Verify new path is active BEFORE removing preference 100
show segment-routing traffic-eng policy name CORP-VOICE-NYC-LAX
! Remove old path after confirmation
no candidate-paths preference 100
```

Do not remove the old path and add the new one in the same change. That's a traffic blackhole window.

### Teardown

Removing an SR-TE policy while traffic is using it is a live operation. The headend will reroute traffic to the best alternate path when the policy goes down — but if there's no alternate (the policy was the only route to this destination for a specific color), traffic blackholes until normal IP routing takes over.

Before tearing down a policy, verify:
1. Is there a fallback path? (non-TE route to the end-point, alternate color policy)
2. What services are steering traffic into this policy?
3. Is there a maintenance window?

A policy removal that drops 50 Gbps of enterprise VPN traffic because someone forgot to check the steering rules is the kind of incident that ends careers.

## Monitoring SRv6

### What Telemetry to Stream

Not all counters are equal. These are the ones that matter for production SRv6 operations:

**Per-SID counters** — bytes/packets forwarded per SID. This is your per-path traffic visibility. If SID `fc00:0:0001:d400::` is your VRF customers-north decap SID, its byte counter tells you how much traffic is landing in that VRF.

**SRH drop counters** — the ASIC's count of packets dropped because they arrived with an SRH and the node didn't know how to process the requested SID. Non-zero and rising means you have SIDs in the network pointing at something that doesn't exist on this node.

**Policy steering counters** — how much traffic is being steered into each SR-TE policy. If a policy is up but the steering counter is zero, the steering rule isn't matching.

**Encap/decap packet counters** — headend nodes add SRH; tailend nodes strip it. Asymmetric counters across your backbone (encap input != decap output, accounting for replication) indicate packet loss somewhere in the SRv6 path.

**IGP SID advertisement counts** — are all nodes advertising their locators and SIDs? A node that drops out of IGP stops advertising its SIDs, and any in-flight packets toward those SIDs blackhole.

### Path Verification

The basic SRv6 equivalent of "traceroute" is still just traceroute — but you need to understand what you're looking at. SRv6 nodes that are processing SRH segments will appear as intermediate hops only if they respond to probe traffic. Many production deployments rate-limit or suppress ICMP responses from P-nodes. You may see fewer hops than you expect.

For explicit path verification on IOS-XR:
```
traceroute ipv6 fc00:0:0020:e000:: source-address fc00:0:0001::
```

This sends an IPv6 probe toward the SID, which causes the path to follow the SRv6 forwarding entries. The intermediate hops are the SRv6 segment endpoints. Compare actual hops against expected SID list.

**SID reachability verification** — confirm your SID is in the forwarding table before concluding a failure is a forwarding issue:
```
! IOS-XR
show segment-routing srv6 sid detail
show route ipv6 fc00:0:0001::/48

! Junos
show srv6
show route table inet6.0 fc00:0:0001::/48
```

## Troubleshooting SRv6

### Packet Captures with SRH

Capturing and reading an SRv6 packet is not the same as capturing a regular IPv6 packet. The SRH is an IPv6 extension header between the IPv6 base header and the payload. Most capture tools handle this — but how you filter for it matters.

**Wireshark display filter for SRH (Routing Header Type 4):**
```
ipv6.routing.type == 4
```

Or to capture at the interface:
```
tcpdump -i eth0 'ip6[6] == 43'
```

IPv6 Next Header value 43 is the Routing Header — this catches all routing extension headers including SRH. If you're doing extended header filtering at the kernel level (for rate limiting captures on a busy interface), use `ip6[6] == 43` to pre-filter.

**Reading the capture:**

In Wireshark, expand "IPv6 Routing Header" to see:
- Routing Type: should be 4 for SRH
- Segments Left: shows which hop the packet is currently on
- Addresses: the full SID list

Remember: Segment List[0] is the **final destination**, Segment List[n] is the **first hop** (encoded in the IPv6 Destination Address). The current segment is at index `Segments Left` — the packet is heading toward Segment List[Segments Left].

A packet captured mid-transit showing Segments Left=1 means it has one more segment to process after the current one.

**Platform capture commands:**

> **⚠️ Platform note:** `monitor capture` is **IOS-XE** (ASR1K, Catalyst 9K) — it does **not** exist on IOS-XR (ASR9K, NCS5500, 8000 series). On IOS-XR, packet capture is done via SPAN/monitor-session or by dropping to the Linux shell.

IOS-XR (NCS5500 / ASR9K) — SPAN to loopback then tcpdump:
```
! Step 1: Configure SPAN to capture interface
monitor-session CAPTURE ethernet
 destination interface Loopback0 !  actual capture via shell

! Step 2: From Linux shell (IOS-XR admin-plane)
tcpdump -i eth0 'ip6[6] == 43' -w /harddisk:/srv6-cap.pcap

! Or via NSim shell for inline analysis:
run tcpdump -i Bundle-Ether0.0 -nn 'ip6[6] == 43' -c 200
```

On IOS-XR platforms, the recommended path for production captures is ERSPAN (mirror to a capture host) or using the `spp` (Service Plane Packet capture) infrastructure where supported. Check your platform's Network Packet Capture guide for the current supported method — it varies by platform and IOS-XR version.

IOS-XE (ASR1K / Catalyst 9K):
```
monitor capture MYCAP interface GigabitEthernet0/0/0 both
monitor capture MYCAP access-list ACL-SRV6
monitor capture MYCAP buffer size 10
monitor capture MYCAP start
! ... wait for traffic ...
monitor capture MYCAP export location flash:/cap.pcap
monitor capture MYCAP stop
```

Junos:
```
monitor traffic interface et-0/0/0 matching "ip6[6]=43" write-file /var/tmp/srh.pcap
```

Wireshark analysis from pcap works identically across platforms. Inline display of SRH fields at line rate is generally not reliable at 100G+ — capture and export is the right workflow.

### SID Resolution Failures

Symptoms: packets arrive at a node with this node's SID in the DA, but the node drops them instead of executing the behavior.

Common causes:

**SID not in local SID table** — The node doesn't have this SID configured. Could be that the SID was never provisioned, the config was removed (BGP/VRF deletion takes the associated SID with it on most platforms), or there's a locator mismatch.

```
! IOS-XR — check local SID table
show segment-routing srv6 sid
show segment-routing srv6 sid fc00:0:0001:d400::

! Junos — check SRv6 state
show srv6
show srv6 locator
show srv6 sid-table
```

> **Junos SRv6 table name note:** The SRv6 SID table in Junos is accessed via `show srv6 sid-table` or `show srv6` aggregate. The routing table name varies by implementation — consult your release's `show srv6` command family output for the correct table name rather than hardcoding it.

**Locator not in IGP** — The SID exists locally but the locator isn't being advertised into IS-IS/OSPFv3. Other nodes can't reach this SID because they have no route to its /48. This happens when you configure a locator but forget to advertise it in the IGP.

```
! IOS-XR — verify locator is in ISIS
show segment-routing srv6 manager detail
show isis route fc00:0:0001::/48

! Junos
show isis route fc00:0:0001::/48
```

**Wrong function code** — The packet arrived with function code `:d400:` but the node only has `:d600:` configured for that VRF (i.e., IPv4 vs IPv6 decap mismatch). The SID exists but the specific function doesn't match.

**Behavior mismatch after VRF add/remove** — When a VRF is deleted, IOS-XR automatically removes the associated END.DT4/DT6 SIDs. If traffic continues arriving for those SIDs (because a remote headend still has a stale SID list), it will silently drop. SID cleanup on headends needs to happen before or simultaneously with VRF teardown.

### Encap/Decap Mismatches

These are the failures where traffic enters the SRv6 domain from the headend but doesn't make it out the far end in the expected form.

**Symptom:** customer traffic disappears at the tailend PE. Headend shows encap counter incrementing. Tailend shows decap counter at zero.

**Diagnosis:**

1. Verify the tailend's SID is in the headend's route table
2. Verify the tailend has the SID provisioned locally
3. Capture at an intermediate node — is the SRH pointing at the right tailend SID?
4. Check MTU at each hop (see Capacity Planning section)

MTU is the sneaky one. Encap looks fine. Decap looks fine. But if an intermediate link can't carry the SRH-padded packet and drops silently (no ICMP Too Big because intermediate nodes often suppress it), you get a unidirectional blackhole that only shows up at high utilization.

### FlexAlgo SID Failures

FlexAlgo-specific SIDs (Algorithm 128+) use separate loopback advertisements. If a node stops participating in a FlexAlgo topology, its FlexAlgo SIDs become unreachable.

Verification on IOS-XR:
```
show isis flex-algo
show isis flex-algo 128 topology
show segment-routing srv6 sid detail | include "Algorithm: 128"
```

A node absent from `show isis flex-algo 128 topology` won't have its FlexAlgo locator in other nodes' routing tables. Any SR-TE policy using that node's FlexAlgo SID will have its path fail.

## uSID Operational Considerations

### SRv6 Endpoint Behavior Flavors (PSP / USP / USD)

Every SRv6 endpoint SID is configured with a **flavor** — an instruction about what the endpoint does when it processes a segment that is the last (or penultimate) one in the Segment List. This is the SRv6 equivalent of MPLS Penultimate Hop Popping (PHP).

| Flavor | Name | Behavior | When Used |
|---|---|---|---|
| PSP | Penultimate Segment Pop | Node before the final segment pops the SRH if Segments Left = 1 | Reduces processing load on final node |
| USP | Ultimate Segment Pop | Final node strips the SRH | Default for most endpoint behaviors |
| USD | Ultimate Segment Decap | Final node strips both SRH and outer IPv6 header | Required for END.DT4/DT6 (L3VPN decap) |

**Why this matters operationally:**

1. **Troubleshooting captures** — If PSP is configured, the SRH is already gone by the time you capture at the penultimate hop. This affects what you see in a Wireshark trace.

2. **SID advertisement** — Each SID in IS-IS/BGP is advertised with its supported flavors. Mismatched flavor expectations between headend and endpoint cause SRH processing errors.

3. **Config validation:**
```
! IOS-XR — verify SID flavors
show segment-routing srv6 sid detail | include "Flavor\|SID"

! SID config with explicit flavors
segment-routing srv6
 locators
  locator MAIN
   prefix fc00:0:0001::/48
   behavior usid                ! uSID mode
```

For uSID (micro-segment) deployments, the flavor shorthand `psp-usd` means the node supports both PSP and USD — common for PE nodes that both penultimate-pop and VRF-decap.

### What uSID Changes Operationally

micro-SID (uSID) compresses SRv6 SIDs by packing multiple 2-byte micro-SIDs into a single 128-bit IPv6 address. Instead of one SID per segment (16 bytes each, carried in a Segment List), multiple micro-SIDs are packed into a single IPv6 address field. The SRH can be eliminated entirely for short paths — the full path is encoded in the IPv6 DA.

This reduces overhead significantly for compressed paths. In the F3216 format (32-bit block + 6×16-bit uSID slots), a single 128-bit IPv6 address accommodates **up to 6 micro-SIDs** — meaning a 6-hop path fits in the IPv6 DA with no SRH required. Compare:

- **SR-MPLS, 4-hop path:** 4 labels = 16 bytes of label stack overhead
- **Regular SRv6, 4-hop path:** 40-byte outer IPv6 + 8-byte SRH base + 64-byte SID list = **112 bytes total overhead**
- **uSID, 4-hop path (F3216):** 4 micro-SIDs packed into one 128-bit address in DA = **40 bytes** (just the outer IPv6 header, no SRH)
- **uSID, 6-hop path (F3216):** Still **40 bytes** — all 6 hops fit in the DA

For paths longer than 6 hops in F3216, additional uSID carrier IPv6 addresses are needed in the SRH (each carrying 6 more hops). Still dramatically more efficient than regular SRv6.

The operational implication: **you're now dealing with 2-byte identifiers embedded in 128-bit addresses**. Reading a uSID packet in Wireshark means decoding something like `fc00:0001:0010:0020:0030::` as "block fc00::/16, then micro-SIDs 0001, 0010, 0020, 0030 in sequence."

### uSID Block Management

Your uSID block should be a dedicated prefix that all nodes know is the uSID space. There is no single industry-standard uSID block — consistency within your own domain is the requirement. RFC 9602 assigned `5f00::/16` as the SRv6 SID range (covering uSID deployments). Cisco lab and documentation convention uses `fcbb:bb00::/32`. Most operators carve a /16 or /32 from their own ULA or PA space. Pick one prefix and apply it uniformly.

The F3216 format uses a shared **uSID block** (e.g., `/32`) that covers all nodes in the domain, and then **per-node uSID locators** within that block at `/48`. The /32 is the domain-wide uSID space; the /48 is the per-node slice.

```
uSID Block:         fc00::/32        (shared uSID block — covers all nodes)
Node uSID locators: fc00:0001::/48   (per-node uSID locator within the block)
```

```
! IOS-XR — uSID block configuration
segment-routing
 srv6
  encapsulation source-address fc00:0:0001::   ! node's IPv6 source for encap
  locators
   locator MAIN
    micro-segment behavior unode psp-usd
    prefix fc00:0001::/48       ! per-node uSID locator (within the /32 block)
```

**Registry for uSID:** same requirements as regular SID registry, but use a separate table. Track both the shared uSID block (/32) and the individual per-node locators (/48). The uSID format carves function space within the per-node /48 using fixed-width 16-bit fields, so you typically need fewer function codes per node in uSID mode — path encoding is done via DA stacking rather than per-node function allocation.

### Debugging Compressed SIDs

The hard part of uSID debugging is reading the packet. Where regular SRv6 has a nicely structured Segment List with one SID per entry, uSID has multiple function codes packed into IPv6 addresses.

Wireshark (4.x+) has native uSID decoding if you configure the uSID block:
- Preferences → Protocols → IPv6 Segment Routing → uSID Block Length

Without proper uSID block configuration in your decoder, Wireshark shows you raw hex. Know your block.

For CLI inspection of uSID forwarding:
```
! IOS-XR
show segment-routing srv6 sid fc00:0001:d400::
show cef ipv6 fc00:0001:0010:0020:: detail
```

## Capacity Planning

### The MTU Math

This is the one that bites operators who test in the lab at 1 Gbps and deploy at 10 Gbps. MTU failures are silent until you hit the packet size boundary under real traffic, then they cause hard-to-diagnose unidirectional drops.

SRv6 adds overhead. Calculate it before deploying:

**H.Encaps (standard) overhead:**

| Component | Size |
|---|---|
| IPv6 outer header | 40 bytes |
| SRH base header | 8 bytes |
| Per-segment in SID list | 16 bytes each |
| **Total overhead, 1-segment path** | **64 bytes** (40 + 8 + 16) |
| **Total overhead, 3-segment path** | **96 bytes** (40 + 8 + 48) |
| **Total overhead, 5-segment path** | **128 bytes** (40 + 8 + 80) |

**H.Encaps.Red (Reduced encapsulation) overhead — IOS-XR/Junos default:**

Modern IOS-XR (NCS5500, 8000 series) and Junos default to **H.Encaps.Red** — the first segment is placed in the IPv6 Destination Address and is *not* duplicated in the SRH Segment List. This reduces overhead by 16 bytes on every packet.

| Encap Mode | 1-segment | 3-segment | 5-segment |
|---|---|---|---|
| H.Encaps (standard) | 64 bytes | 96 bytes | 128 bytes |
| H.Encaps.Red (reduced) | 40 bytes (no SRH) | 80 bytes | 112 bytes |

For a single-segment path with H.Encaps.Red, no SRH is added at all — the destination SID goes directly in the IPv6 DA. This is important for your MTU calculations: verify which encap mode your platform uses before sizing your MTU headroom.

```
! IOS-XR — check encap mode
show segment-routing srv6 manager detail | include encap
```

If your underlay MTU is 9000 bytes (jumbo frames, common in DC interconnects), you can carry a 3-segment SRv6 policy over an inner payload of up to 8904 bytes. That's fine.

If your underlay MTU is 1500 bytes (still common at some peering links and legacy WANs), you can carry a 1-segment SRv6 path over an inner payload of up to 1436 bytes. If the customer sends a 1500-byte packet, your SRv6 headend needs to fragment it or send an ICMP Too Big — and many operators suppress ICMP Too Big, which causes TCP MSS mismatch and silent performance degradation.

**Required action:** Verify MTU end-to-end before enabling SRv6. Set SRv6 headend MTU to account for the maximum expected SID list depth. On IOS-XR, the SRv6 encapsulation size can be configured to influence the IP MTU advertised to customers:

```
segment-routing srv6
 encapsulation source-address fc00:0:0001::
 ! Policy overhead headroom (bytes)
```

For customer-facing interfaces, configure TCP MSS clamping to prevent silent fragmentation issues:

**IOS-XR (ASR9K / NCS5500):**
```
! TCP MSS adjust on IOS-XR uses the policy-map / service-policy path
policy-map PM-TCP-MSS-ADJUST
 class class-default
  set tcp mss 1380

interface Bundle-Ether100.10
 service-policy input PM-TCP-MSS-ADJUST
 service-policy output PM-TCP-MSS-ADJUST
```

**IOS-XE (ASR1K / Catalyst 9K):**
```
interface GigabitEthernet0/0/0
 ip tcp adjust-mss 1400
 ipv6 tcp-mss-adjust 1380
```

> **Platform note:** `ipv4 tcp-mss-adjust` is IOS-XE syntax and is **not valid on IOS-XR**. On IOS-XR, TCP MSS clamping is done via QoS policy-map or via `segment-routing srv6 encapsulation` settings. Check your platform documentation for the appropriate approach.

Adjust values based on your maximum expected SRH overhead. For a 3-segment backbone path, subtract 96 bytes from the interface MTU and clamp TCP MSS to that value.

### ECMP Behavior with SRH

ECMP load balancing in SRv6 uses the same 5-tuple hash as regular IPv6 ECMP — outer IPv6 source + destination + flow label + protocol + ports. The flow label is 20 bits and should be set by the SRv6 headend to provide entropy.

**IOS-XR ECMP with SRv6:**
IOS-XR automatically includes the IPv6 flow label in ECMP hash calculations. Verify that your headend is setting the flow label:
```
show segment-routing traffic-eng forwarding policy name MY-POLICY detail
! Look for "flow label" in the forwarding entry
```

**The entropy label equivalent in SRv6** — For deep label stacks in SR-MPLS, you'd use an Entropy Label (EL, RFC 6790). In SRv6, the flow label serves this role for outer IPv6 ECMP. For inner payload ECMP (when the outer 5-tuple doesn't provide sufficient entropy), some implementations expose additional configuration.

**Polarization risk** — If all your SR-TE policies use the same source address and different destination addresses (different end-point SIDs), ECMP hash polarization can put all policy traffic on the same path. Ensure your headend uses per-flow source variation or configures different flow labels per policy.

### Per-SID Load Balancing

When traffic is steered into an SR-TE policy via BGP color, and multiple equal-cost policies exist (same color, same end-point), the headend load-balances across them. Each policy gets a slice of the traffic. Monitor per-policy forwarding counters to verify the distribution is as expected:

```
show segment-routing traffic-eng forwarding policy all
```

Per-SID traffic stats are distinct from per-policy stats. A single policy can have a SID list that includes a shared infrastructure SID — multiple policies may share intermediate hops. This is fine for forwarding but means per-SID counters include aggregate traffic, not per-policy traffic.

## FlexAlgo Operations

### Algorithm Lifecycle

A FlexAlgo defines a separate topology (metric type and exclusions) that nodes participating in that algorithm maintain alongside the base topology. FlexAlgo 128-255 are user-defined; the base IS-IS/OSPF topology is Algorithm 0.

FlexAlgo must be defined **consistently across all participating nodes**. If one node defines Algorithm 128 as "delay-metric topology, exclude red links" but another node defines it as "TE-metric topology," the computed paths diverge and traffic can loop or blackhole at the boundary.

**Definition consistency check:**
```
! IOS-XR — verify all nodes have same algo definition
show isis flex-algo 128

! Look for: metric-type, include/exclude affinity bits, prefix-metric
! These MUST match on every node participating in Algorithm 128
```

If a new node is added without the correct FlexAlgo definition, it participates with whatever defaults it has, which may silently break path computation for policies that traverse it.

### Topology Validation

Before relying on a FlexAlgo topology for production traffic engineering, verify it's computing sane paths:

```
! IOS-XR
show isis flex-algo 128 topology
show isis flex-algo 128 topology detail
```

The topology output should show every participating node and the best-path metrics between them. Compare against your expected topology. If a node is missing, either it's not participating (check its FlexAlgo config) or its advertisements aren't reaching this node (check IS-IS adjacency and flooding).

**FlexAlgo SR-TE policy SID verification:**
```
show segment-routing srv6 sid detail | include "Algo"
```

Per-algorithm SIDs are separate from Algorithm 0 SIDs. An END SID for Algorithm 128 is different from the END SID for Algorithm 0 — they're different IPv6 addresses, provisioned separately.

### What Breaks When an Algorithm Is Misconfigured

**Wrong metric type** — Algorithm 128 is supposed to use delay metrics but two nodes have it configured with TE metrics. The path they compute diverges from what other nodes expect. Traffic that should take the low-latency path instead takes a TE-optimal path.

**Missing affinity bits** — One node doesn't exclude the "red" affinity class from Algorithm 128. It includes red links in its topology. Traffic that should avoid red links (e.g., maintenance links) may traverse them via that node.

**Node drops out silently** — IS-IS adjacency flaps and comes back, but FlexAlgo participation state doesn't re-sync properly on some platforms. Symptom: `show isis flex-algo 128` shows the node, but its FlexAlgo SIDs aren't in other nodes' route tables. Fix: clear ISIS adjacency and allow it to re-establish, or reload the FlexAlgo participation:

```
! IOS-XR — force IS-IS FlexAlgo resync
clear isis process   ! clears all IS-IS state and re-floods; use with caution

! Softer approach: clear adjacency to trigger re-advertisement
clear isis adjacency *
```

> **⚠️ Command note:** `clear isis flex-algo` is **not a valid IOS-XR command** in any publicly documented release. The supported approaches are `clear isis process` (drastic — resets all IS-IS state) or clearing specific adjacencies. If you need surgical FlexAlgo state refresh, consult TAC for your specific platform and IOS-XR version — undocumented clear commands can cause reconvergence events in production. Use in maintenance window.
