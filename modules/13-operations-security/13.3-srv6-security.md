## Overview

> SRv6 is the most powerful forwarding plane innovation in service provider networking in two decades. It is also, by design, the most porous. Understanding why takes about five minutes. Fixing it takes a lot longer. This section covers the actual security model of SRv6 — not the aspirational one in marketing decks, but the one that exists on your backbone at 2am.

**What you'll learn:**
- Why SRv6 has no unique ethertype and why that matters for every firewall in your network
- The SRH trust boundary model from RFC 8754 and what "untrusted source" means operationally
- How SID spoofing attacks work and how to think about containment
- Infrastructure ACL design for SRv6 at domain edges
- How vendors differ in their SRH handling and filtering capabilities
- SRv6 vs SR-MPLS: an honest security comparison
- Recommended security architecture for production SRv6 backbones

## The Fundamental Problem

This needs to be stated clearly before anything else, because it changes how you think about every security decision in an SRv6 deployment:

**SRv6 is IPv6. There is no distinguishing ethertype.**

SR-MPLS uses Ethertype `0x8847`. Every firewall, ACL engine, and IDS that understands MPLS knows where the MPLS traffic starts. You can filter on the ethertype. You can deny MPLS forwarding from untrusted sources at the edge. The label space is local — labels are only meaningful within the MPLS domain and carry no forwarding semantics outside it.

SRv6 uses Ethertype `0x86DD` — standard IPv6. The Segment Routing Header is Routing Header Type 4 (RH4), an extension header nested inside a normal IPv6 packet. A firewall that inspects IPv6 but does not specifically look for RH4 extension headers sees SRv6 traffic as ordinary IPv6. It applies IPv6 ACL rules. It passes it.

This is not a vendor implementation bug. This is the architecture. SRv6 was designed to work over any IPv6-capable network, including networks whose transit devices have no idea that SRv6 exists. RFC 8754 §3.2 defines this explicitly: "A transit node is not required to be capable of processing a segment or SRH." It is a defined design characteristic that SRv6 can traverse unmodified IPv6 infrastructure.

The security implication is the mirror image of that feature: **if you have IPv6 reachability to an SRv6-capable node, you can potentially inject a crafted SRH and influence how that node forwards traffic.**

This is the threat model. Everything else in this section is a response to it.

## SRv6 Packet Anatomy — Security Perspective

Before discussing defenses, you need to understand exactly what an SRv6 packet looks like, because the attack surface lives in specific header fields.

```
┌────────────────────────────────────────────────────────┐
│ IPv6 Base Header (40 bytes)                            │
│   Version=6  TC  Flow Label                            │
│   Payload Length  Next Header=43  Hop Limit            │
│   Source Address: (outer IPv6 source — the PE)         │
│   Destination Address: ACTIVE SID (current segment)    │
└────────────────────────────────────────────────────────┘
│ Segment Routing Header — RH Type 4 (RFC 8754)          │
│   Next Header       — protocol of payload              │
│   Hdr Ext Len       — length in 8-octet units - 1      │
│   Routing Type = 4  — identifies this as SRH           │
│   Segments Left     — decrements at each SID           │
│   Last Entry        — index of last element in list    │
│   Flags (8 bits)    — currently all reserved           │
│   Tag (16 bits)     — packet classification (optional) │
│   Segment List[0]   — LAST segment (final destination) │  ← wire order: [0] at top
│   ...                                                  │
│   Segment List[n]   — FIRST segment (in IPv6 DA now)   │  ← Segments Left starts at n
│   Optional TLVs     — HMAC TLV, padding, etc.          │
└────────────────────────────────────────────────────────┘
│ Inner Payload (IPv6, IPv4, Ethernet, etc.)              │
└────────────────────────────────────────────────────────┘
```

**Key fields for security analysis:**

**Destination Address** — This is the active SID. At each node, the IPv6 DA is a SID local to that node. After processing, the node decrements Segments Left and sets the DA to the next SID in the list. An attacker who can write the Segment List controls the forwarding path.

**Segment List** — The ordered list of SIDs encoding the full path. Per RFC 8754 §2: "The Segment List is encoded starting from the last segment of the SR Policy." That means Segment List[0] is the **final destination** (last segment), and Segment List[n] is the **first hop** (active segment, already loaded into the IPv6 DA). Processing advances from index n downward to 0. The list is written by the headend; endpoints only advance the pointer.

**Segments Left** — A pointer into the Segment List. When Segments Left=0, the packet is at the final segment endpoint and delivered to the inner payload. This counter cannot be manipulated in transit without breaking the forwarding.

**HMAC TLV** — Optional extension defined in RFC 8754 §2.1.2. Provides cryptographic integrity over the SRH. The headend signs the SRH with an HMAC key shared with segment endpoints. Any modification to the SRH in transit invalidates the HMAC. If the endpoint verifies HMAC and rejects failures, the SRH is tamper-resistant. If the endpoint does not verify (or the HMAC TLV is absent), there is no integrity protection.

## The Trust Boundary Model — RFC 8754

RFC 8754 §5.1 ("Securing the SR Domain") is the key normative reference for SRv6 security. The actual text:

> *"Nodes outside the SR domain are not trusted: they cannot directly use the SIDs of the domain. This is enforced by two levels of access control lists:*
> *1. Any packet entering the SR domain and destined to a SID within the SR domain MUST be dropped. This is implemented via an access control list at the domain boundary that drops packets with a destination address in the SID block of the domain.*
> *2. This distributed protection is complemented by per-node protection: each node MUST drop packets destined to one of its SIDs when the source address is not in the SR domain."*

This is an important nuance: the RFC's **primary** mechanism is **destination-based filtering** at the domain boundary — drop traffic destined to your SID block from outside your domain. The source-address check is a **per-node** second layer that complements the boundary filter. Operationally, both matter:

**Domain boundary:** Filter on destination address — drop packets with DA in your SRv6 locator block that arrive from external interfaces. This is the primary control.

**Per-node:** Each SRv6-capable PE validates that packets destined to its local SIDs come from authorized sources. This catches attacks that bypass the boundary (compromised internal hosts, rogue devices).

The RFC establishes two categories of SRv6 traffic:

**Trusted domain traffic** — SRv6 packets sourced from within your administrative domain. You've assigned the SID block. You control the SRH-capable nodes. Traffic arriving from within the domain and destined to a domain SID is expected and legitimate.

**Untrusted traffic** — Traffic arriving from outside your domain and destined to your SID block. These MUST be dropped at domain edges per RFC 8754 §5.1.

**What "untrusted" means operationally:**

The RFC says drop it. Operationally, you have to decide where "the edge" is. Four models in practice:

| Model | Where to Filter | Pros | Cons |
|-------|----------------|------|------|
| **Customer edge** | CE-facing ports | Maximum protection | Most ACL entries to maintain |
| **Internet edge** | Upstream/peering ports | Protects from external attack | Internal abuse still possible |
| **Domain edge** | SRv6-domain boundary | Matches RFC intent | Must define domain clearly |
| **Nothing** | (no filtering) | Easy | Exposed to all injection attacks |

The right answer is customer edge AND internet edge. Both. Defense in depth.

**The gap between RFC and reality:**

RFC 8754 tells you what to do. It does not tell you:
- What ACL syntax to use on your hardware (it varies by vendor and ASIC generation)
- Whether your hardware can inspect IPv6 extension headers at line rate (it often cannot)
- Whether your firewall's IPv6 inspection engine understands RH Type 4 (many don't)
- What to do about IPv6 transit peers who originate legitimate SRv6 traffic

Every SP deploying SRv6 has to answer these questions in their specific environment. The RFC gives you the policy. Your hardware gives you the capability. The gap between them is your risk exposure.

## Attack Scenarios

### Scenario 1: SRH Injection — Source Routing Through Backbone Nodes

**Setup:**  
An attacker has IPv6 connectivity to your network (customer port, peering, compromised host inside). They craft a packet with a handwritten SRH containing SIDs of your internal backbone nodes. Their goal: route traffic through nodes they shouldn't reach, observe or manipulate traffic at those nodes, or establish a forwarding path that bypasses security controls.

**Attack packet:**
```
Outer IPv6:
  Source: 2001:db8:atk::/48   (attacker-controlled address)
  Destination: fc00:0:pe3:e001::  (SID at PE3 — attacker wants to hit this node first)
  Next Header: 43 (Routing Header)

SRH:
  Routing Type: 4
  Segments Left: 2
  Segment List:
    [0] fc00:0:pe7:dt4::100:1  (final delivery — END.DT4 at PE7, table 100)
    [1] fc00:0:pe5:x001::      (END.X at PE5, steer toward different egress)
    [2] fc00:0:pe3:e001::      (END at PE3 — already in DA)
```

**What happens if you're not filtering:**  
PE3 receives this packet. The SID `fc00:0:pe3:e001::` is in PE3's SID table. PE3 processes it — decrements Segments Left, sets DA to `fc00:0:pe5:x001::`, and forwards toward PE5. PE5 processes `END.X`, steers via the specified adjacency. PE7 processes `END.DT4`, decapsulates, and delivers to VRF table 100.

The attacker has just routed through three backbone PEs, through a VPN table they shouldn't have access to, by writing those SIDs into an SRH. Without filtering, your SID space is reachable from anywhere with IPv6 connectivity.

**What should stop this:**  
Per RFC 8754 §5.1 Level 1: domain boundary ACL drops any packet with DA in your SRv6 SID block (`fc00::/32` in this example) arriving from external interfaces. The attacker's packet has DA `fc00:0:pe3:e001::` — that's in your SID block, arriving from a customer port. Drop. Level 2 (per-node): PE3 itself drops traffic to its local SIDs from sources outside the SR domain.

---

### Scenario 2: SID Enumeration and Topology Mapping

**Attack:** Even without injecting SRH, an attacker with IPv6 connectivity can probe your SID allocation. SIDs are advertised via IS-IS/OSPFv3 extensions — they're reachable IPv6 prefixes. An attacker can:
1. Passively receive routing updates if they have iBGP/IGP access (route server, looking glass)
2. Probe the SID block iteratively (fc00:0:X:func:: for all X) and observe ICMP responses
3. Map your backbone topology: which nodes exist, how many SID types per node, what functions are exposed

This is reconnaissance. The SID block structure often reveals:
- Node count (how many /48s under your SRv6 block are active)
- Function count per node (how many END variants each node has)
- VPN tables (END.DT4/DT6 functions reveal VRF table IDs)

**Mitigation:**  
This is harder to fully prevent because the SIDs must be routable. Best practice:
- Don't expose SID blocks in public routing (BGP communities to control export)
- Use opaque or randomized function assignments (don't use sequential fc00:0:1:1::, fc00:0:1:2::, etc.)
- Implement RTBH (Remote Triggered Blackhole) for probe sources
- Monitor for scanning patterns: large volumes of ICMP toward SID block addresses

---

### Scenario 3: SRH-Based Firewalling Bypass

**Setup:**  
A firewall sits in the path between two network segments. It inspects IPv6 traffic and enforces a policy: traffic from Zone A to Zone B must use explicit permitted paths. An attacker in Zone A crafts an SRH that routes traffic around the firewall — through another SRv6 node that connects A to B without the firewall in the path.

```
Normal path: Zone A host → Firewall → Zone B host  (BLOCKED by FW policy)
SRH path:    Zone A host → SRH[PE4, PE5] → Zone B host  (Firewall never sees it)
```

**Why this works (or used to work):**  
Firewalls that don't inspect the SRH assume the IPv6 DA is the final destination. They see a packet from Zone A to `fc00:0:pe4:e001::` — which might be an "allow" entry (PE4 is inside the network). The packet routes to PE4, which processes the SRH and steers to PE5, which delivers to Zone B. The firewall never evaluated the actual delivery destination.

**Why this is worse than it sounds:**  
End-to-end SRv6 paths can bypass not just firewalls, but any middlebox — IDS/IPS, DLP, traffic inspection, lawful intercept — if those middleboxes don't understand SRH semantics. If your security architecture assumes "all traffic from A to B passes through the inspection point," SRv6 can violate that assumption.

**Mitigation:**  
1. Firewalls in SRv6 paths must understand SRH — they must chase the full segment list to determine actual delivery destination, not just the IPv6 DA
2. If your firewall cannot inspect SRH, it should block RH Type 4 at the segment boundary (fail-closed, not fail-open)
3. Audit your traffic inspection assumptions: if any inspection point assumes path coverage, verify that SRv6 paths cannot bypass it

---

### Scenario 4: Transit Node Exposure — The "Invisible" Forwarding Problem

This scenario is unique to SRv6 and has no direct SR-MPLS analog.

**RFC 8754 §3.2 (Transit Node)** explicitly describes this as a design characteristic: SRv6 can traverse nodes that are IPv6-capable but SRv6-unaware. Those nodes see normal IPv6 traffic. They forward it based on the Destination Address (the active SID). They do not process the SRH. Segments Left never decrements at these nodes.

**The security implication:**  
A "transparent transit" node:
- Has no idea it's in an SRv6 path
- Applies no SRv6-specific filtering (it doesn't know any is needed)
- Cannot log SRH contents (it doesn't know they exist)
- Cannot enforce SID validity (it never looks at the SID)

If an attacker routes SRv6 traffic through a transit device that happens to connect two otherwise-isolated segments, that device has no mechanism to prevent it. Its IPv6 forwarding table says "send this toward the DA" — and that's what it does.

**Practical impact:**  
This matters most in network designs where IPv6 routing and SRv6 routing are not identical. If you have a path that exists in IPv6 (because you're dual-stack for operations) but you never intended SRv6 to use it, SRv6 will use it if a SID is reachable via that path. Your SRv6 traffic engineering topology is a subset of your IPv6 forwarding topology. Misaligned, they create unexpected forwarding paths.

**Mitigation:**  
- Ensure SRv6 locator prefixes are only advertised in the SRv6 domain (not leaked into general IPv6 routing)
- Audit IPv6 reachability between segments: any IPv6 path to a SID is an SRv6 path
- Consider explicit prefix distribution controls: SID block redistribution only within trusted IGP instances

## Infrastructure ACL Design for SRv6

Infrastructure ACLs (iACLs) protect the control and management plane by filtering traffic directed at the router itself. In SRv6, this extends to protecting SID processing — you don't want arbitrary sources triggering SID lookups on your PEs.

### ACL Structure — Protecting SRv6 Nodes

Per RFC 8754 §5.1, the primary mechanism is **destination-based**: drop traffic arriving from outside your domain that is destined to your SID block. The secondary mechanism is source-based: drop traffic from non-domain sources destined to local SIDs (per-node protection).

These two mechanisms translate to ACLs applied at domain boundary interfaces:

**IOS-XR — IPv6 iACL implementing RFC 8754 §5.1 filtering:**

```
! Your SRv6 locator/SID block — all SIDs assigned to your domain
! Replace fc00::/32 with your actual SRv6 allocation
ipv6 prefix-list SRV6_DOMAIN_SIDS seq 10 permit fc00::/32 le 128

! Domain source prefixes — authorized SRv6 headends/sources
ipv6 prefix-list SRV6_DOMAIN_SOURCES seq 10 permit fc00::/32 le 128

! PRIMARY control per RFC 8754 §5.1 Level 1:
! Drop traffic DESTINED to SIDs from non-domain sources
! This ACL applied inbound on external-facing (customer/peering) interfaces
ipv6 access-list SRV6_DOMAIN_BOUNDARY_INBOUND
  !
  ! Permit ICMP (path MTU discovery, ICMPv6 NDP)
  10 permit icmp any any
  !
  ! Deny traffic DESTINED to our SID block from outside sources
  ! (RFC 8754 §5.1 Level 1 — primary boundary protection)
  20 deny ipv6 any fc00::/32
  !
  ! Permit all other IPv6
  30 permit ipv6 any any

! SECONDARY control per RFC 8754 §5.1 Level 2:
! Per-node — also block SRH-bearing traffic from non-domain sources
! This is defense-in-depth, not the primary control
ipv6 access-list SRV6_PER_NODE_INBOUND
  !
  10 permit icmp any any
  !
  ! Permit SRv6 from domain sources (legitimate intra-domain traffic)
  20 permit ipv6 fc00::/32 any routing-type 4
  !
  ! Deny SRH-bearing traffic from non-domain sources
  30 deny ipv6 any any routing-type 4
  !
  40 permit ipv6 any any

! Apply BOUNDARY ACL on external-facing interfaces
interface TenGigE0/0/0/0.100
  ipv6 access-group SRV6_DOMAIN_BOUNDARY_INBOUND ingress
```

**Key caveat:** The `routing-type` keyword for IPv6 ACLs in IOS-XR is hardware- and release-dependent. Cisco documentation for this specific keyword is inconsistent across releases; the keyword name may differ (`routing-type`, `extension-header`, or `ipv6-extension-header` depending on release). The destination-based filter (`deny ipv6 any fc00:0::/32`) in the boundary ACL is more reliable and matches the RFC 8754 §5.1 Level 1 primary mechanism. Always validate ACL behavior with a known-deny test pattern before trusting it in production.

**Junos — IPv6 iACL with SRH filtering:**

```
# Firewall filter to control SRH-bearing IPv6 traffic
# Applied at the routing-instance or interface level

policy-options {
    prefix-list SRV6_DOMAIN_SOURCES {
        fc00::/32;
    }
    prefix-list SRV6_AUTHORIZED_PEERS {
        /* Add authorized transit/peer SRv6 sources here */
    }
}

firewall {
    family inet6 {
        filter SRV6_IACL_INBOUND {
            interface-specific;

            term PERMIT_ICMP {
                from {
                    next-header icmpv6;
                }
                then accept;
            }

            term PERMIT_SRV6_FROM_DOMAIN {
                from {
                    source-prefix-list {
                        SRV6_DOMAIN_SOURCES;
                    }
                    extension-header routing-header;
                    routing-header-type 4;
                }
                then accept;
            }

            term DENY_SRH_UNTRUSTED {
                from {
                    extension-header routing-header;
                    routing-header-type 4;
                }
                then {
                    discard;
                    count SRH_UNTRUSTED_DROPS;
                    syslog;
                }
            }

            term PERMIT_ALL_OTHER {
                then accept;
            }
        }
    }
}

interfaces {
    xe-0/0/0 {
        unit 100 {
            family inet6 {
                filter {
                    input SRV6_IACL_INBOUND;
                }
            }
        }
    }
}
```

**Important Junos caveat:** The `extension-header routing-header` and `routing-header-type` match conditions are supported on MX series with Trio chipset (MPC3/4/5/6/7/10). On older MPCs or EX/QFX platforms, extension header matching is not supported in hardware and the filter will not compile. Always verify platform capability before deploying. Junos will return an error at commit if the hardware doesn't support the match condition — which is better behavior than silently ignoring it.

### Protecting the SID Block — Destination-Based Filtering

Beyond source-based filtering at ingress, consider filtering based on the destination SID block at peering and upstream handoffs. You don't want externally-originated traffic with your SID block in the Destination Address (indicating they're trying to trigger SID processing on your nodes).

**IOS-XR — Block SID-destined traffic from non-domain sources at upstream peers:**

```
! Your SRv6 locator block
ipv6 prefix-list MY_SRV6_LOCATOR_BLOCK seq 10 permit fc00:0::/32 le 128

! On upstream eBGP-facing interfaces — deny traffic toward your SID block from outside
ipv6 access-list UPSTREAM_INGRESS_FILTER
  ! Permit traffic FROM your SID block (your traffic going out and coming back)
  10 permit ipv6 fc00:0::/32 any
  ! Deny traffic TO your SID block from outside (not your sources)
  20 deny ipv6 any fc00:0::/32
  ! Permit everything else
  30 permit ipv6 any any
```

This is conceptually similar to Unicast Reverse Path Forwarding (uRPF) for SRv6 — don't accept traffic destined to your SID space from sources outside your domain.

### HMAC TLV Deployment

RFC 8754 §2.1.2 defines the HMAC TLV as an optional integrity mechanism. When deployed:
- The headend computes an HMAC over the SRH (using a shared key between headend and endpoint)
- The endpoint verifies the HMAC before processing the SID
- Any modification of the SRH in transit (adding, removing, or changing segments) invalidates the HMAC

**Why it's not widely deployed:**

1. **Key distribution** — HMAC requires every headend to share keys with every segment endpoint. At SP scale with hundreds of PEs, this is a key management problem. IPsec has ISAKMP/IKE. SRv6 HMAC has no equivalent automated key distribution mechanism. You're looking at manual key deployment or a custom KMS integration.

2. **Performance cost** — Computing and verifying HMAC at line rate on every SRv6 packet is expensive. ASIC support for HMAC TLV is not universal.

3. **Not always needed** — If your SRH filtering is solid (untrusted SRH dropped at domain edges), HMAC is defense-in-depth rather than the primary mechanism. Most SP deployments prioritize filtering over HMAC and treat HMAC as aspirational.

**Where HMAC makes sense:** High-security environments where you also have SRv6-based service chaining (SFC) that routes traffic through security appliances, and you need assurance that the service chain hasn't been tampered with. HMAC provides that assurance.

## Vendor Implementation Differences

SRv6 security support varies meaningfully across vendors. This section is honest about where the gaps are.

### Cisco IOS-XR

**SRH filtering support:** Yes, with caveats
- `ipv6 access-list ... routing-type 4` syntax is supported on NCS 5500 series (with Jericho/Jericho+/Jericho2 ASICs) and ASR 9000 with 64-bit line cards
- On older ASR 9000 line cards (Typhoon, Trident), extension header matching is software-only or unsupported
- Verify hardware support by testing with a known-deny pattern on a lab device before deploying to production

**SID authorization:** IOS-XR only processes SIDs that are locally instantiated in the SID table — packets with SIDs matching the locator prefix but not bound to a configured function are not SRH-processed (they're forwarded as plain IPv6 toward the DA). This is inherent behavior, not an explicit configuration. Verify your SID table is complete and no unintended functions are instantiated.
- `show segment-routing srv6 sid` shows all instantiated SIDs — audit this periodically
- SIDs not in this table will be forwarded as IPv6, not processed as SRH segments

**uSID hardware support:** Present on NCS 5500, NCS 540, and ASR 9000-X64 (IOS-XR 7.4+)
- uSID compresses multiple SIDs into a single 128-bit IPv6 address
- Security implication: SIDs are harder to inspect (they look like unusual IPv6 addresses); syslog/flow records may not decode them correctly; some ACL match conditions operate on the full 128-bit address and may miss uSID packets if the prefix doesn't align

**Useful verification commands:**
```
! Check SRv6 SID table — what SIDs this node will process
show segment-routing srv6 sid

! Check locator binding — what blocks are advertised
show segment-routing srv6 locator

! Verify ACL is being applied and matched
show access-lists ipv6 SRV6_IACL_INBOUND hardware ingress interface TenGigE0/0/0/0

! Check SRv6 global configuration including HMAC key config (if deployed)
show segment-routing srv6
```

---

### Juniper Junos (MX Series)

**SRH filtering support:** Yes, on Trio-based MPCs
- `extension-header routing-header; routing-header-type 4;` supported on MPC3E-3D, MPC4E, MPC5E, MPC6E, MPC7E, MPC10E
- EX/QFX: no extension header matching support in hardware on most generations
- PTX: SRv6 SRH forwarding support added in Junos 20.2, iACL extension header matching varies by FPC

**SRH source validation:** Supported via firewall filter with prefix-list source matching
- Junos does not have a dedicated SID source validation configuration knob; the RFC 8754 §5.1 Level 2 control is implemented through firewall filters on SRv6-active interfaces (as shown in the ACL section above)

**SRv6 policy with authorization:** Junos implements `seg-routing-mpls` and `source-packet-routing` under `protocols isis` — SRv6 locators are bound to IS-IS. Unauthorized function values will not match the SID table and will be forwarded as plain IPv6 (not SRH-processed), which is safe behavior but means they don't fail loudly.

**Useful verification commands:**
```
# Show SRv6 SID table (command name varies by Junos release — check release notes)
show spring srv6 sid-table                # Junos 20.x+
show route table inet6 protocol sr-srv6  # fallback: check routes associated with SRv6 locators

# Show firewall filter match counts (for ACL verification — this syntax is stable)
show firewall filter SRV6_IACL_INBOUND

# Show interfaces with the filter applied
show interfaces xe-0/0/0.100 detail | grep filter

# Check SRH counters via traffic statistics on SRv6-active interfaces
show interfaces xe-0/0/0 statistics detail
```

### Firewalls and Middleboxes — The Honest Assessment

Most production firewalls do not understand SRv6 SRH semantics in the way they need to for security enforcement. The landscape as of 2026:

**Palo Alto Networks (PAN-OS):**  
PAN-OS inspects IPv6 but its SRv6 awareness is limited. As of PAN-OS 11.x, PAN can parse SRH headers in application identification, but deep SID-based policy enforcement (e.g., "policy decision based on final SID destination, not DA") is not a first-class feature. SRH filtering can be achieved via custom application filters or zone protection profiles targeting RH Type 4, but this requires validation in your specific deployment. Official PAN support documentation is sparse.

**Fortinet FortiGate:**  
FortiOS IPv6 inspection can identify extension headers. SRH-specific policy enforcement (decision-based on SRH segment list) is limited. FortiGate can block routing headers broadly via IPv6 extension header restrictions in zone policies. This is a blunt instrument — blocks all RH types, not just RH4 — but if your use case doesn't need RH Type 2 (MIPv6 Home Address option), blocking all routing headers is a defensible policy.

**The general principle:**  
Any firewall in an SRv6 path that cannot inspect the full SRH is fail-open for SRv6-based bypasses. Two choices:
1. Ensure the firewall inspects SRH and enforces policy based on the full segment list
2. Block RH Type 4 on the interfaces facing untrusted segments (fail-closed)

If you can't verify which behavior your firewall implements, assume fail-open and block RH4 at the perimeter.

## SRv6 vs SR-MPLS: The Honest Security Tradeoff

This comparison matters for SP architecture decisions. People often say "SRv6 is more flexible" without discussing what that flexibility costs in security. Here it is:

| Aspect | SR-MPLS | SRv6 | Winner |
|--------|---------|------|--------|
| **Ethertype visibility** | Unique (`0x8847`), filterable everywhere | Same as IPv6 (`0x86DD`), invisible to most filters | SR-MPLS |
| **Label space scope** | Domain-local. Labels only meaningful within MPLS domain | SIDs are globally routable IPv6 addresses | SR-MPLS |
| **Injection attack surface** | Attacker needs MPLS access to inject label stacks | Attacker needs IPv6 access (much more common) | SR-MPLS |
| **Firewall support** | Universal. Every firewall understands MPLS | Sparse. Few firewalls understand SRH semantics | SR-MPLS |
| **Transit node security** | Non-MPLS nodes drop MPLS traffic (no forwarding) | Non-SRv6 nodes forward SRH as IPv6 (transparent) | SR-MPLS |
| **Hardware ACL support** | Widespread (label matching well-supported) | Limited (extension header matching varies by ASIC) | SR-MPLS |
| **HMAC protection** | N/A (label scope provides inherent isolation) | Available (RFC 8754 §2.1.2) but rarely deployed | Neutral |
| **SID/label distribution** | IS-IS/OSPF TLVs; routed within domain | IS-IS/OSPFv3 TLVs; SIDs are globally routable | SR-MPLS |
| **Operational simplicity** | Single protocol stack (IP + MPLS) | Single protocol stack (pure IPv6) | Neutral |
| **DPI and forensics** | Well-understood by existing tools | SRH parsing not universal; uSID adds opacity | SR-MPLS |

**The honest summary:** SR-MPLS has fundamentally better inherent security properties because the label space is domain-local. You cannot inject an MPLS label from outside the MPLS domain and have it processed — there's no IP routing path to an MPLS label. SRv6 SIDs are IPv6 addresses — and if you have IPv6 reachability, you can try to hit them.

This doesn't mean SRv6 is insecure. It means SRv6 requires explicit security design that SR-MPLS gets partly for free. The firewall, ACL, and monitoring investments required for a secure SRv6 deployment are higher than for SR-MPLS. That cost is worth it for deployments that genuinely need what SRv6 offers (pure IPv6, programmability, no MPLS complexity). It's not worth it if MPLS works fine for your use case.

No marketing material will tell you this. Now you know.

## Recommended Security Architecture

A production SRv6 backbone with a defensible security posture needs all of the following. Missing any one of these is a gap.

### Layer 1: Domain Edge Filtering

Per RFC 8754 §5.1, every external-facing interface must implement two levels of control:

**Level 1 (Primary — destination-based):** Drop any packet arriving from outside the domain that is destined to an address within your SRv6 SID block. This is the boundary control — your SIDs should never be reachable from external networks.

**Level 2 (Secondary — per-node source+destination):** Each SRv6-capable node drops packets destined to its local SIDs when the source is not within the SR domain. This catches attacks that bypass the boundary (compromised internal hosts).

Both levels should also log drops for incident detection. The ACL example in the Infrastructure ACL section above implements both levels.

### Layer 2: SID Source Validation

Per RFC 8754 §5.1 Level 2, each SRv6-capable PE should drop packets destined to its local SIDs from sources outside the SR domain. This is implemented via per-interface iACLs or per-node firewall filters (see the ACL examples in the Infrastructure ACL section). Both platforms enforce this through ACL configuration — there is no single "enable SID source validation" knob. Apply the `SRV6_PER_NODE_INBOUND` ACL (or Junos equivalent) on all interfaces that may receive SRv6 traffic, not just external-facing ones.

This is defense-in-depth behind the edge ACL. If an attacker bypasses the edge (compromised internal host, rogue device), per-node ACLs are the second line.

### Layer 3: SID Allocation Hygiene

- Use a dedicated, non-overlapping SRv6 block separate from your operational IPv6 addresses
- Don't advertise the SRv6 block into public BGP — keep it internal (no-export community at a minimum)
- Assign opaque function values — don't use sequential integers that make enumeration trivial
- Maintain a SID registry (spreadsheet at minimum, IPAM integration ideally) — know what every SID does and who owns it
- Audit SID table periodically: SIDs should match your registry; unexpected SIDs indicate a misconfiguration or unauthorized allocation

### Layer 4: Monitoring and Detection

Build detection for the attack patterns described above:

**SRH injection detection:**
- Alert on SRH-bearing packets that match your drop ACL (source outside trust domain)
- Baseline normal SRH volume; alert on significant increases
- Monitor SID lookup failures: SIDs that arrive but don't match the local SID table indicate either misconfiguration or injection attempts

**Topology reconnaissance detection:**
- Monitor for ICMP probes sweeping your SRv6 locator block
- Alert on ICMP TTL-exceeded responses from backbone nodes destined to external addresses (may indicate traceroute-based mapping)

**Relevant telemetry paths:**
```
# IOS-XR — SRv6 SID counters
Cisco-IOS-XR-segment-routing-srv6-oper:srv6/active/locators/locator/sid-table

# IOS-XR — ACL counters (including SRH drops)
Cisco-IOS-XR-es-acl-oper:es-acl/active/oper-summary

# Junos — SRv6 SID statistics
/junos/spring/srv6/sid-table/entry

# Junos — Firewall filter counters (for SRH drop counting)
/junos/firewall/
```

### Layer 5: Incident Response Preparation

Before you need it, prepare:

1. **Packet capture tooling for SRH:** Confirm that your capture tools (Wireshark, tcpdump, network TAPs) decode SRH correctly. Wireshark has supported RH Type 4 since version 3.4. tcpdump displays extension headers but doesn't label SRH semantics by default. Test before the incident.

2. **SRH-aware forensics:** If you have an incident involving SRv6 forwarding, you need to trace the segment list. Establish which commands give you SID forwarding history and path verification:
   ```
   # IOS-XR — trace SRv6 packet
   traceroute srv6 policy name <policy-name>
   
   # Junos — SRv6 path verification
   ping routing-instance default fc00:0:pe7:dt4::100:1 routing-options source fc00:0:pe1:e001::
   ```

3. **Document your SID-to-function mapping:** During an incident, you'll be reading SIDs in packet captures. If you don't have a lookup table for `fc00:0:pe7:e010::` → "END.DT4 at PE7, VRF customer-100," you're decoding hex during an outage. Pre-build this reference.

## Review Questions

**1.** RFC 8754 §5.1 requires dropping SRH-bearing packets from untrusted sources. Your team deploys SRH filtering on upstream-facing interfaces only (peering and transit), not on customer-facing interfaces. A customer is running SRv6 in their own network and sends SRv6-encapsulated traffic toward your backbone. Describe what happens if this is not filtered, what attack scenarios this opens, and how you would design the filtering policy to permit legitimate customer SRv6 while blocking SRH injection.

**2.** You are deploying SRv6 in a brownfield network that already has Palo Alto firewalls performing full inspection on all inter-zone IPv6 traffic. Your security team says "the firewalls handle IPv6 security." Why is this assertion potentially incorrect for SRv6? What specific capability would the firewalls need to have, and what is the fail-open risk if they lack it?

**3.** Compare the SRH injection attack surface in SRv6 with the equivalent in SR-MPLS. In what specific scenario could an attacker inject SR-MPLS label stacks from outside the MPLS domain? Why does this scenario not apply in a well-architected MPLS deployment, and why does the equivalent SRv6 scenario require additional controls beyond routing architecture?

**4.** Your SRv6 deployment uses uSID (micro-segment IDs) to compress multiple SIDs into a single 128-bit address. Your existing IPv6 ACL infrastructure uses prefix-list-based source matching. Describe two specific ways that uSID complicates your existing SRH filtering ACLs and how you would address each.

**5.** A post-incident review reveals that an internal host (compromised by malware) originated SRH-bearing packets that were processed by backbone PEs, forwarding traffic through VPN tables the host should not have accessed. Domain edge filtering was in place on all customer-facing interfaces. Explain how the attack succeeded despite the edge filtering, what additional controls would have caught or prevented it, and what monitoring you would implement to detect this pattern going forward.
