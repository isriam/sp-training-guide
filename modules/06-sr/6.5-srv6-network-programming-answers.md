# 6.5 — SRv6 Network Programming — Answer Key

## Question 1
> Your SRv6 backbone uses uSID with /32 blocks. A new site requires traffic to traverse 8 intermediate nodes before reaching the egress PE. How many carrier SIDs do you need in the SRH, and what's the overhead? What if you used standard (non-uSID) SRv6?

### Answer

### uSID calculation

With a **/32 block** and **16-bit micro-SIDs**, each 128-bit carrier SID holds:
- 32 bits for the block (locator prefix)
- 96 bits remaining ÷ 16 bits per uSID = **6 uSIDs per carrier**

You need to traverse **8 intermediate nodes** plus reach the **egress PE** = **9 SIDs total** (8 transit + 1 egress endpoint).

9 SIDs ÷ 6 uSIDs per carrier = **2 carrier SIDs** (first carrier holds 6, second holds 3, with 3 uSID slots zero-padded).

**SRH overhead with uSID:**
- IPv6 base header: 40 bytes
- SRH header: 8 bytes (fixed)
- Segment list: 2 carriers × 16 bytes = 32 bytes
- **Total SRv6 overhead: 40 + 8 + 32 = 80 bytes**

(If the outer IPv6 DA also serves as the first active segment, the SRH carries 2 entries but the active SID is in the DA — exact overhead depends on implementation, but 80 bytes is the right ballpark.)

### Standard SRv6 calculation

With standard (non-uSID) SRv6, each SID is a full **128-bit (16-byte)** address in the segment list.

9 SIDs × 16 bytes = **144 bytes** in the segment list alone.

**Total SRv6 overhead:**
- IPv6 base header: 40 bytes
- SRH header: 8 bytes
- Segment list: 144 bytes
- **Total: 40 + 8 + 144 = 192 bytes**

### Comparison

| Metric | uSID | Standard SRv6 |
|--------|------|---------------|
| Carrier SIDs in SRH | 2 | 9 |
| Segment list bytes | 32 | 144 |
| Total overhead | ~80 bytes | ~192 bytes |
| Payload space (1500 MTU) | ~1420 bytes | ~1308 bytes |

uSID reduces the overhead by roughly **58%** for this path. On a 1500-byte MTU link, that's the difference between fitting a reasonable payload and needing jumbo frames just to carry the headers.

**This is exactly why uSID exists.** Standard SRv6 at 8+ hops is impractical without jumbo MTU everywhere. uSID makes it viable.

## Question 2
> A customer wants to insert a firewall into their VPN traffic path using SRv6 service chaining. The firewall doesn't understand SRv6. Which endpoint behavior(s) would you use, and what are the trade-offs between End.AD and End.AM?

### Answer

Both **End.AD (Dynamic Proxy)** and **End.AM (Masquerading)** solve the same problem: inserting an SRv6-unaware service function (firewall) into an SRv6 service chain. The difference is how they handle the SRH while the packet visits the firewall.

### End.AD (Endpoint with Dynamic Proxy)

**How it works:**
1. Packet arrives at the SRv6 proxy node with End.AD SID
2. Proxy **caches the SRH** in local state (keyed by flow — typically 5-tuple)
3. Proxy **strips all SRv6 headers**, sends a bare IPv4/IPv6 packet to the firewall
4. Firewall processes the packet normally (inspect, NAT, filter, whatever)
5. Firewall returns the packet to the proxy
6. Proxy **restores the cached SRH** and continues SRv6 forwarding

**Pros:**
- Firewall sees a completely clean packet — no SRv6 artifacts
- Works with ANY firewall, regardless of IPv6 extension header handling
- Firewall can modify packet contents (NAT, payload rewrite) without breaking SRv6

**Cons:**
- **Per-flow state on the proxy** — the proxy must maintain a cache mapping each flow to its SRH. At scale (millions of flows), this is memory-intensive
- Breaks the "stateless core" principle of SR
- Flow cache timeout/eviction can cause packets to be dropped if the cache entry expires before the flow completes
- Bidirectional flows need state in both directions

### End.AM (Endpoint with Masquerading)

**How it works:**
1. Packet arrives with SRv6 headers
2. Proxy **masquerades** the packet — swaps the IPv6 DA to the firewall's address but keeps the SRH intact (repositioned as an extension header)
3. Firewall sees what looks like a normal IPv6 packet, processes it, returns it
4. Proxy **de-masquerades** — restores the DA from the SRH, continues forwarding

**Pros:**
- **No per-flow state** — the SRH travels with the packet through the firewall
- Scales better than End.AD (stateless proxy)
- Simpler implementation

**Cons:**
- **Only works if the firewall preserves IPv6 extension headers it doesn't understand.** Many firewalls strip or drop unknown extension headers as a security policy. If the firewall destroys the SRH, the packet is dead.
- Firewall must NOT modify the extension header area
- Less universal — requires testing with the specific firewall vendor/model

### Recommendation

| Scenario | Use |
|----------|-----|
| Firewall strips unknown IPv6 extension headers | **End.AD** (only option) |
| Firewall preserves extension headers, high flow count | **End.AM** (stateless, scales better) |
| Firewall does NAT or payload rewrite | **End.AD** (safer — clean packet to firewall) |
| Greenfield with SRv6-aware firewall available | **Neither** — use native SRv6 endpoint on the firewall |

**For most real-world firewalls today (Palo Alto, Fortinet, Check Point): End.AD.** Most enterprise firewalls strip unknown extension headers by default, making End.AM unreliable without vendor-specific testing.

## Question 3
> After migrating from SR-MPLS to SRv6 with uSID, VPN traffic works for most prefixes but one remote PE's routes are unreachable. BGP shows the route with an SRv6 SID, the SID is in the local table, but packets are dropped. The egress PE has mixed linecards (old and new). What's your diagnosis?

### Answer

**The old linecard on the egress PE doesn't support SRv6 in hardware.**

This is the classic mixed-hardware SRv6 failure. The symptoms are textbook:
- BGP route exists with SRv6 SID ✓ (control plane is fine)
- SID is in the local SID table ✓ (the router knows about it)
- Packets are dropped ✗ (forwarding plane can't process them)
- Most prefixes work ✓ (linecards serving those prefixes are new/capable)
- One remote PE's routes fail ✗ (traffic hits the old linecard)

### What's happening

The packet arrives at the egress PE with an SRv6 header. The router needs to:
1. Look up the SID in the Local SID Table
2. Execute the endpoint behavior (e.g., End.DT4 — decapsulate and route in VRF)
3. Forward the decapsulated packet to the CE

Steps 1-2 require **SRv6 hardware support on the ingress linecard** (the one receiving the SRv6 packet). If the packet arrives on a port belonging to an old linecard that doesn't have SRv6 ASIC support, it either:
- Gets punted to software (slow path, drops under load)
- Gets silently dropped (ASIC doesn't understand the SRH)
- Gets forwarded without processing the SRH (wrong behavior, eventual drop)

### Diagnosis steps

```
IOS-XR:
  show platform | include <old-linecard>
  ! Verify linecard type and firmware

  show segment-routing srv6 sid <sid> detail
  ! Check: is the SID programmed on ALL linecards?
  ! Some platforms program SIDs only on capable hardware

  show cef ipv6 <sid> detail hardware location <linecard>
  ! Compare hardware FIB on old vs new linecard

  show controllers np counters all location <old-linecard> | include drop|srv6|punt
  ! Look for SRv6-related drops on the old linecard
```

### Fix options

1. **Move the affected VPN interface to a port on the new linecard** — quickest fix, no hardware purchase
2. **Upgrade the old linecard firmware** — if a newer firmware adds SRv6 support for that ASIC
3. **Replace the old linecard** — the permanent fix
4. **Route traffic to avoid the old linecard** — use policy or interface assignment so SRv6 packets always arrive on capable hardware

### Key lesson

SRv6 hardware support is **per-linecard**, not per-chassis. A router can have mixed linecards where some support SRv6 and some don't. The control plane (BGP, IS-IS) doesn't know or care — it advertises SIDs regardless. The forwarding plane is where it breaks. Always verify hardware capability per linecard before migration, not just per platform.
