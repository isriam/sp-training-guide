# 6.3 — TI-LFA — Answer Key

## Question 1
> You enable TI-LFA on all routers, but `show isis fast-reroute summary` shows 85% coverage. What are three possible causes?

### Answer

TI-LFA promises "topology-independent" 100% coverage in theory, but 85% in practice means something is constraining the backup computation. Three likely causes:

**1. Node protection is enabled but some topologies don't allow it.**

If you configured `node-protection` (which is stronger than link-protection), TI-LFA tries to find a backup path that avoids both the failed link AND the next-hop node. In some topologies — especially stub or ring segments — there's no node-diverse path. The router can't build a repair segment list that avoids the node entirely.

Check:
```
IOS-XR: show isis fast-reroute summary
        ! Compare link-protection coverage vs node-protection coverage
        ! If link-protection = 100% but node-protection = 85%, that's your answer

Junos:  show isis backup coverage
```

**Fix:** Accept link-protection as fallback where node-protection isn't achievable. Most implementations do this automatically, but verify the configuration allows fallback.

**2. Backup label stack depth exceeds the platform's MSD (Maximum SID Depth).**

TI-LFA backup paths may require 2-4 labels (Adj-SID to avoid failure + Prefix-SID to PQ-node + Prefix-SID to destination). If the platform's hardware supports only 3-4 labels total (including the primary service labels), TI-LFA can't install the backup for paths requiring deeper stacks.

Check:
```
IOS-XR: show isis fast-reroute <prefix> detail
        ! Look at the repair segment list — how many labels?
        ! Compare to: show segment-routing traffic-eng topology | include MSD
```

**3. Some interfaces or prefixes are excluded from TI-LFA computation.**

Configuration issues:
- TI-LFA not enabled on all IS-IS/OSPF interfaces
- Specific prefixes excluded via policy
- Passive interfaces or loopbacks not participating in FRR computation
- Mismatched IS-IS levels (TI-LFA enabled on L2 but some adjacencies are L1-only)

```
IOS-XR: show run router isis | include ti-lfa
        ! Is it under the right address-family and level?

Junos:  show isis interface detail
        ! Check which interfaces have backup-spf-options enabled
```

**The 15% gap is almost always cause #1 (topology limitations for node-protection) or cause #3 (config coverage gap).** Cause #2 is rarer but shows up in older hardware.

## Question 2
> You're deploying TI-LFA in a network with 1,000 links. Each link needs BFD at 50ms. Your router CPU is at 40% baseline. What should you do to avoid CPU overload?

### Answer

1,000 links × BFD at 50ms intervals = a lot of BFD packets to process. Each BFD session sends/receives a packet every 50ms (20 packets/second per session). With a multiplier of 3, the router must track timing precisely to avoid false positives.

A central P router might terminate 20-40 BFD sessions. That's 400-800 BFD packets/second just for that one router. Not huge — but at 40% baseline CPU, you need to be careful.

**Recommendations:**

**1. Use BFD hardware offload — this is the real answer.**

Modern platforms (ASR 9000, MX Series, NCS 5500+) offload BFD to the line card or ASIC. The main CPU never sees BFD packets. This eliminates the CPU concern entirely.

```
IOS-XR: bfd hardware-offload   (platform-dependent)
Junos:  set protocols bfd interface <> hardware-offload  (or it's automatic on supported LCs)
```

If hardware offload is available, use 50ms intervals freely. The CPU impact is near zero.

**2. If hardware offload is NOT available, use tiered BFD intervals.**

Not all links are equally critical:
- **Core P-to-P links (critical):** BFD at 50ms × 3 = 150ms detection
- **PE-to-P edge links (important):** BFD at 100ms × 3 = 300ms detection
- **Low-priority / redundant links:** BFD at 300ms × 3 = 900ms detection, or no BFD at all

This reduces the total BFD packet rate significantly while keeping fast detection where it matters most.

**3. Monitor CPU impact during rollout.**

Don't deploy BFD on all 1,000 links at once. Roll out in stages:
- Stage 1: Core links (maybe 100-200)
- Monitor CPU for 24h
- Stage 2: PE-facing links
- Stage 3: Everything else

```
IOS-XR: show processes cpu | include BFD
        show bfd session | count
Junos:  show system processes extensive | match bfd
        show bfd session summary
```

**4. Ensure BFD echo mode is disabled on high-session-count routers.**

BFD echo mode sends packets that are looped back by the neighbor. It can reduce detection time but doubles the BFD packet rate. At scale, disable echo and rely on async mode.

**Bottom line:** Hardware offload is the answer. If you have it, 50ms BFD on 1,000 links is trivial. If you don't have it, tier your intervals by link criticality and monitor CPU during rollout.

## Question 3
> During a link failure, traffic is blackholed for 3 seconds instead of 50ms. BFD is configured at 50ms intervals. What are the top 3 things to check?

### Answer

3 seconds of blackhole with BFD at 50ms means something is fundamentally broken in the fast-failure pipeline. BFD at 50ms × 3 = 150ms detection. TI-LFA switchover should be near-instant after detection. 3 seconds means the fast path didn't work.

**1. Is BFD actually running on that specific interface?**

BFD may be configured globally but not active on the failed link. The adjacency might be using IGP hello-based detection instead (IS-IS default dead timer = 30s × multiplier, OSPF dead = 40s).

```
IOS-XR: show bfd session interface <failed-interface>
        ! Is there an active BFD session? State: Up?
        ! If no session → BFD isn't actually protecting this link

Junos:  show bfd session interface <failed-interface>
        ! Same check — is there an active session?
```

If BFD isn't running on that link, detection falls back to IGP timers — which explains 3 seconds (could be even longer with default timers).

**2. Is BFD running in software instead of hardware offload?**

Software-based BFD can miss its timers under CPU load. If the router was under stress during the failure event (RSVP re-signaling, BGP convergence, etc.), BFD running in software may have been delayed, effectively extending the detection time well beyond 150ms.

```
IOS-XR: show bfd session detail | include Offload
        show bfd counters | include delayed
```

If BFD is in software and CPU was spiking, that's your 3 seconds.

**3. Is TI-LFA backup path actually installed in the FIB?**

BFD may have detected the failure quickly (150ms), but if TI-LFA didn't have a pre-computed backup path installed in hardware, the router detected the failure and then had to wait for:
- IGP convergence (SPF computation + RIB + FIB)
- New best path calculation
- Hardware programming

That full convergence pipeline is typically 1-3 seconds — which matches your symptom exactly.

```
IOS-XR: show isis fast-reroute <prefix-behind-failed-link> detail
        ! Was there a backup path installed BEFORE the failure?
        ! If "No backup" → TI-LFA didn't protect this prefix

Junos:  show route <prefix> detail
        ! Check for "backup next-hop" entry
```

**Summary of the 3-second failure:**
- BFD not running → IGP detection at ~3s ✓
- BFD in software under CPU load → delayed detection ✓
- BFD detected fast but no TI-LFA backup → full convergence at ~3s ✓

Check all three. The fix depends on which one broke.
