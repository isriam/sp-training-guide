# 6.7 — Flexible Algorithm (FlexAlgo)

## Overview

FlexAlgo — Flexible Algorithm — is the feature that turns a single physical backbone into multiple logical networks, each with different forwarding behavior, running on the same hardware. One algorithm optimizes for lowest latency. Another avoids a set of links you're about to maintain. A third provides path diversity by excluding shared risk groups. All of them computed independently by the IGP, all of them coexisting on the same routers, all of them available as steering targets for VRFs and SR-TE policies.

If Segment Routing gave you source routing, and TI-LFA gave you fast protection, FlexAlgo gives you topology slicing. It's how you answer the question: "How do I give 200 different business segments different forwarding paths through the same backbone without building 200 separate networks?"

The concept is deceptively simple. IS-IS and OSPF already compute shortest paths using Algorithm 0 — the default SPF algorithm that considers IGP metric. FlexAlgo lets you define additional algorithms — numbered 128 through 255 — each with their own metric type, constraints, and topology rules. The IGP computes separate SPF trees for each algorithm. Traffic assigned to Algorithm 130 follows a completely different path than traffic assigned to Algorithm 0, even though both run on the same physical links and routers.

Cisco invented FlexAlgo and pushed it through the IETF as RFC 9350. Juniper supports it on MX and PTX. Nokia supports it on 7750. It works with both SR-MPLS and SRv6, which means it's available regardless of which data plane you choose. It's also vendor-interoperable — a Cisco NCS and a Juniper MX can participate in the same FlexAlgo domain as long as they both advertise support for the same algorithm.

## How FlexAlgo Works

### The Core Idea

Every router running IS-IS or OSPF computes a shortest-path tree using SPF. By default, this is Algorithm 0 — the standard metric-based computation that every networking engineer knows. All routers compute the same tree (assuming consistent link-state databases), and all traffic follows the same shortest paths.

FlexAlgo adds parallel computations. You define Algorithm 128 with the constraint "use minimum delay metric instead of IGP metric." Now every participating router computes a second SPF tree using delay as the cost. The resulting forwarding paths are completely different from Algorithm 0 — traffic follows the lowest-latency path, not the lowest-IGP-metric path.

You define Algorithm 129 with the constraint "exclude all links with affinity color RED." Every participating router computes a third SPF tree that doesn't include any RED links. If you color your links that share fiber conduit as RED, Algorithm 129 gives you paths that avoid shared-risk infrastructure.

Each algorithm gets its own set of Prefix-SIDs (in SR-MPLS) or SRv6 SIDs. When you want a VRF to use the low-latency topology, you steer it to Algorithm 128 SIDs. When you want a VRF to use the conduit-diverse topology, you steer it to Algorithm 129 SIDs. The physical network is the same. The forwarding behavior is different per algorithm.

This is the key insight: FlexAlgo is not an overlay. It's not tunneling. It's the IGP itself computing multiple realities on the same topology. The forwarding plane follows whichever reality you assign to each traffic class.

### Algorithm Definition and Advertisement

A FlexAlgo is defined by three things: a metric type, a set of constraints, and an algorithm number.

The **algorithm number** ranges from 128 to 255. Numbers 0 through 127 are reserved — 0 is the default SPF, and the rest are for IETF-defined algorithms. You get 128 custom algorithms, which is far more than any real deployment uses. Most production networks use 3 to 8.

The **metric type** determines what the SPF computation optimizes for:
- **IGP metric** — same as Algorithm 0, but with different constraints. Useful when you want the same metric-based shortest paths but with certain links excluded.
- **TE metric** — uses the traffic engineering metric advertised via IGP TE extensions. TE metrics can be set independently from IGP metrics, giving you a separate "cost dial" for FlexAlgo without affecting default routing.
- **Minimum unidirectional link delay** — uses the real-time delay measurements advertised via IGP. This is the latency-optimized algorithm. Routers measure one-way delay per link (using RFC 7810 / IETF performance measurement) and advertise it in IS-IS or OSPF. FlexAlgo computes the path with the lowest cumulative delay.

The **constraints** define which links are included or excluded from the algorithm's topology:
- **Include-any** — link must have at least one of the specified affinity colors
- **Include-all** — link must have all specified affinity colors
- **Exclude-any** — link must not have any of the specified affinity colors

Affinity colors are 32-bit values assigned to links. You define what each color means operationally. Color 1 might mean "submarine cable." Color 2 might mean "shared conduit with provider X." Color 3 might mean "high-bandwidth tier." The names are arbitrary — what matters is consistent assignment across the network.

The algorithm definition is advertised by one or more routers in the IGP domain. Typically, you configure the definition on a few designated routers — often the route reflectors or dedicated management nodes — and all other routers learn the definition from the IGP flooding. You don't need to configure the full algorithm definition on every router. You only need to configure each router's participation: "I support Algorithm 128" with the appropriate Prefix-SID or SRv6 locator.

This is important for operations: the algorithm definition is a single point of configuration, but it's distributed via the IGP. If the defining router goes down, other routers that have already learned the definition continue to use it. But if you need to change the definition, you change it on the defining router and it floods network-wide.

### Prefix-SID Allocation per Algorithm

In SR-MPLS, each algorithm gets its own Prefix-SID on each node. If Router 1 has Prefix-SID index 1 for Algorithm 0, it might have Prefix-SID index 1001 for Algorithm 128 and Prefix-SID index 2001 for Algorithm 129. The SRGB is shared — all algorithms use the same label range — but the SID indices are unique per algorithm.

In SRv6, each algorithm gets its own locator. Router 1 might have locator `fc00:0:1::/48` for Algorithm 0 and locator `fc00:0:1:1::/48` for Algorithm 128. The IGP advertises each locator with its associated algorithm, and the SRv6 SID table has entries for each.

When you build an SR-TE policy or steer a VRF, you specify which algorithm's SIDs to use. Traffic following Algorithm 128 SIDs takes the paths computed by Algorithm 128's SPF tree. Traffic following Algorithm 0 SIDs takes the default paths. Same router, same links, different forwarding behavior.

## Design Patterns

### Pattern 1: Low-Latency Topology

The most common FlexAlgo use case. Define an algorithm that uses minimum delay metric instead of IGP metric.

Algorithm 128: metric = min-unidirectional-delay, no constraints.

Every router computes the lowest-latency tree. Financial services, real-time communications, gaming — any VRF that cares about latency gets steered to Algorithm 128 SIDs. Everything else stays on Algorithm 0 with IGP metric.

The beauty of this pattern is that it requires zero link exclusions. The delay metric alone produces different paths. A 10G link across a continent has high delay but might have low IGP metric (if you set metrics based on bandwidth). Algorithm 0 would prefer it. Algorithm 128 would avoid it in favor of a shorter geographic path, even if that path has less bandwidth.

The requirement is that your routers actually measure and advertise link delay. On IOS-XR, this is done via performance measurement probes — TWAMP or PM delay measurement. On Junos, it's configured under `protocols isis interface` with delay measurement enabled. If delay isn't advertised, the algorithm can't compute — it falls back to IGP metric, defeating the purpose.

### Pattern 2: Maintenance Avoidance

Define an algorithm that excludes links tagged with a "maintenance" affinity color.

Algorithm 129: metric = IGP metric, exclude-any = color MAINTENANCE (bit 10).

Before a maintenance window, you tag the links you're about to work on with affinity color 10. All traffic steered to Algorithm 129 automatically avoids those links — the SPF tree is recomputed without them. After maintenance, remove the color and the topology returns to normal.

This is vastly simpler than the traditional approach of adjusting IGP metrics, draining RSVP-TE tunnels, or manually rerouting traffic. One affinity tag change, and the algorithm handles the rest.

The pattern extends naturally to SRLG avoidance. Tag all links in a shared risk group with the same color, define an algorithm that excludes that color, and traffic avoids the entire SRLG without any per-link metric manipulation.

### Pattern 3: Disjoint Path Diversity

Define two algorithms that use complementary affinity constraints to produce non-overlapping paths.

Algorithm 130: include-any = color EAST (bit 1). Only uses links tagged EAST.
Algorithm 131: include-any = color WEST (bit 2). Only uses links tagged WEST.

If you tag your east-coast links with color 1 and your west-coast links with color 2, Algorithm 130 produces paths entirely through the east coast and Algorithm 131 through the west. Assign your primary VRF traffic to Algorithm 130 and backup traffic to Algorithm 131, and you have geographic path diversity without any manual path computation.

This is how you solve the "diverse routing" requirement that compliance teams ask for. Instead of hoping the IGP computes diverse paths (it won't — it computes shortest paths, which are often the same path), you guarantee diversity through topology constraints.

### Pattern 4: Tiered Service Quality

Combine multiple algorithms for a service-tier model.

Algorithm 0: default best-effort. IGP metric, no constraints.
Algorithm 128: premium low-latency. Delay metric, no constraints.
Algorithm 132: bulk transfer. IGP metric, include-any = color HIGH-BW (bit 3).

VRFs get assigned to algorithms based on their service tier. Gold customers get Algorithm 128. Bulk data transfer VRFs get Algorithm 132, which only uses links tagged as high-bandwidth. Everyone else gets Algorithm 0.

This is what Cisco pitches as the killer feature for large enterprise backbones with hundreds of VRFs. Each VRF gets a service tier without any per-VRF traffic engineering. The algorithm does the work.

## VRF Steering with FlexAlgo

The whole point of FlexAlgo is steering traffic. There are two ways to assign a VRF to an algorithm.

### Method 1: BGP Color Extended Community + SR-TE Automated Steering

The VRF's routes are tagged with a BGP color extended community. SR-TE automated steering matches the color to an SR-TE policy, and the policy uses the algorithm's SIDs.

This is the most flexible approach because the color-to-algorithm mapping is done at the SR-TE policy level, not in the IGP. You can change which algorithm a VRF uses by changing the color community — no IGP changes needed.

### Method 2: Per-VRF Algorithm Assignment

On IOS-XR, you can assign an algorithm directly to a VRF's SRv6 locator. The VRF's traffic uses that algorithm's SID table. This is simpler than the color community approach but less flexible — changing the algorithm means reconfiguring the VRF.

### Method 3: ODN (On-Demand Nexthop) with FlexAlgo

Cisco's On-Demand Nexthop automatically instantiates SR-TE policies based on BGP color communities. When a VRF route arrives with a color, ODN creates a policy that uses the appropriate FlexAlgo SIDs. This is the most automated approach — you tag routes with colors in BGP, and ODN + FlexAlgo handle everything else.

## Operational Considerations

### Algorithm ID Management

Algorithm IDs are a global resource. Algorithm 128 means the same thing across your entire IGP domain. If one team defines Algorithm 128 as "low latency" and another team configures Algorithm 128 as "exclude submarine links," you have a conflict that will produce unpredictable forwarding.

Maintain an algorithm registry — a simple document that maps algorithm IDs to definitions. Treat it like you treat your IP address plan. Changes go through change control. No ad-hoc algorithm creation.

### What Happens When Not All Routers Participate

A router that doesn't participate in Algorithm 128 simply doesn't have Algorithm 128 SIDs. Traffic following Algorithm 128 can still transit that router using Algorithm 0 SIDs for that hop, but the path computation for Algorithm 128 will avoid routing through non-participating nodes when possible.

If a critical transit node doesn't participate, the algorithm's SPF tree may not have a path to certain destinations. This creates reachability gaps — traffic steered to Algorithm 128 can reach some destinations but not others. This is the most common deployment mistake: enabling FlexAlgo on edge nodes but forgetting transit nodes.

Rule of thumb: every node in the forwarding path of a FlexAlgo topology must participate in that algorithm. This includes P-nodes that you might think are "just transit." If a P-node doesn't advertise Algorithm 128 SIDs, traffic can't be forwarded through it using Algorithm 128.

### Delay Measurement Accuracy

If you're using min-delay metric, your paths are only as good as your delay measurements. Noisy measurements — caused by queuing delay, scheduling jitter, or probe loss — will produce SPF churn as the delay values fluctuate.

Best practices for delay-based FlexAlgo:
- Use hardware timestamping for delay probes when available
- Configure dampening on delay advertisements — only advertise a change if the delay shifts by more than a threshold (e.g., 1ms)
- Monitor for SPF churn caused by delay oscillation — if you see frequent recomputation for Algorithm 128 but not Algorithm 0, delay instability is the cause
- Don't use delay metric on links with highly variable queuing delay (e.g., congested links) — the measurements will be unreliable

### FlexAlgo and TI-LFA

TI-LFA (Section 6.3) computes backup paths per algorithm. If a link fails, TI-LFA provides a pre-computed backup path that respects the algorithm's constraints. Algorithm 128 traffic doesn't suddenly take a non-low-latency backup path — TI-LFA computes a backup within the Algorithm 128 topology.

This is a significant advantage over RSVP-TE FRR, where the backup path has no awareness of TE constraints. In FlexAlgo + TI-LFA, protection is algorithm-aware by default.

The caveat: if the Algorithm 128 topology is very constrained (few links), TI-LFA may not find a backup path. In that case, traffic drops until the IGP reconverges. Design your algorithms with enough topology redundancy for TI-LFA to compute viable backups.

### Monitoring and Troubleshooting

**Verify algorithm participation.** On IOS-XR: `show isis flex-algo 128`. On Junos: `show isis flex-algorithm 128`. These show which routers are participating, who's advertising the definition, and the current algorithm parameters.

**Verify SID allocation.** Each algorithm should have its own Prefix-SID or SRv6 SID on each participating node. If a node shows Algorithm 0 SIDs but no Algorithm 128 SIDs, it's either not configured to participate or hasn't received the algorithm definition from the IGP.

**Check for topology gaps.** Run `show isis topology flex-algo 128` (IOS-XR) to see the computed SPF tree for that algorithm. Verify all expected destinations are reachable. If a destination is missing, either that node doesn't participate or there's no path to it within the algorithm's constraints.

**Watch for SPF churn.** `show isis statistics` should show SPF computation counts per algorithm. If Algorithm 128 is recomputing far more often than Algorithm 0, you likely have unstable delay measurements or flapping affinity advertisements.

## Configuration Reference

### IOS-XR: FlexAlgo Configuration

**Define the algorithm (on definition-advertising routers):**
```cisco
router isis CORE
 flex-algo 128
  metric-type delay
  advertise-definition
 !
 flex-algo 129
  metric-type te
  affinity exclude-any
   color RED
  !
  advertise-definition
 !
!
```

**Configure affinity colors on interfaces:**
```cisco
router isis CORE
 interface TenGigE0/0/0/0
  affinity flex-algo
   color RED
  !
 !
!
```

**Assign Prefix-SID per algorithm (SR-MPLS):**
```cisco
router isis CORE
 interface Loopback0
  address-family ipv4 unicast
   prefix-sid index 1
   prefix-sid algorithm 128 index 1001
   prefix-sid algorithm 129 index 2001
  !
 !
!
```

**Assign SRv6 locator per algorithm:**
```cisco
segment-routing
 srv6
  locators
   locator MAIN
    prefix fc00:0:1::/48
   !
   locator LATENCY
    prefix fc00:0:1:1::/48
    algorithm 128
   !
  !
 !
!
```

**ODN policy with FlexAlgo color steering:**
```cisco
segment-routing
 traffic-eng
  on-demand color 128
   dynamic
    sid-algorithm 128
   !
  !
 !
!
```

**Performance measurement for delay (required for delay-based FlexAlgo):**
```cisco
performance-measurement
 interface TenGigE0/0/0/0
  delay-measurement
   advertise-delay 1000
  !
 !
!
```

### Junos: FlexAlgo Configuration

**Define the algorithm:**
```junos
set protocols isis flexible-algorithm 128 definition metric-type delay
set protocols isis flexible-algorithm 128 definition priority 200
set protocols isis flexible-algorithm 128 advertise
```

**Configure affinity on interfaces:**
```junos
set protocols isis interface ge-0/0/0 level 2 flex-algo-affinity-name RED
```

**Assign Prefix-SID per algorithm (SR-MPLS):**
```junos
set protocols isis source-packet-routing flex-algorithm 128 prefix-sid 1001
```

**Delay measurement:**
```junos
set protocols isis interface ge-0/0/0 level 2 delay-measurement
```

## Review Questions

1. Your backbone has 6 P-nodes. You enable FlexAlgo 128 (delay metric) on 4 of them but forget the other 2. What happens to traffic steered to Algorithm 128 that needs to transit the non-participating nodes?

2. Two teams independently configure FlexAlgo 128 with different definitions on different routers. What happens when both definitions flood through IS-IS? How does the IGP resolve the conflict?

3. You're running delay-based FlexAlgo 128. During a traffic spike, queuing delay on several links increases by 5ms. The SPF tree for Algorithm 128 recomputes and shifts traffic to alternate paths, which then also become congested. Describe this feedback loop and how you'd prevent it.

5. Design a FlexAlgo scheme for a backbone with 3 service tiers: latency-sensitive (financial), standard (enterprise), and bulk (backup/replication). Define the algorithms, metric types, constraints, and VRF steering method.

## Summary

FlexAlgo is the missing piece that makes Segment Routing a complete replacement for RSVP-TE. Where RSVP-TE required per-tunnel state, per-tunnel bandwidth reservation, and per-tunnel FRR computation, FlexAlgo delivers topology slicing at the IGP level — no signaling protocol, no per-flow state, no tunnel management.

The power is in the simplicity. Define an algorithm, assign SIDs, steer traffic. The IGP handles the path computation. TI-LFA handles the protection. The physical network stays the same. The logical networks multiply.

The operational cost is algorithm management — keeping IDs consistent, ensuring all transit nodes participate, monitoring delay stability for latency-based algorithms, and maintaining the affinity color assignments. None of this is hard, but all of it requires discipline. FlexAlgo is easy to deploy and easy to break if you're not systematic about it.

For any SP evaluating Cisco's SRv6 + FlexAlgo pitch: this is their strongest differentiator. Juniper and Nokia support FlexAlgo, but Cisco's implementation is the most mature, the most feature-complete, and the most deeply integrated with their ODN and Crosswork tooling. If FlexAlgo is a core requirement, Cisco has the edge. If it's nice-to-have, all three vendors can deliver.
