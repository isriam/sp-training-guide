# 6.4 SRv6 Fundamentals

## Overview

SRv6 — Segment Routing over IPv6 — is the next evolution of Segment Routing. Same source routing principles as SR-MPLS, but instead of MPLS labels, it uses IPv6 extension headers. The promise is seductive: all the benefits of SR — stateless core, source-routed traffic engineering, TI-LFA protection — without needing an MPLS data plane at all. Just IPv6, end to end.

For greenfield IPv6 networks or SPs looking to simplify their protocol stack down to one forwarding plane, SRv6 is a compelling story. For existing MPLS deployments that already run SR-MPLS, the migration math is less obvious. You're replacing a working MPLS data plane with an IPv6-native one that has higher per-packet overhead and less mature hardware support. Whether that tradeoff makes sense depends on your 5-year architecture vision.

This section covers SRv6 architecture from the ground up. We'll start with why SRv6 exists, then walk through the SID structure, the Segment Routing Header, and — critically — the endpoint behaviors that make SRv6 more than just "MPLS with bigger labels." The endpoint behaviors are where the real power is, and they're where most people's understanding gets fuzzy. We're going to fix that.

## Why SRv6 Exists

SR-MPLS works. It works really well. So why build something new?

SR-MPLS has three architectural constraints that SRv6 removes. First, every device in the forwarding path must support MPLS. If you have a node that only does IPv6 forwarding — a cloud edge router, a containerized function, a disaggregated whitebox — it can't participate in your SR-MPLS domain. Second, MPLS labels are 20 bits. That gives you about a million possible labels, which sounds like a lot until you're managing a global backbone with thousands of VRFs, hundreds of nodes, and multiple label spaces that need to be coordinated. Third, the label stack has depth limits on many hardware platforms. A 3-label stack is fine; a 12-label stack for a complex traffic engineering policy might exceed what your ASIC can handle.

SRv6 addresses all three. It uses pure IPv6 forwarding, so any device that can route IPv6 can participate — even if it doesn't know what SRv6 is, it just forwards the packet based on the destination address. SRv6 SIDs are 128 bits, giving you a functionally infinite address space for segments. And the segment list is carried in an IPv6 extension header with no hardware-imposed depth limit — though practical limits still exist because of MTU.

The tradeoff is overhead. An MPLS label is 4 bytes. An SRv6 SID is 16 bytes, plus you pay for a 40-byte IPv6 outer header and an 8-byte SRH base header that don't exist in SR-MPLS. For a 3-segment path, SR-MPLS costs you 12 bytes of overhead. SRv6 costs you 96. That's not a rounding error — it's a design constraint that affects your MTU planning, your link utilization, and your hardware forwarding performance.

There's one more critical requirement that trips people up: SRv6 requires a native IPv6 underlay. You cannot run SRv6 over an IPv4-only core. Unlike SR-MPLS, which works on any IP transport, SRv6 needs IPv6 end-to-end. If your backbone is IPv4-only today, migrating to SRv6 means migrating to dual-stack or IPv6-only first. That's not a configuration change — that's an infrastructure program.

## The SRv6 SID: Location Plus Behavior in One Address

This is the conceptual leap that makes SRv6 different from everything that came before it. An SRv6 SID is not just an address. It's an address that also encodes an instruction.

Every SRv6 SID is a 128-bit IPv6 address divided into three logical parts: a locator, a function, and an optional argument.

The **locator** is the network-routable prefix. It tells the network which node owns this SID. A typical locator is a /48 prefix — something like `fc00:0:1::/48` meaning "this SID belongs to Router 1." The locator is advertised through IS-IS or OSPFv3 just like any other IPv6 prefix, so the network knows how to reach it. This is important: the locator makes SRv6 SIDs routable by default. Any IPv6 router in the path can forward toward an SRv6 SID even if it has no idea what SRv6 is. It just sees an IPv6 destination address and routes it.

The **function** is what makes SRv6 special. It's a value — typically 16 bits — appended after the locator that tells the owning node what to do with the packet when it arrives. Different function values mean different behaviors. `e000` might mean "basic transit forwarding." `d400` might mean "decapsulate this packet and look up the inner destination in VRF customer-a." The function is what turns a passive address into an active instruction.

The **argument** is optional additional data that some functions need. For example, an argument might specify which interface to forward on, or which sub-table to look up. Most basic SRv6 deployments don't use arguments extensively.

Here's a concrete example. The SID `fc00:0:1:e000::` breaks down as: locator `fc00:0:1::/48` identifies Router 1, function `e000` means "END" which is basic transit forwarding, and there's no argument. When a packet arrives at Router 1 with this SID as the destination, Router 1 looks it up in its local SID table, sees "END behavior," and executes that behavior — which means processing the Segment Routing Header and forwarding to the next segment.

Compare this to SR-MPLS where a Prefix-SID is just a label — say, label 16001. The label tells you "forward toward this prefix" but it doesn't encode any behavior beyond that. VPN decapsulation, cross-connects, binding SIDs — those are all separate signaling constructs. In SRv6, the behavior is baked into the SID itself. One address, one instruction.

The comparison is worth making explicit. An SR-MPLS label is 20 bits and tells you where to forward. An SRv6 SID is 128 bits and tells you where to forward AND what to do when you get there. SR-MPLS allocates labels from a globally-coordinated SRGB range — typically 16000 through 23999 — and you need to make sure no two nodes claim the same index. SRv6 allocates SIDs from IPv6 prefix blocks, and as long as your locators don't overlap, you're fine. The risk of accidental collision is near zero with proper IPAM, but if two nodes accidentally get the same /48 locator, traffic will blackhole. Use your IP address management system. Don't hand-assign locators.

## The Segment Routing Header

The SRH is how SRv6 carries the segment list — the ordered list of SIDs the packet should traverse. It's an IPv6 extension header with Routing Type 4, defined in RFC 8754.

Here's how it works in plain English. When a headend node wants to send a packet along a specific SRv6 path, it encapsulates the original packet inside a new IPv6 header. The outer IPv6 destination address is set to the first SID in the path. The SRH is attached as an extension header containing the complete segment list — every SID the packet needs to visit, in order.

The segment list is stored in reverse order. This confuses everyone the first time they see it, but there's a logical reason. If you want traffic to go R1, then R3, then R5, then R7, the segment list is stored as [R7, R5, R3]. The list is indexed from zero, and a counter called Segments Left starts at the highest index. As each SRv6 node processes the packet, it decrements Segments Left and copies the next SID from the list into the IPv6 destination address. By storing the list in reverse, the indexing math works out cleanly — Segments Left counts down from the top of the list toward zero, and each decrement moves you one hop further along the intended path.

Let's walk through a concrete example. Traffic needs to go from R1 through R3, R5, and finally R7. The headend at R1 builds the SRH with Segments Left set to 2 and the segment list [fc00:0:7::, fc00:0:5::, fc00:0:3::]. The IPv6 destination address is set to the active segment — Segment List index 2, which is fc00:0:3::. The packet enters the network heading toward R3.

R3 receives the packet, matches the destination against its local SID table, and finds an END behavior. It decrements Segments Left from 2 to 1, copies Segment List index 1 — fc00:0:5:: — into the IPv6 destination address, and forwards the packet toward R5.

R5 does the same thing. Decrements Segments Left from 1 to 0, copies Segment List index 0 — fc00:0:7:: — into the destination, and forwards toward R7.

R7 receives the packet with Segments Left at 0. This means R7 is the final segment. It executes whatever endpoint behavior its SID specifies. If R7's SID has an END.DT4 behavior, it decapsulates the packet and routes the inner IPv4 payload through the specified VRF. If it's a simple END, it just forwards the decapsulated packet.

The SRH itself has a few important fields beyond the segment list. The Segments Left counter we just described. A Last Entry field that records the index of the final segment in the list — this tells the node how deep the list goes. Flags and a Tag field for optional metadata. And then the array of 128-bit SIDs.

One detail that matters operationally: at the final destination, what happens to the SRH depends on the endpoint behavior. For basic END behavior, the SRH stays in the packet. For decapsulation behaviors like END.DT4 or END.DT6, the entire outer IPv6 header and SRH are stripped off, and the inner packet is extracted and forwarded as if the SRv6 encapsulation never existed. This is important for services — the customer's packet arrives at the egress PE clean, with no SRv6 artifacts.

## Endpoint Behaviors: The Heart of SRv6

This is where SRv6 stops being "MPLS with bigger labels" and starts being something genuinely new. Each SRv6 SID encodes a specific behavior that the owning node executes when a packet arrives. These behaviors are defined in RFC 8986 and they're the building blocks of everything SRv6 can do.

### END — The Basic Transit Function

END is the simplest behavior and the one you'll see most often. It's the SRv6 equivalent of a Prefix-SID in SR-MPLS. When a packet arrives at a node and matches an END SID, the node decrements Segments Left, updates the IPv6 destination address to the next segment in the list, and forwards the packet. That's it. No decapsulation, no VRF lookup, no special handling.

Every SRv6-capable node has at least one END SID. It's what makes the node reachable as a transit point in a segment list. If you're building an SRv6 path through R1, R3, R5, R7 — the END SIDs on R3 and R5 are what allow packets to traverse those nodes. Without END SIDs, there's nothing to match against and no behavior to execute.

Think of END as the glue that holds segment lists together. It's not glamorous, but without it, nothing works.

### END.X — Strict Next-Hop Forwarding

END.X is the SRv6 equivalent of an Adjacency-SID in SR-MPLS. Where END says "forward based on the routing table," END.X says "forward out this specific interface to this specific next-hop." It bypasses the routing table entirely for that hop.

Why would you want this? Traffic engineering. If you need traffic to take a specific physical path — maybe you're avoiding a congested link, or you need path diversity for compliance reasons — END.X lets you pin traffic to an exact next-hop. The routing table might say "the best path to R5 is through R4," but an END.X SID on R3 can say "no, send this traffic directly to R5 over the R3-R5 link."

END.X is critical for strict explicit paths in SR-TE policies. If your policy says "go through these exact links in this exact order," every hop needs an END.X SID for the specific outgoing interface.

### END.DT4 — IPv4 VRF Decapsulation

This is the big one for L3VPN services, and it's where most people's understanding breaks down. Let's be really precise about what END.DT4 does and why it exists.

In a traditional MPLS L3VPN, when a packet arrives at the egress PE, it carries two labels. The outer transport label gets the packet to the right PE — that's the Prefix-SID or LDP label. The inner VPN label identifies which VRF to look up the destination in. The PE pops both labels, looks at the inner VPN label to identify the VRF, then does an IPv4 route lookup in that VRF to find the next-hop toward the CE.

END.DT4 does exactly the same job, but in one SID instead of two labels. When a packet arrives at an SRv6 node and matches an END.DT4 SID, the node does three things in order. First, it strips the entire outer IPv6 header and SRH — complete decapsulation. Second, it identifies the VRF associated with this specific END.DT4 SID. Third, it performs an IPv4 route lookup in that VRF and forwards the inner IPv4 packet toward the customer.

The "DT" stands for "Decapsulation and Table lookup." The "4" means IPv4. So END.DT4 literally means: decapsulate the SRv6 encapsulation, then do an IPv4 lookup in the specified routing table.

Here's why this is powerful. In SR-MPLS, you need two labels working together — the transport label and the VPN label — plus MP-BGP to signal VPN label bindings between PEs. In SRv6, the END.DT4 SID encodes both the destination PE and the VRF in a single 128-bit address. MP-BGP still distributes the SID between PEs, but the forwarding is cleaner: one SID in the segment list, the packet arrives, decap, VRF lookup, done.

Each VRF on a PE gets its own unique END.DT4 SID. If Router 7 has three VRFs — customer-a, customer-b, and customer-c — it will have three END.DT4 SIDs, one per VRF. Something like `fc00:0:7:d401::` for customer-a, `fc00:0:7:d402::` for customer-b, `fc00:0:7:d403::` for customer-c. The locator `fc00:0:7::/48` gets the packet to Router 7. The function `d401`, `d402`, or `d403` tells Router 7 which VRF to use.

On IOS-XR, you configure this under the BGP VRF address family. You tell BGP "for this VRF, use this SRv6 locator, and allocate SIDs per-VRF." BGP then automatically allocates an END.DT4 SID for each VRF and advertises it to remote PEs via MP-BGP VPNv4 updates. The remote PE learns "to reach VRF customer-a on Router 7, use SID fc00:0:7:d401::."

The allocation can be per-VRF — one SID shared by all prefixes in the VRF — or per-CE, where each CE-facing interface gets its own SID. Per-VRF is simpler and uses fewer SIDs. Per-CE gives you more granular traffic steering at the cost of more SIDs to manage.

### END.DT6 — IPv6 VRF Decapsulation

END.DT6 is the IPv6 counterpart to END.DT4. Exact same concept: decapsulate the SRv6 encapsulation, identify the VRF, perform an IPv6 route lookup in that VRF. The "6" means IPv6.

You might wonder: if SRv6 is already IPv6, why do you need a separate decap behavior for IPv6 VRF traffic? Because the outer IPv6 header is the SRv6 transport header — it carries the SRH and SID list. The inner payload might be an IPv6 customer packet that has nothing to do with SRv6. END.DT6 strips the outer SRv6 encapsulation and routes the inner IPv6 packet in the customer's VRF. The outer and inner IPv6 headers are completely independent.

In practice, most L3VPN deployments need both END.DT4 and END.DT6 — one for customer IPv4 traffic, one for customer IPv6 traffic. Some implementations offer END.DT46, a dual-stack variant that handles both IPv4 and IPv6 inner packets with a single SID. This reduces the SID count per VRF from two to one, which helps at scale.

### END.DX4 and END.DX6 — Cross-Connect Decapsulation

END.DX4 and END.DX6 are like END.DT4 and END.DT6 but with a twist: instead of doing a table lookup in the VRF, they forward to a specific pre-configured next-hop. The "X" stands for cross-connect.

Where END.DT4 says "decapsulate and look up the destination in the VRF routing table," END.DX4 says "decapsulate and send the packet directly to this specific next-hop on this specific interface." No routing table lookup. The next-hop is hardcoded in the SID configuration.

When would you use this? Point-to-point VPN services where there's only one CE behind the PE, or where you want deterministic forwarding without relying on the VRF routing table. It's the SRv6 equivalent of a VPWS (Virtual Private Wire Service) pseudowire hand-off.

END.DX6 is the same thing for IPv6 inner packets. Decapsulate, cross-connect to a specific IPv6 next-hop. No table lookup.

### END.B6.ENCAPS — The Binding SID

END.B6.ENCAPS is SRv6's equivalent of the Binding SID in SR-MPLS. It's a SID that represents an entire SR Policy. When a packet arrives and matches a Binding SID, the node encapsulates it in a new SRv6 header with a new segment list — the policy's segment list.

This is how you chain policies together or create hierarchical routing. Node A has a policy "take path X to reach Node D." Instead of putting all of path X's segments into the original packet's SRH, Node A just puts the Binding SID of that policy. When the packet hits the policy headend, the Binding SID triggers encapsulation with the full segment list.

Binding SIDs are critical for scalability. They compress complex multi-hop paths into a single SID, keeping the SRH shorter and the overhead lower.

## SRv6 vs SR-MPLS: The Honest Comparison

Let's cut through the marketing and compare these side by side on the dimensions that actually matter for deployment decisions.

**Overhead.** SR-MPLS labels are 4 bytes each. SRv6 SIDs are 16 bytes each, plus you pay 40 bytes for the outer IPv6 header and 8 bytes for the SRH base. For a 3-segment path, SR-MPLS adds 12 bytes. SRv6 adds 96 bytes. That's 8x the overhead. This directly impacts your effective MTU, your link utilization calculations, and your hardware forwarding performance. The gap narrows with uSID compression — covered in Section 6.5 — but it doesn't disappear.

**Hardware support.** SR-MPLS is supported on every major SP routing platform shipped in the last decade. ASR 9000, MX series, NCS, PTX, Nokia 7750 — all of them. SRv6 requires specific hardware. On Cisco, you need NCS 5500, ASR 9000 with X64 line cards, or the 8000 series. On Juniper, PTX10003 and PTX10008 support it; MX support is limited and version-dependent. Not all platforms that advertise SRv6 support actually support all SRv6 functions — some only do END and END.DT4, missing END.X or more complex behaviors. Always verify the platform feature matrix for your specific hardware and software version.

**SID space.** SR-MPLS has a 20-bit label space — about a million labels, shared across the SRGB. SRv6 has 128-bit SIDs. You will never run out. This matters at scale when you're managing thousands of VRFs, each needing multiple SIDs per PE.

**Operational maturity.** SR-MPLS has years of production deployment across hundreds of service providers. The troubleshooting tools are mature, the failure modes are well-documented, and there's a deep bench of engineers who understand it. SRv6 is still early. The troubleshooting tools are less polished, packet captures with SRH headers are harder to read, and fewer engineers have hands-on experience. Traceroute behavior is different in SRv6 — it doesn't naturally show the SRH path like MPLS traceroute shows the label stack.

**Future-proofing.** The industry trend is toward SRv6. Cisco is pushing it hard with uSID, FlexAlgo over SRv6, and NCS 8000 as the flagship platform. The long-term vision is an IPv6-only network with SRv6 as the unified forwarding plane — no MPLS, no LDP, no RSVP-TE. Whether that vision materializes in 3 years or 10 years is the gamble you're making.

**Bottom line for 2026.** If you're building a new backbone today, SR-MPLS is the safe choice with a clear migration path to SRv6 later. If you're Cisco-committed, have modern hardware, and want to be on the leading edge, SRv6 with uSID is viable but you're an early adopter. If you're running Juniper MX, SRv6 support is limited enough that SR-MPLS is the obvious choice for now.

## Configuration

The following configurations are vendor-specific. IOS-XR config does not work on Junos, and vice versa. These examples show the minimum viable SRv6 deployment — locator, IGP advertisement, and VPN service binding.

### IOS-XR: SRv6 Configuration

**Enable SRv6 and define the locator.** The locator is the /48 block assigned to this node. The source address is used as the outer IPv6 source for SRv6-encapsulated packets.

```cisco
segment-routing
 srv6
  encapsulation
   source-address fc00:0:1::1
  !
  locators
   locator MAIN
    micro-segment behavior unode psp-usd
    prefix fc00:0:1::/48
   !
  !
 !
!
```

The `micro-segment behavior unode psp-usd` line enables uSID mode — compressed micro-segment IDs that pack multiple logical SIDs into a single 128-bit address. PSP means Penultimate Segment Pop, where the second-to-last node pops the SRH before the final destination. USD means Ultimate Segment Decapsulation, where the final node decapsulates and processes the payload. Both are standard SRv6 flavors covered in depth in Section 6.5.

**Advertise the locator via IS-IS.** This is how the rest of the network learns that this node owns the `fc00:0:1::/48` SID block.

```cisco
router isis CORE
 net 49.0001.0000.0000.0001.00
 address-family ipv6 unicast
  segment-routing srv6
   locator MAIN
  !
 !
!
```

Unlike SR-MPLS, there's no `prefix-sid index` to configure. The locator itself is the advertisement — IS-IS distributes it using SRv6 Locator TLVs and the node's SIDs are derived from it.

**Bind a VRF to SRv6 for L3VPN service.** This tells BGP to allocate END.DT4 SIDs for this VRF and advertise them to remote PEs.

```cisco
router bgp 65000
 vrf cust-a
  address-family ipv4 unicast
   segment-routing srv6
    locator MAIN
    alloc mode per-vrf
   !
  !
 !
!
```

The `alloc mode per-vrf` means one END.DT4 SID is allocated for all prefixes in this VRF. The alternative, `per-ce`, allocates a separate SID for each CE — more granular but more SIDs to manage.

### Junos: SRv6 Configuration

**Define the SRv6 block and locator.**

```junos
set routing-options source-packet-routing srv6 block MAIN-BLOCK address fc00:0:1::/48
set routing-options source-packet-routing srv6 locator MAIN
```

**Advertise via IS-IS.**

```junos
set protocols isis source-packet-routing srv6 locator MAIN
set protocols isis interface lo0.0 ipv6-adjacency-segment advertise-sids
```

**VRF service binding** is configured in the routing-instance hierarchy. The specific syntax depends on Junos version and whether you're using END.DT4, END.DT46, or per-CE allocation. Consult Juniper's SRv6 feature guide for your platform and release.

## Verification

### Checking Your SRv6 Deployment

**Step 1 — Verify SRv6 is enabled and SIDs are allocated.**

On IOS-XR, `show segment-routing srv6 sid` displays the local SID table. You should see at least one END SID from IS-IS and one END.DT4 SID per VRF from BGP.

```
RP/0/RP0/CPU0:R1# show segment-routing srv6 sid
SID                      Behavior          Context                 Owner
fc00:0:1:e000::          END               'default':4369          IS-IS
fc00:0:1:d400::          END.DT4           'cust-a'                BGP-VPNv4
fc00:0:1:d600::          END.DT6           'cust-a'                BGP-VPNv6
```

On Junos, `show spring srv6 locator` gives you the equivalent view.

```
root@R1> show spring srv6 locator
Locator: MAIN
  Prefix: fc00:0:1::/48, Algorithm: 0
  SID                      Behavior     Owner
  fc00:0:1:e000::          End          IS-IS
  fc00:0:1:d400::          End.DT4      BGP-VPN
```

If your SID table is empty, the locator isn't configured or the IGP isn't advertising it. Go back to step one.

**Step 2 — Verify the locator is in the IGP.**

On IOS-XR: `show isis database verbose | include SRv6`. Look for the SRv6 Locator TLV in the LSP. On Junos: `show isis database extensive | match SRv6`. Same thing.

**Step 3 — Verify remote locators are reachable.**

Check that you have an IPv6 route to each remote node's locator prefix. On both platforms: `show route fc00:0:<remote-node>::/48`. If the route is missing, the remote node isn't advertising its locator or IS-IS adjacency is down.

**Step 4 — Check the forwarding table for a specific SID.**

On IOS-XR: `show cef fc00:0:3:e000:: detail`. On Junos: `show route forwarding-table destination fc00:0:3:e000:: extensive`. This confirms the FIB has a valid entry and shows the next-hop. If the next-hop is wrong, you have a routing issue, not an SRv6 issue.

**Step 5 — For VPN services, verify VRF bindings.**

On IOS-XR: `show bgp vpnv4 unicast vrf cust-a`. On Junos: `show route table cust-a.inet.0 protocol bgp`. Look for the remote PE's END.DT4 SID in the route attributes. If it's missing, the remote PE isn't advertising its VRF SID via MP-BGP, or the BGP session is down.

**Step 6 — Check MTU.**

This is the one people forget. On both platforms: check interface MTU. IPv6 base header is 40 bytes. SRH is 8 bytes plus 16 bytes per SID. A 3-SID path costs 96 bytes of overhead. If your interface MTU is 1500 and your customer sends a 1500-byte packet, the encapsulated packet is 1596 bytes and it will be dropped — IPv6 routers do not fragment transit packets. The source gets an ICMPv6 Packet Too Big message, but only if your infrastructure passes ICMP. Many firewall policies block it, creating a silent black hole.

Best practice: set core interface MTU to at least 9000 bytes. Problem goes away.

## Troubleshooting

### Common Failures and How to Fix Them

**SRv6 SID not appearing in the SID table.** The locator isn't configured, or IS-IS isn't advertising it. Check the locator configuration first — is the prefix correct? Is the locator referenced in the IGP config? On IOS-XR, verify under `router isis <tag> / address-family ipv6 unicast / segment-routing srv6 / locator <name>`.

**Packets dropped at an SRv6 transit node.** The SID doesn't exist in the local SID table. Either the SID was misconfigured in the segment list — a typo in a 128-bit address is easy to make and hard to spot — or the remote node hasn't allocated that SID. Check `show srv6 sid` on the node that's dropping.

**END.DT4 decapsulation fails silently.** The packet arrives at the egress PE, matches the END.DT4 SID, but has no valid inner IPv4 payload. The PE drops the packet and may log an error. Check packet captures to verify the inner payload is actually an IPv4 packet and not malformed. Also verify the VRF exists and has routes.

**MTU black hole — large packets disappear.** Classic SRv6 issue. The SRH overhead pushes the packet over the interface MTU. IPv6 routers don't fragment — they send ICMPv6 Packet Too Big. But if a firewall in the path blocks ICMP, the source never learns the path MTU and keeps sending oversized packets. TCP eventually works because MSS negotiation limits segment size, but UDP traffic dies silently. Fix: increase core MTU to 9000+. Also: make sure your firewalls pass ICMPv6 Packet Too Big messages — blocking these is a cardinal sin in IPv6.

**Traffic not following the SR Policy.** The policy's color doesn't match the BGP community on the route. SR-TE automated steering matches traffic to policies based on color extended communities. If the color in the policy definition doesn't match what BGP is tagging on the route, the traffic goes through the IGP best path instead of the policy. Check `extcommunity-set color` and the route-policy that applies it.

**Locator not advertised in IS-IS.** The IGP configuration is missing the `segment-routing srv6 locator` statement under the IPv6 address family. This is the most common configuration miss — you can define the locator globally but forget to activate it under IS-IS.

## Design Considerations

### Locator Planning

Locator design is the SRv6 equivalent of IP address planning — get it wrong early and you'll be living with the pain for years.

A /48 per node is the standard recommendation. This gives you 65,536 possible /64 function blocks, which is vastly more than any single node needs. Within the /48, organize by function type. Allocate a /64 block for END functions, another for END.DT4, another for END.DT6, another for Binding SIDs. This makes ACL filtering and operational identification easier — you can look at a SID and immediately know its function type from the locator structure.

For address space, use ULA (Unique Local Address) space — specifically fd00::/8 — for private networks. Don't use fc00::/8 even though you'll see it in vendor examples. RFC 4193 technically requires central registry assignment for fc00::/8, and no such registry exists. fd00::/8 is the self-assigned variant and is the correct choice for private use.

For inter-domain or public-facing SRv6, use globally unique address space from your RIR allocation.

### When to Deploy SRv6

**Good fit:** Greenfield IPv6 network with no MPLS legacy. Organization committed to IPv6-only long-term. Hardware is SRv6-capable. Engineering team is comfortable with IPv6 operations. Willing to accept early-adopter risk in exchange for future-proofing.

**Bad fit:** Existing MPLS network with SR-MPLS already working. Hardware doesn't support SRv6. Engineering team's IPv6 troubleshooting skills are immature. Organization needs battle-tested, well-documented technology. Cost of MTU redesign and IPv6 migration outweighs SRv6 benefits.

**The pragmatic path:** Deploy SR-MPLS today. Design for SRv6 migration later. Use IS-IS, keep IPv6 enabled everywhere, avoid MPLS-specific assumptions in your automation. When SRv6 hardware matures and your team is ready, the migration is incremental — you don't have to rip and replace.

## War Story: The SRv6 MTU Black Hole

This is a composite scenario based on real deployment issues that multiple SPs have encountered.

A large European ISP deployed a greenfield SRv6 core. All interfaces were left at default 1500-byte MTU. SR Policies were built with 6-SID segment lists for traffic engineering — 144 bytes of SRv6 overhead.

Customer traffic started flowing. TCP worked fine because MSS negotiation limited segment sizes. But approximately 15% of UDP traffic was silently dropping. The packets that were 1500 bytes — the maximum — became 1644 bytes after SRv6 encapsulation. Transit routers generated ICMPv6 Packet Too Big messages, but a mid-path firewall had a security policy blocking IPv6 fragments and ICMPv6 from the core. The Packet Too Big messages never reached the source. UDP applications had no way to discover the path MTU, so they kept sending 1500-byte packets that kept getting dropped.

The outage lasted 8 hours. The root cause was three failures stacked together: MTU wasn't planned for SRv6 overhead, the firewall team wasn't told SRv6 was going live, and nobody tested with realistic packet sizes before production.

The fix was layered. Immediately, they reduced segment lists to 3 SIDs, cutting overhead to 96 bytes. Short-term, they increased core MTU to 9000 across all interfaces. Long-term, they deployed uSID compression to reduce SID counts further. And they added SRv6 MTU validation to their pre-deployment checklist.

The lesson: SRv6 overhead is not like MPLS overhead. A 4-label SR-MPLS stack costs 16 bytes. A 4-SID SRv6 path costs 104 bytes. If you don't plan your MTU around SRv6, you will have a bad day.

---

## Summary

SRv6 is an elegant architecture. Pure IPv6, no MPLS, behaviors encoded in addresses, a programmable forwarding plane. The theory is sound and the long-term direction is clear — this is where the industry is heading.

The 2026 reality is messier. Hardware support is limited. Overhead is significant. Operational tooling is less mature than SR-MPLS. The security model — which we'll cover in depth in Module 13 — has real gaps around SRH filtering and firewall awareness.

For most SPs today, SR-MPLS remains the right choice. SRv6 is the right investment for the future, and the migration path from SR-MPLS to SRv6 is well-defined. Deploy SR-MPLS now, design for SRv6 later. You won't regret being on SR-MPLS in 2026. You might regret being on SRv6 if your hardware, team, and MTU design aren't ready for it.

The endpoint behaviors — END, END.X, END.DT4, END.DT6, END.DX4, END.DX6, END.B6.ENCAPS — are the conceptual core of SRv6. If you understand what each one does and why it exists, you understand SRv6. Everything else is implementation detail.

```cisco
segment-routing
 traffic-eng
  srv6
   source-address fc00:0:1::1

   srv6
    locator MAIN binding-sid dynamic behavior ub6-insert-reduced
   !

    preference 10
     explicit segment-list R1-R3-R5-R7
      index 10 sid fc00:0:3:e000::
      index 20 sid fc00:0:5:e000::
      index 30 sid fc00:0:7:e000::
     !
    !
   !
  !
 !
!
```

**Key IOS-XR configuration knobs:**

- `micro-segment behavior unode psp-usd` — enables uSID mode with Penultimate Segment Pop and Ultimate Segment Decapsulation. Without this, SIDs are full 128-bit.
- `alloc mode per-vrf` — one END.DT4/DT6 SID per VRF. Alternative: `per-ce` allocates one SID per CE interface — more granular, more SIDs to manage.
- `binding-sid dynamic behavior ub6-insert-reduced` — the Binding SID uses reduced SRH, omitting the first segment from the SRH if it matches the IPv6 DA. Saves 16 bytes per policy. Alternative: `ub6-insert-encaps` for full encapsulation with all segments in SRH.
- `source-address` — outer IPv6 source for SRv6-encapsulated packets. Must be routable and unique per node.

```cisco
!
```

### SRv6 Interworking with SR-MPLS

If you have an existing SR-MPLS core and want to add SRv6 edges — or vice versa — you need a gateway node that translates between the two data planes.

**Gateway functions:**
- **SR-MPLS → SRv6**: Gateway encapsulates the MPLS packet in IPv6 + SRH
- **SRv6 → SR-MPLS**: Gateway decapsulates IPv6, exposes the MPLS label stack

RFC 9252 covers BGP overlay service signaling for SRv6. The data-plane interworking — actually translating between MPLS labels and SRv6 SIDs — is implementation-specific. Cisco uses END.BM (per RFC 8986) for MPLS-to-SRv6 translation.

**IOS-XR gateway example — hybrid SR-MPLS + SRv6 policy:**
```cisco
segment-routing
 traffic-eng
  policy MPLS-to-SRv6
   color 200 end-point ipv6 fc00:0:7::1
   candidate-paths
    preference 10
     explicit segment-list hybrid
      index 10 mpls label 16003
      index 20 srv6 sid fc00:0:7:e000::
     !
    !
   !
  !
 !
!
```

This is advanced — most SPs avoid interworking by picking one data plane and sticking with it. But during migration, you'll likely run both for a period, and the gateway is how you bridge them.

### Hardware Support Matrix (2026)

**Supported:**
- Cisco NCS 5500/5700, ASR 9000 (X64 line cards only), 8000 series
- Juniper PTX10003/PTX10008
- Juniper MX (limited — check release notes for your specific platform/version)
- Nokia 7750 SR-1 (FP4/FP5 line cards only)
- Huawei NE9000, NetEngine 8000

**NOT supported:**
- Cisco ASR 9000 legacy line cards (non-X64)
- Cisco NCS 540 (XR6 platform)
- Juniper MX960/480/240 with Trio chipset
- Most campus/enterprise routers

Before deploying SRv6, always verify your specific platform, line card, and software version against the vendor feature navigator. Some platforms advertise SRv6 support but only implement a subset of endpoint behaviors — for example, END and END.DT4 but not END.X or END.B6.ENCAPS. Partial support can be worse than no support because it fails silently when you try to use an unsupported behavior.
