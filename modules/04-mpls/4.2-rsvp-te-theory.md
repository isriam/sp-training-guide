# 4.2 — RSVP-TE — Protocol Theory

## Protocol Overview

RSVP-TE (Resource Reservation Protocol — Traffic Engineering) is the signaling protocol for establishing MPLS traffic-engineered LSPs. It extends the original RSVP (RFC 2205, designed for IntServ QoS reservation) with TE objects (RFC 3209) to set up explicit label-switched paths with bandwidth guarantees, explicit routing, and fast reroute.

Unlike LDP, which creates LSPs that follow IGP shortest paths with no constraints, RSVP-TE creates LSPs along explicitly computed paths that may deviate from IGP shortest paths. The head-end router runs CSPF (Constrained Shortest Path First) to compute a path that satisfies bandwidth, metric, affinity, and other constraints, then signals this path hop-by-hop using PATH and RESV messages.

RSVP-TE is a soft-state protocol. Every LSP must be periodically refreshed; if refreshes stop, the reservation is torn down. This is fundamentally different from LDP's hard-state TCP sessions and BGP's TCP-based persistence. Soft state provides self-healing (stale reservations automatically expire) at the cost of refresh overhead.

RSVP-TE runs directly over IP (protocol number 46). It does not use TCP or UDP. Each RSVP message is an IP packet with protocol 46 in the IP header. Because it doesn't use TCP, RSVP-TE must handle its own reliability — primarily through the refresh mechanism and summary refresh extensions (RFC 2961).

## Core Mechanisms

### 1. RSVP Message Format

**Common RSVP Header (8 bytes):**
```
├─ Version (4 bits): always 1
├─ Flags (4 bits): includes Refresh Reduction capable flag
├─ Message Type (1 byte)
├─ RSVP Checksum (2 bytes)
├─ Send_TTL (1 byte): TTL when sent (for hop detection)
├─ Reserved (1 byte)
├─ RSVP Length (2 bytes): total message length
```

**Core Message Types:**

| Type | Name | Direction | Function |
|------|------|-----------|----------|
| 1 | Path | Head-end → Tail-end | Establish path, carry TE parameters |
| 2 | Resv | Tail-end → Head-end | Confirm reservation, distribute labels |
| 3 | PathErr | Toward Head-end | Signal error on path setup or maintenance |
| 4 | ResvErr | Toward Tail-end | Signal error on reservation |
| 5 | PathTear | Head-end → Tail-end | Tear down path state |
| 6 | ResvTear | Tail-end → Head-end | Tear down reservation state |
| 12 | Bundle | Any | Bundled refresh messages (RFC 2961) |
| 15 | Srefresh | Any | Summary refresh (RFC 2961) |

### 2. PATH Message and Objects

The PATH message travels from head-end to tail-end, hop by hop, establishing path state at each transit router.

**Key objects carried in PATH:**

**SESSION Object**: Identifies the LSP tunnel.
- Tunnel endpoint address (tail-end IP)
- Tunnel ID (16-bit, locally assigned by head-end)
- Extended Tunnel ID (typically head-end's IP address)
- Together these form a globally unique LSP identifier.

**SENDER_TEMPLATE**: Identifies the sender within the session.
- Sender address (head-end IP)
- LSP ID (16-bit, incremented for make-before-break)

**LABEL_REQUEST**: Requests that each downstream hop assign a label.
- L3PID (protocol being labeled — typically 0x0800 for IPv4)

**ERO (Explicit Route Object)**: The explicit path the LSP must follow.
- Ordered list of subobjects, each identifying a hop:
  - **Strict**: Must be the next hop (directly connected).
  - **Loose**: Must be reachable, but intermediate hops are allowed.
- Each subobject can be an IPv4/IPv6 prefix or an Autonomous System number.
- CSPF computes the ERO; the PATH message carries it.

**SENDER_TSPEC (Traffic Specification)**: Describes the traffic the LSP will carry.
- Peak rate, average rate, burst size.
- In practice, often just a bandwidth reservation value.

**SESSION_ATTRIBUTE**: Carries flags and attributes.
- Setup priority (0-7, lower = higher priority)
- Holding priority (0-7)
- Local Protection Desired (flag for FRR)
- Label Recording Desired (flag for RRO label recording)
- SE-style (Shared Explicit — required for make-before-break)

**ADMIN_STATUS**: Administrative status (up, down, testing).

**FAST_REROUTE Object**: Requests FRR protection.
- Setup priority, holding priority for backup
- Hop limit, bandwidth for backup path
- Include-any, include-all, exclude-any affinity flags for backup path constraints

### 3. RESV Message and Objects

The RESV message travels from tail-end back to head-end, hop by hop, confirming the reservation and distributing labels.

**Key objects carried in RESV:**

**LABEL Object**: The MPLS label assigned by each hop.
- Each transit router assigns a label and sends it upstream in the RESV.
- The upstream router uses this label when pushing traffic onto the LSP.
- Label allocation proceeds tail-end → head-end, one hop at a time.

**FLOWSPEC**: Confirms the bandwidth reservation.

**RRO (Record Route Object)**: Records the actual path taken.
- List of (IP address, label) pairs for each hop the RESV traversed.
- Used for loop detection: if a router sees its own address in the RRO, there's a loop.
- Also used for make-before-break validation and path recording.

**STYLE**: Reservation style.
- **Fixed-Filter (FF)**: One reservation per sender. Simple.
- **Shared Explicit (SE)**: Multiple senders share a reservation. Required for make-before-break (old and new LSP share bandwidth during transition).

### 4. Soft State and Refresh

RSVP-TE is soft-state: all path and reservation state must be periodically refreshed.

**Default refresh interval**: 30 seconds. Each router sends PATH refresh downstream and RESV refresh upstream every 30 seconds.

**Cleanup timeout**: If 3 consecutive refreshes are missed (default: 3 × 30 = 90 seconds), the state is deleted.

**The refresh problem**: In a network with 10,000 LSPs, each transit router must process 10,000 PATH refreshes and 10,000 RESV refreshes every 30 seconds. That's ~667 RSVP messages per second just for keepalive. This is the primary scalability concern with RSVP-TE.

**Refresh Reduction (RFC 2961):**
- **Summary Refresh (Srefresh)**: Instead of sending full PATH/RESV messages, send a compact message listing the MESSAGE_ID of each active state. Much smaller than full refresh.
- **Reliable messaging**: MESSAGE_ID and MESSAGE_ID_ACK objects provide reliable delivery without full refresh. If a message is acknowledged, the refresh timer can be extended significantly.
- **Bundle messages**: Multiple RSVP messages combined into one IP packet to reduce per-packet overhead.

With refresh reduction, the per-LSP refresh overhead drops dramatically, allowing RSVP-TE to scale to tens of thousands of LSPs per router.

### 5. CSPF (Constrained Shortest Path First)

CSPF is not a protocol — it's an algorithm run locally at the head-end to compute the ERO.

**Input**: The TE Database (TED), populated by IGP-TE extensions (IS-IS RFC 5305 / OSPF RFC 3630). The TED contains:
- Link bandwidth (maximum, reservable, unreserved per priority level)
- TE metric (may differ from IGP metric)
- Administrative groups (32-bit bitmask, also called "affinities" or "link colors")
- SRLG (Shared Risk Link Group) membership

**Algorithm:**
1. Start with the full topology from the TED.
2. Prune links that don't satisfy constraints:
   - Remove links with insufficient unreserved bandwidth at the requested priority.
   - Remove links that don't match affinity constraints (include-any, include-all, exclude-any).
   - Remove links in excluded SRLGs.
3. Run Dijkstra's algorithm on the pruned topology using TE metrics.
4. The result is the shortest constrained path — the ERO.

**If CSPF fails** (no path satisfies all constraints), the LSP is not established and the head-end reports an error.

**CSPF and equal-cost paths**: When multiple equal-cost constrained paths exist, implementations use various tiebreakers:
- Random selection
- Least-loaded path (most unreserved bandwidth)
- Most-loaded path (bin packing)

### 6. Bandwidth Reservation and Preemption

RSVP-TE reserves bandwidth on each link along the path. This bandwidth is advertised back into the IGP TED as "reserved," reducing the "unreserved bandwidth" available for future LSPs.

**Priority levels**: RSVP-TE uses 8 priority levels (0-7, where 0 is highest). Each LSP has:
- **Setup priority**: Priority used when establishing the LSP. The LSP can preempt existing LSPs with lower holding priority.
- **Holding priority**: Priority used once established. The LSP can be preempted by new LSPs with higher setup priority.

**Preemption rule**: A new LSP with setup priority P can preempt an existing LSP with holding priority H if P < H (numerically lower = higher priority). When preempted, the existing LSP is torn down (ResvTear) and the head-end must find a new path.

**Typical SP convention:**
- Core traffic: setup 7, hold 0 (easy to set up, hard to preempt)
- Customer VPN: setup 5, hold 5 (balanced)
- Best-effort: setup 7, hold 7 (lowest priority, easily preempted)

### 7. Make-Before-Break (MBB)

MBB allows an RSVP-TE LSP to be rerouted without traffic disruption.

**Mechanism:**
1. Head-end computes a new path (triggered by re-optimization timer, bandwidth change, or admin action).
2. Head-end signals a new LSP (same SESSION, new LSP ID) along the new path using Shared Explicit (SE) style.
3. SE style allows the old and new LSP to share bandwidth on common links — preventing double-counting.
4. Once the new LSP is established (RESV received), the head-end switches traffic to the new LSP.
5. The old LSP is torn down (PathTear with old LSP ID).

**Critical property**: Traffic flows on the old path until the new path is fully established. There is no "break" before the "make." This is essential for hitless re-optimization in SP networks.

### 8. Fast Reroute (FRR) — RFC 4090

FRR provides sub-50ms local repair for RSVP-TE LSPs. When a link or node fails, the Point of Local Repair (PLR) — the router immediately upstream of the failure — switches traffic to a pre-computed backup path.

**Two FRR methods:**

#### One-to-One Backup (Detour)
- A separate backup LSP (detour) is created for each protected LSP, for each potential failure.
- The detour merges back into the primary path downstream of the failure.
- Pro: Per-LSP granularity, each LSP gets its own backup bandwidth.
- Con: N protected LSPs × M potential failures = N×M detour LSPs. Doesn't scale.

#### Facility Backup (Bypass Tunnel)
- A single bypass tunnel protects all LSPs traversing a protected link or node.
- When failure occurs, the PLR pushes traffic into the bypass tunnel (label stacking: bypass label on top of original LSP label).
- The bypass tunnel endpoint (Merge Point, MP) pops the bypass label and forwards based on the original LSP label.
- Pro: One bypass per protected element, regardless of how many LSPs use it. Scales.
- Con: Bandwidth accounting is approximate (all LSPs share the bypass bandwidth).

**FRR types:**
- **Link protection**: Bypass tunnel protects against failure of the link between PLR and next-hop. The bypass goes around the failed link to the next-hop.
- **Node protection**: Bypass tunnel protects against failure of the next-hop node entirely. The bypass goes around the next-hop to the next-next-hop. Strictly stronger than link protection.

**FRR in action:**
1. PLR detects failure (BFD, interface down).
2. PLR immediately swaps the primary LSP label stack to include the bypass tunnel label.
3. Traffic flows through the bypass tunnel to the merge point.
4. The merge point pops the bypass label and forwards normally.
5. Meanwhile, the head-end is notified (PathErr) and re-signals the primary LSP on a new path.
6. Once the new primary is up, traffic moves back and the bypass is no longer used.

Total repair time: BFD detection (50-150ms) + local label swap (microseconds) = sub-200ms.

## Protocol Interactions

**RSVP-TE ↔ IGP-TE**: RSVP-TE consumes TE information from the IGP (bandwidth, admin groups, TE metrics). When RSVP-TE reserves bandwidth, it updates the TE database, which the IGP re-floods. This closed loop (RSVP reserves → IGP updates → CSPF sees new state) is how the network self-adjusts.

**RSVP-TE ↔ LDP**: Can coexist. LDP-over-RSVP-TE tunneling allows LDP label bindings to be transported through RSVP-TE tunnels, combining TE path control with LDP's VPN label distribution simplicity.

**RSVP-TE ↔ BGP**: BGP can use RSVP-TE tunnels as next-hop resolution. When a BGP next-hop is reachable via an RSVP-TE tunnel, BGP traffic follows the engineered path. This is the primary mechanism for TE-aware VPN forwarding.

**RSVP-TE ↔ BFD**: BFD provides fast failure detection for FRR. When BFD detects a link/node failure, the PLR activates the bypass tunnel within milliseconds.

**RSVP-TE ↔ Segment Routing**: SR-TE replaces RSVP-TE. SR encodes the explicit path as a segment list in the packet header — no per-hop signaling state, no refresh overhead, no FRR bypass tunnels (TI-LFA handles protection). SR-TE is the modern successor for all new TE deployments.

## Edge Cases & Gotchas

**1. Refresh overhead at scale**: Even with RFC 2961 refresh reduction, tens of thousands of LSPs generate significant RSVP processing load. Each transit router must maintain per-LSP state. This is RSVP-TE's fundamental scalability limit and the primary motivation for SR-TE migration.

**2. CSPF finds a path but signaling fails**: CSPF computes the ERO based on a snapshot of the TED, which may be stale. Between CSPF computation and PATH message arrival at a transit node, bandwidth may have been consumed by another LSP. Result: PathErr with "Admission Control Failure." The head-end retries with a new CSPF.

**3. Preemption cascades**: Preempting one LSP may force its head-end to reroute. The rerouted LSP may preempt another LSP. That LSP reroutes and preempts yet another. In the worst case, a cascade of preemptions ripples through the network. Careful priority assignment and bandwidth planning prevent this.

**4. FRR bypass bandwidth accounting**: Facility bypass tunnels share bandwidth among all protected LSPs. During failure, actual traffic may exceed the bypass capacity. There's no per-LSP bandwidth guarantee in the bypass — it's best-effort protection.

**5. MBB and double bandwidth**: During make-before-break, the old and new LSP coexist briefly. On shared links, SE style prevents double-counting. But on non-shared links, bandwidth is temporarily consumed by both LSPs. In bandwidth-constrained networks, MBB may fail because the new path doesn't have room while the old path still holds bandwidth.
