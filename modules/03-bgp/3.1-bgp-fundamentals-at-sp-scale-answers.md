# 3.1 — BGP Fundamentals at SP Scale — Answer Key

## Question 1
> You see a BGP peer stuck in `Active` state. The peer IP is reachable via ping. What are the three most likely causes and how would you diagnose each?

### Answer

`Active` state in the BGP FSM means the router is actively trying to initiate a TCP connection to port 179 on the peer but failing. Ping works because ICMP uses a different path than TCP/179 — different protocol, different port, potentially different ACL rules.

**1. TCP port 179 is blocked by an ACL or firewall.**

The most common cause. An interface ACL, control-plane policing policy, or intermediate firewall is permitting ICMP (ping) but blocking TCP/179. On IOS-XR, the default control-plane policy may not include BGP if it was customized.

Diagnose:
```
IOS-XR: show access-lists interface GigabitEthernet0/0/0/0
        show lpts bindings brief          ← check LPTS allows BGP
        telnet <peer-ip> 179              ← test TCP/179 reachability
Junos:  show firewall log                 ← look for rejected TCP/179
        test policy <filter-name> 179
```

**2. BGP `update-source` mismatch — the source IP doesn't match what the peer expects.**

If Router A peers with Router B's loopback (10.0.0.2) but Router A's TCP connection originates from its physical interface IP (192.168.1.1), Router B receives the TCP SYN from an IP it doesn't recognize as a configured neighbor and ignores it. Ping works because you're pinging the peer's physical interface, not establishing a BGP session.

Diagnose:
```
IOS-XR: show run router bgp | include neighbor|update-source
        ← verify update-source matches what the remote expects
Junos:  show configuration protocols bgp group <name> local-address
```

**3. Incorrect or missing route to the peer's loopback (for iBGP sessions).**

iBGP sessions typically peer via loopback addresses, resolved through IGP. If the IGP route to the peer's loopback is missing (IS-IS adjacency down, OSPF area issue), the TCP SYN reaches the peer via a physical path (ping works) but the return TCP SYN-ACK can't find a route back to your loopback. Or the outbound TCP SYN uses the wrong source without `update-source`.

Diagnose:
```
IOS-XR: show route <peer-loopback>       ← is there an IGP route?
        show bgp neighbor <peer> | include State|Last error
Junos:  show route <peer-loopback>
        show bgp neighbor <peer> | match "State|Last"
```

**Additional diagnostic:** `show bgp neighbor <peer-ip>` will show the last reset reason and TCP connection attempts. Common last-error messages: "Connection refused" (peer not configured), "Transport connection timeout" (ACL/firewall), "No route to host" (routing issue).

## Question 2
> Explain why IOS-XR's explicit address-family activation model is safer than IOS-XE's default behavior (auto-activate IPv4 unicast). What goes wrong on an IOS-XE router serving as both internet PE and MPLS L3VPN PE if you forget `no bgp default ipv4-unicast`?

### Answer

**IOS-XR requires explicit activation.** When you configure a BGP neighbor on IOS-XR, it establishes the TCP session but exchanges zero routes until you explicitly enter the address-family and activate the neighbor:

```
router bgp 65000
 neighbor 10.0.0.2
  remote-as 65000
  address-family ipv4 unicast     ← must explicitly configure
  address-family vpnv4 unicast    ← must explicitly configure
```

No `address-family` stanza = no routes exchanged. This is fail-safe — you can't accidentally advertise routes in an address family you didn't intend.

**IOS-XE auto-activates IPv4 unicast.** When you configure a BGP neighbor, IOS-XE automatically activates it under `address-family ipv4 unicast` unless you explicitly disable this with `no bgp default ipv4-unicast` at the BGP process level. This default exists for historical simplicity but creates a dangerous trap in SP environments.

**What goes wrong on a dual-role internet PE + L3VPN PE:**

The router has iBGP sessions with route reflectors for two purposes:
1. **IPv4 unicast** — internet routing table (~950K prefixes)
2. **VPNv4 unicast** — MPLS L3VPN routes (potentially millions of VPN prefixes)

If you forget `no bgp default ipv4-unicast`, every iBGP neighbor configured for VPNv4 also automatically gets IPv4 unicast activated. This means:

- **The internet full table gets advertised to VPN-focused RRs** (and vice versa) — massive unnecessary route exchange
- **VPN-focused RRs that shouldn't carry internet routes** now receive and reflect them, consuming memory and CPU on devices sized for VPN scale, not internet scale
- **Route leaking risk** — with both IPv4 and VPNv4 active on the same session, misconfigured policies could leak internet routes into VRFs or VPN routes into the global table
- **Memory exhaustion** — a PE receives the full internet table on sessions that were only intended for VPN routes, potentially doubling its BGP RIB memory consumption

The fix is one line at the top of the BGP config:
```
router bgp 65000
 no bgp default ipv4-unicast
```

Then explicitly activate each neighbor only in the address families it needs. This should be in every SP BGP template. No exceptions.

## Question 3
> A customer reports that traffic to a specific destination is taking a suboptimal path through your network. You check the BGP table and see two paths — one via eBGP (AS_PATH length 3) and one via iBGP from a route reflector (AS_PATH length 1). The eBGP path is being preferred. Explain why and how to fix it.

### Answer

**Step 7 of the BGP best path algorithm: eBGP is preferred over iBGP.**

The BGP best path selection is ordered. Before the algorithm reaches Step 4 (shortest AS_PATH), it must pass through Steps 1-3. But both paths survive those steps with equal values (same weight, same LOCAL_PREF, neither is locally originated). At Step 4, the iBGP path wins (AS_PATH length 1 vs 3).

However, after Step 4, the algorithm continues. At **Step 7: prefer eBGP over iBGP.** Wait — that can't be right either, because the iBGP path has a shorter AS_PATH and should have been selected at Step 4.

**The real answer:** Something earlier in the best path algorithm is causing the eBGP path to win before AS_PATH is even compared. The most likely culprits:

**1. LOCAL_PREF difference (Step 2).** If the eBGP path was learned from a customer or peer and tagged with LOCAL_PREF 150 by import policy, while the iBGP path from the RR has LOCAL_PREF 100 (default), LOCAL_PREF wins at Step 2 — AS_PATH is never compared. Check:
```
IOS-XR: show bgp <prefix> | include LOCAL_PREF
Junos:  show route <prefix> detail | match local-pref
```

**2. Weight set on the eBGP neighbor (Step 1, Cisco-specific).** If someone configured `weight 1000` on the eBGP neighbor, it beats everything. Weight is local-only and the highest priority.

**3. If LOCAL_PREF and weight are equal,** then AS_PATH length SHOULD select the iBGP path (length 1 < 3). If eBGP is still winning, check whether `bgp bestpath as-path ignore` is configured — this disables AS_PATH length comparison entirely, and then Step 7 (eBGP over iBGP) kicks in.

**How to fix it:**

- If LOCAL_PREF is the cause: adjust the import policy on the eBGP session to set appropriate LOCAL_PREF (transit routes should get LOCAL_PREF 80, not 150)
- If weight is the cause: remove the weight configuration
- If `as-path ignore` is configured: remove it unless there's a specific design reason
- Proper SP design: LOCAL_PREF hierarchy — customers: 150, peers: 100, transit: 80. This ensures customer and peer routes are preferred over transit, and the iBGP path (via RR) with shorter AS_PATH would naturally win at Step 4 when LOCAL_PREF is equal
