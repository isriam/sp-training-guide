# 3.3 — eBGP Peering — Protocol Theory

## Protocol Overview

eBGP (External BGP) is BGP peering between routers in different autonomous systems. If BGP is the language of the Internet, eBGP is the part where strangers exchange maps while quietly trying to manipulate each other. It is the protocol used for transit, peering, customer/provider edges, IXP route servers, and inter-AS service handoff.

At a protocol level, eBGP and iBGP use the same message types, same FSM, and same path attribute framework. The differences are behavioral rules:

- eBGP prepends the local AS to AS_PATH on export.
- eBGP usually changes NEXT_HOP to self.
- eBGP defaults to TTL 1 for directly connected peers.
- eBGP advertises only the best path, with normal export policy and loop prevention based on AS_PATH.
- eBGP strips LOCAL_PREF on export and does not accept routes containing its own AS in AS_PATH.

These rules make eBGP a policy boundary. The session itself is just a TCP connection on port 179, but operationally it's a legal treaty with packet forwarding attached.

## Core Mechanisms

### 1. eBGP Session Formation

An eBGP session starts exactly like any BGP session: TCP three-way handshake, OPEN exchange, KEEPALIVE confirmation, then UPDATEs. The protocol differences start once routes are exchanged.

**Default assumptions for classic eBGP:**
- Neighbor is directly connected
- TTL = 1
- NEXT_HOP is rewritten to self on export
- AS_PATH gets local AS prepended on export
- Routes with local AS already present in AS_PATH are rejected on import

**Why TTL = 1?**
Classic eBGP assumed adjacent routers on a physical interconnect. TTL 1 ensures the peer is directly connected — a packet that traverses any intermediate hop dies before reaching the neighbor. This became a crude security mechanism before GTSM formalized it.

**Directly connected check**: Many implementations require the eBGP peer IP to be directly connected unless `ebgp-multihop` or an equivalent knob is configured. This prevents a router from accidentally peering across the core unless explicitly allowed.

### 2. AS_PATH Loop Prevention

BGP's primary loop prevention mechanism is brutally simple: when a router receives a route, it checks the AS_PATH. If its own AS is present, the route is rejected.

**Import logic:**
1. Receive UPDATE.
2. Parse AS_PATH.
3. If local AS appears anywhere in AS_SEQUENCE or AS_SET, reject the route.
4. Exception: `allowas-in` explicitly relaxes this, typically for customer multi-homing with the same provider AS appearing multiple times.

**Export logic:**
1. Select best path.
2. On eBGP export, prepend local AS to AS_PATH.
3. Send route.

**AS override**: In carrier supporting carrier or managed customer scenarios, the provider may replace the customer's AS in the AS_PATH with its own when advertising back to the customer (`as-override`). This allows a customer using the same ASN at multiple sites to receive its own routes back without tripping loop detection.

**allowas-in**: Permits routes containing the local AS in the path, up to a configurable count. Dangerous if used carelessly — it weakens the core loop prevention rule. Use only when you understand exactly why you're breaking the law of the land.

### 3. NEXT_HOP Behavior

In eBGP, the advertising router normally sets NEXT_HOP to itself. This is one of the cleanest differences from iBGP.

**Why?**
The receiving AS should forward traffic to the advertising border router, not to some third-party next-hop from the remote AS's internal topology.

**Default rule:**
- eBGP export → rewrite NEXT_HOP to local interface/session address
- iBGP export → preserve NEXT_HOP unless `next-hop-self` configured

**Third-party next-hop exception:**
On shared multi-access networks (especially IXPs), BGP may preserve a third-party next-hop if the true origin and the receiver are on the same subnet and can reach each other directly. This avoids a trombone path through the advertising router.

Example:
- Router A and Router C are on the same IXP LAN.
- Router B receives a route from A and advertises it to C.
- B may set NEXT_HOP = A instead of itself, because C can reach A directly.

This optimization is elegant until someone filters ARP, breaks the shared subnet assumptions, or forgets that IX fabrics are full of knives.

### 4. Multihop eBGP

When eBGP peers are not directly connected, `ebgp-multihop` (or vendor equivalent) increases the TTL and suppresses the directly-connected check.

**Use cases:**
- Route server peering at IXPs using loopbacks
- Inter-AS peering across a transport network
- Customer/provider peering over a tunnel
- eBGP between loopbacks for stability and fast reroute design

**Protocol consequences:**
- TTL > 1 allows packets to traverse intermediate hops
- GTSM may need different configuration (expected TTL not just 255/254 model)
- IGP/static route reachability to peer loopbacks is now required
- TCP path is longer and can be impacted by intermediate failures unrelated to the actual BGP speaker

**Operational risk:**
Multihop eBGP turns BGP from "border protocol on a wire" into "application over an underlay." Which means underlay reachability, BFD design, ACLs, and path asymmetry all matter. You traded physical simplicity for logical flexibility. That's usually fine, just don't pretend it's still a directly connected peering.

### 5. GTSM / TTL Security (RFC 5082)

GTSM (Generalized TTL Security Mechanism) protects BGP sessions from off-link spoofing by enforcing a high incoming TTL.

**Classic directly connected model:**
- Sender transmits with TTL 255
- Receiver accepts only packets with TTL >= 254
- Any remote attacker more than one hop away cannot craft a valid packet, because TTL will decrement in transit

**Why this works:**
TTL is set by the real sender's IP stack. An attacker cannot send a packet through multiple hops and still have it arrive with TTL 254 or 255.

**Benefit over TCP MD5 alone:**
- GTSM blocks spoofed traffic before TCP processing
- Reduces CPU impact from bogus packets
- Works even if TCP MD5 is absent

**Limitation:**
GTSM only helps if hop count is tightly bounded and predictable. On multihop eBGP, you can still use TTL security, but you need to set the accepted hop-count threshold appropriately. The farther away the peer is, the weaker the protection.

### 6. BGP Session Protection: TCP MD5 and TCP-AO

Since BGP rides on TCP, it is vulnerable to forged TCP resets, spoofed opens, and session hijacking attempts.

#### TCP MD5 (RFC 2385)
Each TCP segment carries an MD5 digest computed over the TCP pseudo-header, segment contents, and a shared secret.

**Properties:**
- Receiver validates signature before accepting the segment
- Invalid segments are dropped silently
- Protects against spoofed RST/ACK injection
- Requires matching secret on both peers

**Limitations:**
- MD5 is old and crusty
- No algorithm agility
- No seamless key rotation
- Operationally awkward on large peering fabrics

#### TCP-AO (RFC 5925)
The TCP Authentication Option modernizes session authentication.

**Improvements over MD5:**
- Multiple algorithms supported
- Better replay protection
- Key rollover support
- Cleaner cryptographic design

**Reality:**
TCP-AO is technically superior. TCP MD5 is still more commonly deployed. Because networking, like archaeology, is mostly layers of historical disappointment.

### 7. Graceful Restart (RFC 4724)

Graceful Restart lets a BGP speaker restart the control plane without immediately flushing routes.

#### Capability negotiation
During OPEN, each side advertises:
- Whether it can restart gracefully
- Restart time (seconds)
- Which AFI/SAFI it can preserve forwarding state for

#### Restart process
1. BGP speaker restarts.
2. Peer retains routes learned from the restarting speaker as "stale" instead of withdrawing them.
3. Restarting speaker re-establishes session.
4. Restarting speaker re-advertises routes.
5. Peer replaces stale routes with fresh ones.
6. Any stale routes not refreshed by end-of-rib / restart timer are removed.

**Key concept:**
Graceful Restart assumes forwarding continues during control-plane restart. If the restarting router lost forwarding state too, retaining stale routes is dangerous — blackholing can result.

#### Helper mode
A router can act as a **helper** for a peer's graceful restart without itself being restart-capable. This means it will preserve the peer's stale routes but doesn't promise the reverse.

#### End-of-RIB marker
The restarting speaker sends an End-of-RIB marker (an empty UPDATE or MP_UNREACH/MP_REACH completion signal, depending on AFI/SAFI) to indicate initial routing replay is complete. This tells the helper it can now safely purge any stale routes that were not refreshed.

### 8. Non-Stop Routing / Non-Stop Forwarding

These are often confused with Graceful Restart but are not the same thing.

- **Graceful Restart**: Peer-assisted. Assumes peer keeps stale routes while router restarts.
- **NSR/NSF**: Local redundancy. The router itself preserves protocol state across RE/RP switchover using checkpointing or stateful failover.

**NSR (Non-Stop Routing)**: Routing protocol sessions do not reset during control-plane failover because backup RP already has full protocol state.

**NSF (Non-Stop Forwarding)**: Forwarding continues during control-plane switchover even if the protocol session resets and peers must help. Often paired with Graceful Restart.

In SP networks, the goal is usually:
- NSR/SSO inside the box if supported
- Graceful Restart across the peering boundary
- BFD tuned carefully so it doesn't immediately tear down the session during a planned restart event

### 9. Route Servers at IXPs

A route server is a BGP speaker at an Internet Exchange Point that redistributes routes between participants without being in the forwarding path.

**Crucial distinction:**
A route server is NOT a transit router.
- It receives routes from participants.
- It re-advertises them to other participants.
- But it preserves the original NEXT_HOP, so traffic flows directly between participants across the exchange fabric.

This makes route servers behave somewhat like route reflectors for eBGP, but with important policy transparency requirements.

**RFC 7947 route server principles:**
- Do not modify NEXT_HOP unless necessary
- Usually do not prepend own AS to AS_PATH in a way that hides origin path semantics
- Apply participant-specific import/export policy
- Prevent route leaks and enforce filtering

**Why route servers exist:**
Without them, N participants at an IXP need N(N-1)/2 bilateral BGP sessions. With a route server, each participant peers once (or twice for redundancy) and gains reachability to everyone permitted by policy.

### 10. Prefix Filtering and Maximum Prefix

BGP's protocol model assumes speakers may send anything. Sanity requires explicit filtering.

**Prefix filtering**:
- Import: accept only prefixes the peer is authorized to originate or transit
- Export: advertise only customer routes, local routes, or policy-approved transit sets
- Based on IRR, RPKI, explicit prefix-lists, or generated policy from authoritative data

**Maximum prefix protection**:
A session can be configured with a maximum accepted prefix count. If the peer exceeds it:
- Log warning at threshold (e.g., 75%, 90%)
- Optionally tear down session at hard limit
- Optionally just stop accepting new routes

This protects against accidental full-table leaks, customer meltdowns, and fat-fingered apocalypse updates.

### 11. RPKI Origin Validation (RFC 6811)

RPKI adds cryptographic origin validation to BGP.

**Core objects:**
- **ROA (Route Origin Authorization)**: Cryptographically signed statement saying "AS X may originate prefix P up to maxLength Y"
- **RPKI cache**: Validator server that retrieves ROAs from the RPKI repository system and presents validated state to routers via RTR (RFC 8210)

**Validation outcomes:**
- **Valid**: Route origin AS is authorized for prefix and prefix length
- **Invalid**: Route origin AS is not authorized, or prefix length exceeds ROA maxLength
- **NotFound**: No covering ROA exists

**Important**: RPKI origin validation checks only the origin AS, not the whole AS_PATH. It does NOT validate that the path itself is legitimate. That's BGPsec territory, and basically nobody runs that at Internet scale because life is short.

## Protocol Interactions

**eBGP ↔ IGP/static routing**: Directly connected eBGP doesn't require IGP reachability to the peer, but multihop eBGP does. When peering via loopbacks or across transport infrastructure, the underlay determines whether the BGP session lives. This is why multihop eBGP often gets paired with static host routes, IGP, or BFD.

**eBGP ↔ BFD**: BFD is commonly bound to eBGP sessions for fast failure detection. Without BFD, session failure depends on Hold Timer expiry or TCP failure detection, both slow. With BFD, a dead peer can be detected in ~150ms. But BFD and Graceful Restart can conflict — if BFD kills the session instantly during a planned restart, GR gets no chance to help.

**eBGP ↔ RPKI**: RPKI validation is typically applied on import from external peers. Invalid routes may be rejected or heavily de-preferenced. NotFound routes are usually accepted but tagged. The validation result itself can be attached to policy logic.

**eBGP ↔ IX route servers**: Route server participants typically peer eBGP with the route server but forward traffic directly to each other. This means NEXT_HOP preservation is essential, and policy validation at the route server is everything.

## Edge Cases & Gotchas

**1. Graceful Restart + bad forwarding state = blackhole**: GR assumes forwarding survives control-plane restart. If the restarting router lost FIB/programming state, peers retaining stale routes just keep sending traffic into a void. This is why helper policies sometimes reject GR for peers/platforms not trusted to actually preserve forwarding.

**2. Multihop eBGP kills the security assumptions**: Once TTL is greater than 1 and the peer is no longer directly connected, simple GTSM logic weakens and the session depends on the underlay. That's fine, but pretending a multihop eBGP session is as trustworthy as a cross-connect is how you end up with surprises.

**3. Third-party NEXT_HOP at IXPs breaks quietly**: If participants don't have proper reachability, ARP/ND, or ACL allowances to the true next-hop, routes look fine in BGP but forwarding fails. The route server isn't broken — your fabric assumptions are.

**4. Max-prefix shutdown can be both life-saving and catastrophic**: It protects the network from route leaks, but if threshold values are too tight or full tables grow beyond expected limits, you'll flap healthy peers. Set warning thresholds, review growth, and don't fossilize old numbers.

**5. RPKI Invalid is not the same as evil**: Invalid can result from stale ROAs, bad maxLength, or operator mistakes. The route is cryptographically unauthorized, yes, but the fix may be "call the remote NOC," not "assume active hijack." Same outcome for policy, different outcome for your blood pressure.
