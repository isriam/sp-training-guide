## Theory

### Physical Topology Models

Every SP network is built on a physical topology — the actual fiber routes connecting sites. The choice of topology drives cost, resilience, capacity efficiency, and operational complexity. No real network uses a single pure topology; they're all hybrids. But understanding the pure models is essential for understanding the tradeoffs.

#### Full Mesh

Every node connects directly to every other node.

```
        A ─────────── B
       /|\           /|\
      / | \         / | \
     /  |  \       /  |  \
    /   |   \     /   |   \
   D────|────\───/────|────C
    \   |     \ /     |   /
     \  |      X      |  /
      \ |     / \     | /
       \|    /   \    |/
        E ─────────── F
```

**Pros:** Maximum resilience — any single (or multiple) link failure leaves all nodes reachable. Shortest path between any two nodes. Minimum hops = minimum latency.

**Cons:** Prohibitively expensive. A full mesh of N nodes requires N×(N-1)/2 fiber routes. For 10 backbone nodes, that's 45 fiber routes. For 20 nodes, 190. The cost scales quadratically and becomes absurd beyond ~6 nodes. Most of those fiber routes will carry minimal traffic.

**Where it's used:** Almost never in pure form. Small backbone cores (3–5 nodes) sometimes approximate full mesh. Submarine cable landing arrangements between a small number of coastal cities.

#### Partial Mesh

A subset of possible connections, chosen based on traffic demand, geography, and cost.

```
        A ─────────── B
        |            /|
        |           / |
        |          /  |
        |         /   |
        D────────/────C
        |             |
        |             |
        E ─────────── F
```

**Pros:** Balances cost and resilience. Links are placed where traffic justifies them. Can be incrementally expanded — add a link between E and B when demand appears.

**Cons:** Not all node pairs have direct paths — some traffic takes multiple hops. Requires careful design to ensure no single link failure partitions the network. Uneven resilience — some nodes may have 2 paths while others have 4.

**Where it's used:** Backbone/core networks. This is what most Tier 1 and Tier 2 SP backbones actually look like. The backbone is a partial mesh optimized for the top 80% of traffic demand, with the remaining 20% handled via transit through intermediate nodes.

#### Ring

Nodes connected in a closed loop.

```
        A ──── B
       /        \
      /          \
     F            C
      \          /
       \        /
        E ──── D
```

**Pros:** Simple, well-understood protection. Any single failure (link or node) leaves all surviving nodes reachable via the opposite direction. Sub-50ms protection switching with standard mechanisms (SONET/SDH APS, ROADM restoration). Fiber utilization is efficient when traffic is hub-and-spoke or adjacent-node dominant. Linear cost scaling — N nodes require N fiber routes.

**Cons:** Worst-case path length is N/2 hops (halfway around the ring). Capacity is constrained — all traffic between non-adjacent nodes must transit intermediate nodes, consuming capacity on shared spans. Dual failures on opposite sides of the ring partition the network. Doesn't scale well beyond ~8-10 nodes per ring.

**Where it's used:** Metro/regional transport. The workhorse of metro DWDM. Most metro networks start as one ring and evolve to interconnected rings as they grow. Also common in access aggregation (SONET/SDH legacy, now Ethernet rings).

#### Hub-and-Spoke (Star)

All traffic flows through a central hub.

```
        A       B       C
         \      |      /
          \     |     /
           \    |    /
            \   |   /
              HUB
            /   |   \
           /    |    \
          /     |     \
         /      |      \
        D       E       F
```

**Pros:** Cheapest topology to build — N-1 fiber routes for N nodes. Simple traffic engineering — everything goes through the hub. Easy to manage and monitor.

**Cons:** Single point of failure at the hub. All traffic takes two hops (spoke→hub→spoke), even if the endpoints are geographically adjacent. Hub becomes a capacity bottleneck. No survivability for hub failure.

**Where it's used:** Access aggregation (CO/PoP as hub, customer sites as spokes). Small regional networks. Content delivery architectures where a single data center serves all endpoints. Almost never used for backbone transport due to the hub SPOF.

#### Hybrid: What Real Networks Look Like

Real SP networks combine topologies at different tiers:

```
                    ┌─────── BACKBONE (Partial Mesh) ────────┐
                    │                                         │
              ┌─────┴─────┐                           ┌──────┴────┐
              │  Core PoP  │───────────────────────────│ Core PoP  │
              │  (City A)  │                           │ (City C)  │
              └─────┬─────┘                           └──────┬────┘
                    │                                         │
                    │ ◄── METRO (Rings) ──►                   │
                    │                                         │
              ┌─────┴─────┐                           ┌──────┴────┐
         ─────│  Metro     │─────                ─────│  Metro     │─────
        /     │  Ring 1    │     \              /     │  Ring 2    │     \
       /      └────────────┘      \            /     └────────────┘      \
    [Agg1]                      [Agg2]      [Agg3]                     [Agg4]
      │                           │           │                           │
      │ ◄── ACCESS (Hub-Spoke/Ring) ──►       │ ◄── ACCESS ──►           │
      │                           │           │                           │
   [CPE]  [CPE]  [CPE]        [CPE]       [CPE]  [CPE]              [CPE]
```

**The pattern:** Backbone = partial mesh (resilience + direct paths for major traffic flows). Metro = rings (protection + cost efficiency). Access = hub-and-spoke or small rings (cost minimization, simple aggregation).

**Why this works:** Each tier optimizes for the dominant constraint at that level. Access is cost-sensitive and low-bandwidth — hub-and-spoke is fine. Metro needs protection but has moderate traffic diversity — rings are optimal. Backbone carries the aggregate of all metro rings and needs both resilience and direct paths — partial mesh.

### Fiber Route Diversity

Topology diagrams lie. A "dual-homed" design on a whiteboard means nothing if both fiber paths share a conduit under the same highway overpass. Physical diversity is the foundation of network survivability, and it's harder to achieve than most network engineers realize.

#### Conduit-Level Diversity

True diversity requires fiber paths in **separate physical conduits** with **separate right-of-way (ROW)**. This means:

- Different trenches, different sides of the road (minimum)
- Different corridors entirely (ideal) — one route follows a railroad, another follows a highway
- No shared structures — bridges, tunnels, river crossings, building entrance facilities
- Verified by physical survey, not by looking at a map

```
     ✗ NOT DIVERSE (same conduit)          ✓ DIVERSE (separate ROW)
     
    Site A ═══════════════ Site B         Site A ─── Path 1 (Hwy 50) ──── Site B
           [conduit with                        \                         /
            both cables]                         └─ Path 2 (Rail ROW) ──┘
                                                    (20+ mile separation)
```

#### CLLI Diversity

**CLLI (Common Language Location Identifier)** codes uniquely identify physical locations in telecom networks (e.g., `DENVCO56` = Denver, Colorado, building 56). CLLI-level diversity means your primary and protect paths don't share any intermediate **buildings** or **entrance facilities**.

This matters because fires, power outages, and flooding affect entire buildings. Two "diverse" fiber routes that both enter the same building through the same entrance facility aren't diverse — they share a common point of failure at the building level.

#### The Shared Risk Link Group (SRLG) Problem

An **SRLG** is a set of links that share a common physical resource — a conduit, a bridge, a right-of-way segment, a building. If that resource fails, ALL links in the SRLG fail simultaneously.

```
    SRLG Example: Bridge Crossing

    Site A ────── fiber path 1 ──────┐
                                      ├── [Bridge over River] ── Site C
    Site B ────── fiber path 2 ──────┘
    
    IP view: Two independent links (A→C, B→C)
    Physical reality: Both die when the bridge is hit
    SRLG: {link_A-C, link_B-C} share SRLG "Bridge-42"
```

RFC 4202 (Routing Extensions in Support of Generalized MPLS) defines how SRLGs are represented and advertised in routing protocols. Each SRLG is identified by a 32-bit number. Links sharing a physical risk are assigned the same SRLG value.

**How SRLGs are used in path computation:**

| Mechanism | How It Uses SRLGs | Reference |
|-----------|-------------------|-----------|
| RSVP-TE | SRLG values advertised in IGP (OSPF-TE, IS-IS TE), CSPF excludes primary's SRLGs when computing backup | RFC 4202, RFC 4090 |
| SR-TE (PCEP) | PCE computes SRLG-disjoint path pairs using global topology view | RFC 5440, RFC 8664 |
| SR-TE (ODN) | On-demand nexthop constraints can include SRLG exclusion | draft-ietf-pce-segment-routing |
| TI-LFA | Backup path computation can be SRLG-aware (exclude links sharing SRLG with protected link) | RFC 8402, vendor extensions |

**The hard part isn't the protocol — it's the data.** SRLG assignment requires accurate physical plant records: which fibers share which conduits, which conduits cross which bridges, which buildings share entrance facilities. Many SPs have incomplete or outdated SRLG data, which means their "SRLG-aware" TE is operating on a fiction.

### Overlay/Underlay Alignment

The most dangerous failure mode in SP networks isn't a single fiber cut — it's a single fiber cut that takes down what the IP layer thinks are multiple independent paths. This is the overlay/underlay alignment problem.

#### The Mapping Problem

Your IP/MPLS topology (the overlay) is a logical construct. Router A has adjacencies to routers B, C, and D. The routing protocol treats these as independent links. But physically, the A→B and A→C wavelengths might ride the same DWDM system, over the same fiber, in the same conduit.

```
    IP/MPLS View (Overlay):          Physical View (Underlay):
    
    A ──── B                         A ═══════╗
    |      |                                   ║ Same fiber,
    |      |                         B ────────╢ same conduit
    D ──── C                                   ║
                                     C ────────╝
                                     
                                     D ════ [different conduit] ════ A
    
    IP thinks: 4 independent links    Reality: 3 links share fate
    IP thinks: A has 3 diverse paths  Reality: A has 2 (fiber + conduit)
```

#### Shared Fate Analysis

**Shared fate analysis** maps every logical link to its physical path, identifying which logical links die together when a single physical resource fails.

The process:
1. Document every IP/MPLS adjacency
2. Trace each adjacency through the optical layer: transponder → ROADM → fiber route → amplifiers → ROADM → transponder
3. Map each fiber route to physical conduit segments
4. Identify conduit segments shared by multiple logical links
5. Assign SRLGs based on shared conduit/structure/ROW segments
6. Verify that the IP topology's backup paths don't share SRLGs with primary paths

**What you typically find:** In networks that haven't done this analysis, 30-50% of "backup" paths share physical fate with their primary paths. The IP layer's fast-reroute is routing around a failure... onto a path that's also failed.

#### SRLG-Aware Traffic Engineering

Once you have accurate SRLG data, you can enforce diversity at the IP layer:

**RSVP-TE approach:** The headend router computes a primary LSP, then computes a backup LSP with a constraint: "exclude all SRLGs used by the primary." CSPF uses SRLG data advertised via OSPF-TE (RFC 4203) or IS-IS TE (RFC 5307) extensions.

**SR-TE with PCE approach (RFC 5440):** A stateful PCE maintains a global view of both the IP topology and SRLG data. When computing disjoint path pairs, the PCE can globally optimize for SRLG diversity — something a distributed headend computation can't do as effectively. This is the preferred modern approach.

**SR-TE with ODN (On-Demand Nexthop):** The headend signals intent (e.g., "low-latency, SRLG-diverse from path X"), and the PCE computes the path. The headend doesn't need to know the SRLG details.

### Backbone Capacity Planning

Getting the topology right is necessary but not sufficient. You also need to plan how much capacity goes on each link, when to upgrade, and how much headroom to maintain.

#### Redundancy Models

**N+1 redundancy:** N working resources plus 1 protect resource. If any single working resource fails, the protect resource absorbs its load. Works when failures are independent and singular. Example: 4 working wavelengths + 1 protect wavelength on a DWDM ring — any one wavelength failure is restored, but two simultaneous failures cause service impact.

**N+M redundancy:** N working resources plus M protect resources. Handles M simultaneous failures. More expensive but necessary for high-value services. Example: 4 working fiber pairs + 2 protect fiber pairs on a backbone route — survives any 2 simultaneous fiber pair failures.

**The real question is:** What's your target availability, and what failure scenarios must you survive?

| Availability | Downtime/Year | Requires |
|-------------|---------------|----------|
| 99.9% (three 9s) | 8.76 hours | N+1, single-failure protection |
| 99.99% (four 9s) | 52.6 minutes | N+M, diverse paths, fast failover |
| 99.999% (five 9s) | 5.26 minutes | Dual-diverse, multi-layer protection, geo-redundant |

#### Fill Levels and Headroom

Industry practice for backbone link utilization:

| Metric | Target | Why |
|--------|--------|-----|
| Peak utilization | 30–50% | Absorb failure reroute traffic without congestion |
| Post-failure utilization | < 80% | After worst-case single failure, rerouted traffic shouldn't congest surviving paths |
| Growth headroom | 6–18 months | Time to procure, install, and light new capacity |

**Why 30-50% and not 80%?** Because when a link fails, its traffic reroutes onto surviving links. If your links are running at 70% utilization and a failure shifts 30% more traffic onto the survivors, you're at 100% — and now you have congestion-induced packet loss on top of the original failure. The 30-50% peak target ensures that post-failure utilization stays below the congestion threshold.

**The formula:** If your network has K equal-capacity paths between two points, and you need to survive any single path failure, your maximum per-path utilization is roughly (K-1)/K × target_post_failure_util. For K=3 paths with 80% post-failure target: 2/3 × 80% ≈ 53%.

#### Growth Modeling

SP backbone traffic historically doubles every 18–24 months (though this has slowed for some networks to 24–36 months). Capacity planning must account for:

1. **Current peak utilization** — measured, not estimated
2. **Growth rate** — based on historical trends, sales pipeline, and market analysis
3. **Time to deploy** — new wavelengths: 2–4 weeks. New fiber routes: 6–24 months. New conduit: 12–36 months
4. **Failure headroom** — the post-failure utilization target

**Planning horizon rule of thumb:** If deploying new wavelengths on existing DWDM systems, plan 6–12 months ahead. If lighting new fiber pairs, plan 12–24 months ahead. If building new fiber routes, plan 24–36 months ahead. You're always planning at the speed of your slowest upgrade path.

#### Wavelength Planning

DWDM wavelength planning on backbone routes involves:

**Channel spacing:** Standard 50 GHz fixed grid gives 80–96 channels in C-band. Flex-grid (6.25 GHz granularity per ITU-T G.694.1) allows variable-width channels — critical for 400G+ wavelengths that may need 75 GHz or wider slots.

**Band allocation:**
- **C-band** (1530–1565 nm): Primary band, best amplifier performance, ~4.8 THz usable bandwidth
- **L-band** (1565–1625 nm): Extension band, ~80-90% of C-band capacity, slightly worse EDFA noise figure
- **S-band** (1460–1530 nm): Emerging, requires new amplifier technology (Thulium-doped or semiconductor), in early trials

**When to add wavelengths vs. new fiber pairs:**

| Add Wavelengths When... | Light New Fiber When... |
|------------------------|------------------------|
| DWDM system has unused channels | DWDM system is near channel exhaustion |
| Amplifiers have margin for additional channels | Amplifiers are saturated (adding channels degrades existing) |
| Incremental cost (transponder only) | Step-function cost (new DWDM line system, amplifiers, ROADMs) |
| Weeks to deploy | Months to deploy |
| Typical: first 70-80% of system life | Typical: when C-band is >80% full and L-band is already deployed or not available |

### Long-Haul Design Constraints

Backbone design isn't just about topology and capacity — physical constraints dictate what's possible.

#### Latency Budgets

Light travels at approximately 200,000 km/s in single-mode fiber (speed of light in vacuum ÷ refractive index ≈ 299,792 / 1.468). This gives a propagation delay of approximately **~4.9 µs/km** (or ~5 µs/km as the common approximation).

**One-way latency** for a fiber route is:

```
Latency = (route_distance_km × 4.9 µs/km) + equipment_delay

Equipment delay per node: ~5–50 µs (ROADM pass-through: ~5 µs, 
O-E-O regen: ~20-50 µs, router hop: ~5-15 µs)
```

**Key insight:** Fiber routes are not straight lines. A 1,000 km straight-line distance between two cities might be 1,300 km of actual fiber route (following highways and railroads). The route factor (actual distance / straight-line distance) is typically 1.2–1.5 for terrestrial routes.

| Route Example | Straight-Line | Fiber Route | One-Way Latency (fiber only) |
|--------------|---------------|-------------|------------------------------|
| NYC → Chicago | ~1,150 km | ~1,400 km | ~6.9 ms |
| NYC → LA | ~3,950 km | ~5,000 km | ~24.5 ms |
| London → Frankfurt | ~650 km | ~800 km | ~3.9 ms |
| Tokyo → Osaka | ~400 km | ~500 km | ~2.5 ms |

**Latency-sensitive applications** (financial trading, real-time communications, database replication) may drive route selection. A shorter fiber route with higher latency equipment (more regen sites) may be worse than a longer route that's all-optical.

#### The Amplifier Chain: EDFA Spacing and OSNR Cascade

Long-haul DWDM systems require amplification every 80–100 km (span length depends on fiber type and loss budget — standard G.652 fiber has ~0.2 dB/km loss at 1550 nm, so an 80 km span has ~16 dB loss before connectors and splices).

Each EDFA amplifies the signal AND adds ASE (Amplified Spontaneous Emission) noise:

```
    Launch      Span 1     EDFA 1     Span 2     EDFA 2     Span 3     Receiver
    ┌───┐     (80 km)     ┌───┐     (80 km)     ┌───┐     (80 km)     ┌───┐
    │Tx │──────────────▶│Amp│──────────────▶│Amp│──────────────▶│Rx │
    └───┘                └───┘                └───┘                └───┘
    
    OSNR:  ∞ (clean)     32 dB       28 dB       25 dB       22 dB
    
    Each amplifier adds ~3-4 dB of noise degradation.
    After 10 spans: OSNR ≈ launch OSNR - 10×log₁₀(N_amps) - NF adjustments
```

**The OSNR cascade formula** (simplified):

```
OSNR_total ≈ 58 - NF - span_loss - 10×log₁₀(N_amps) - 10×log₁₀(Δf)

Where:
  NF = amplifier noise figure (~5-6 dB for typical EDFA)
  span_loss = loss per span in dB (~16-20 dB for 80-100 km)
  N_amps = number of amplifiers in chain
  Δf = noise reference bandwidth (0.1 nm standard)
```

**Distributed Raman Amplification (DRA)** improves OSNR by 3–6 dB per span by using the transmission fiber itself as a gain medium (backward-pumped from each amplifier site). This effectively reduces the noise figure of each amplification stage, extending reach by 30–50%.

#### Regeneration vs. All-Optical Reach

When accumulated noise degrades OSNR below the receiver's threshold for the chosen modulation format, you need **regeneration (3R)** — the signal is converted to electrical, cleaned up (re-amplified, re-shaped, re-timed), and retransmitted optically.

Modern coherent DSP has dramatically extended all-optical reach:

| Generation | Typical All-Optical Reach (at max capacity) | Example |
|-----------|----------------------------------------------|---------|
| 10G NRZ (2005) | 2,000–3,000 km | Basic long-haul |
| 100G DP-QPSK (2012) | 3,000–6,000 km | Transcontinental |
| 400G DP-16QAM (2020) | 500–1,500 km | Regional/metro |
| 400G DP-QPSK @ high baud (2024) | 3,000+ km | Transcontinental (fewer λs per fiber) |
| 800G PCS-64QAM (2025) | 80–300 km | Metro/DCI only |

**Design decision:** Regeneration sites add cost (transponder pairs, power, space, management), but they reset OSNR. The tradeoff is between (a) lower modulation order for longer reach but lower per-wavelength capacity, and (b) higher modulation order for higher capacity but shorter reach requiring regen sites. The optimal choice depends on route length, available fiber pairs, and the cost of regen site infrastructure vs. additional fiber pairs.

#### Submarine Cable Considerations

Submarine cables are a distinct discipline but share core optical concepts with terrestrial long-haul:

- **Amplifier spacing:** ~50–70 km (shorter than terrestrial due to higher reliability requirements — repairs take weeks to months, so margins must be larger)
- **Power feeding:** Amplifiers are powered from shore stations via a copper conductor in the cable, with voltages up to 15 kV DC. Power budget limits total amplifier count and therefore cable length
- **Cable design:** Typically 6–16 fiber pairs per cable, each running independent DWDM systems
- **Repair:** Specialized cable ships, 2–8 week repair times depending on depth and location. This drives extreme design conservatism — submarine systems run with significantly more OSNR margin than terrestrial
- **Capacity:** Modern submarine cables (e.g., Google's Dunant, SubCom's recent builds) carry 250+ Tbps total across all fiber pairs using advanced spatial division multiplexing (SDM) concepts

**For the backbone designer:** Submarine cables matter when your SP has international routes or when you're buying capacity on submarine systems. The key constraint is that submarine cable capacity is fixed at deployment (you can't add amplifiers or change fiber types post-install), so capacity planning must account for the full 25-year cable life.

## Design Considerations

### Building a Resilient Backbone: A Design Checklist

1. **Start with traffic matrices** — know where traffic flows before drawing topology
2. **Map physical constraints** — available fiber routes, conduit sharing, bridge/tunnel crossings
3. **Assign SRLGs** — every shared physical resource gets an SRLG ID
4. **Design IP topology with SRLG awareness** — no primary/backup pair should share SRLGs
5. **Set fill level targets** — 30-50% peak utilization with post-failure < 80%
6. **Plan wavelength growth** — know when C-band exhaustion will force L-band or new fiber
7. **Verify diversity physically** — GIS survey, not just vendor documentation
8. **Model failure scenarios** — what happens when you cut each conduit segment?
9. **Document everything** — SRLG databases rot fast; assign ownership for maintenance

### Common Design Mistakes

| Mistake | Consequence | Prevention |
|---------|-------------|------------|
| Trusting vendor "diversity" claims | Shared fate discovered during outage | Physical survey, GIS validation |
| Running links at 70%+ utilization | Post-failure congestion cascade | Enforce 50% peak fill targets |
| IP topology designed without physical awareness | Backup paths share fiber with primary | Shared fate analysis, SRLG-aware TE |
| No SRLG database maintenance | Stale data → false sense of diversity | Assign ownership, annual audit |
| Uniform capacity on all backbone links | Over-investment on low-traffic routes | Traffic matrix-driven capacity allocation |
| Planning for current demand only | Capacity crisis in 18 months | Growth modeling with 24-month horizon |
