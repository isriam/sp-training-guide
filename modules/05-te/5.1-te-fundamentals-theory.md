# 5.1 — TE Fundamentals — Protocol Theory

## Protocol Overview

Traffic Engineering is the discipline of steering traffic away from the IGP shortest path to achieve objectives that shortest-path routing cannot: bandwidth optimization, path diversity, latency minimization, or policy compliance. TE doesn't replace the IGP — it overrides specific forwarding decisions where the IGP's greedy shortest-path algorithm produces suboptimal results.

The core problem TE solves: IGP shortest-path routing concentrates traffic on the mathematically shortest paths, potentially congesting those links while leaving alternative paths underutilized. A network with 10 Gbps of capacity between two cities might have 8 Gbps on the shortest path and 2 Gbps on an alternate path, with the shortest path at 90% utilization while the alternate sits at 10%. TE redistributes traffic to balance load.

TE in SP networks operates through three mechanisms working together:
1. **Information distribution**: The IGP floods TE attributes (bandwidth, metrics, admin groups) via IS-IS TLV 22 sub-TLVs or OSPF Type 10 Opaque LSAs, building a Traffic Engineering Database (TED).
2. **Path computation**: CSPF (Constrained Shortest Path First) runs at the head-end, computing a path through the TED that satisfies all constraints.
3. **Path signaling**: RSVP-TE (or SR-TE with segment lists) establishes the computed path, reserving bandwidth and installing forwarding state at each hop.

## Core Mechanisms

### 1. The Traffic Engineering Database (TED)

The TED is a topology database enriched with TE attributes. Every TE-capable router maintains a TED, populated by IGP-TE extensions.

**TED contents per link:**

| Attribute | Source | Purpose |
|-----------|--------|---------|
| Maximum bandwidth | IGP-TE | Physical link capacity |
| Maximum reservable bandwidth | IGP-TE | Max bandwidth available for TE reservations |
| Unreserved bandwidth (8 priorities) | IGP-TE | Available bandwidth per priority level (0-7) |
| TE metric | IGP-TE | Cost for CSPF computation (may differ from IGP metric) |
| Administrative group (affinity) | IGP-TE | 32-bit bitmask for link coloring |
| SRLG | IGP-TE | Shared Risk Link Group membership |
| Link-local/remote identifiers | IGP-TE | Unnumbered link identification |

**TED vs LSDB**: The TED is derived from the LSDB but is a separate database optimized for CSPF. The LSDB contains all LSPs/LSAs; the TED extracts only the TE-relevant information. Some implementations maintain a unified database; others keep them separate.

**TED freshness**: The TED is only as current as the last IGP-TE update. When RSVP-TE reserves bandwidth on a link, the router updates the unreserved bandwidth and re-floods via IGP. But IGP flooding has inherent delay (LSP/LSA generation throttle + propagation). During this window, the TED at remote routers is stale. This is the fundamental race condition in TE: CSPF computes paths based on information that may be milliseconds to seconds old.

**Bandwidth accounting**: Unreserved bandwidth is tracked per priority level. When an LSP with setup priority P reserves bandwidth B on a link, the unreserved bandwidth at priority P and all lower priorities (higher numbers) decreases by B. Higher priorities (lower numbers) are unaffected — they can still preempt.

### 2. IGP-TE Extensions

#### IS-IS TE Extensions (RFC 5305)

TE attributes are carried as sub-TLVs under TLV 22 (Extended IS Reachability):

| Sub-TLV | Type | Description |
|----------|------|-------------|
| Admin Group | 3 | 32-bit bitmask of link colors |
| Link Local/Remote ID | 4 | Unnumbered link identifiers |
| IPv4 Interface Address | 6 | Local interface IP |
| IPv4 Neighbor Address | 8 | Remote interface IP |
| Maximum Link Bandwidth | 9 | IEEE float, bytes/sec |
| Maximum Reservable Bandwidth | 10 | IEEE float, bytes/sec |
| Unreserved Bandwidth | 11 | 8 IEEE floats (one per priority), bytes/sec |
| TE Default Metric | 18 | 24-bit TE cost |

**TLV 134 (TE Router ID)**: Advertises the router's TE Router ID (typically the loopback). Used as the CSPF node identifier.

**Bandwidth encoding**: IEEE 754 single-precision floating point, in bytes per second. A 10 Gbps link is encoded as 1,250,000,000 bytes/sec (1.25 × 10^9). The floating-point format provides enough precision for all practical bandwidth values.

#### OSPF TE Extensions (RFC 3630)

TE attributes are carried in Type 10 Opaque LSAs (area-scoped):

**Opaque Type 1 (TE LSA)** contains top-level TLVs:
- **TLV 1 (Router Address)**: TE Router ID
- **TLV 2 (Link)**: Contains sub-TLVs for each TE-enabled link

**Link sub-TLVs** mirror IS-IS sub-TLVs: link type, link ID, local/remote interface IP, TE metric, max bandwidth, max reservable bandwidth, unreserved bandwidth per priority, admin group.

The information is identical to IS-IS TE — just different encoding and flooding mechanism.

### 3. CSPF Algorithm

CSPF is Dijkstra's algorithm with a constraint-pruning preprocessing step.

**Algorithm steps:**

**Step 1 — Build constrained topology:**
Start with the full TED. Remove links that violate constraints:
- Links with unreserved bandwidth (at the requested priority) less than the requested bandwidth
- Links whose admin group doesn't match the affinity constraints
- Links in excluded SRLGs
- Links to/from excluded nodes (if node exclusion is requested)

**Step 2 — Run Dijkstra:**
On the pruned topology, compute the shortest path from head-end to tail-end using TE metrics.

**Step 3 — Generate ERO:**
Convert the computed path into an Explicit Route Object — an ordered list of strict or loose hops.

**Affinity matching** uses three optional bitmask constraints:
- **Include-any**: At least one bit in the link's admin group must match. (Link has any of these colors.)
- **Include-all**: All bits in the constraint must be present in the link's admin group. (Link has all of these colors.)
- **Exclude-any**: No bit in the constraint may be present in the link's admin group. (Link has none of these colors.)

Example: A link has admin group `0x00000006` (bits 1 and 2 set).
- Include-any `0x00000002` (bit 1): match (bit 1 present)
- Include-all `0x00000006` (bits 1,2): match (both present)
- Exclude-any `0x00000004` (bit 2): no match — link excluded (bit 2 present)

**CSPF failure**: If no path satisfies all constraints, CSPF fails and the LSP is not established. The head-end may retry periodically or alert the operator. Common causes: insufficient bandwidth, overly restrictive affinities, or network partition in the constrained topology.

**Tie-breaking**: When multiple equal-cost paths exist after CSPF:
- **Random**: Pick one randomly (good for distributing LSPs across paths)
- **Least-fill**: Pick the path with the most unreserved bandwidth (spread load)
- **Most-fill**: Pick the path with the least unreserved bandwidth (bin pack, fewer paths used)

### 4. TE Metric vs IGP Metric

CSPF can use two different metrics:

**IGP metric**: The normal link cost used by SPF. If CSPF uses IGP metrics, TE paths follow shortest-path routing but with bandwidth/affinity constraints added.

**TE metric**: An independent metric assigned to each link specifically for CSPF. Allows TE path computation to optimize for different criteria than IGP.

**Use cases for separate TE metrics:**
- **Latency-based TE**: Set TE metric to measured one-way delay. CSPF finds the lowest-latency path. IGP metric stays bandwidth-based for normal IP routing.
- **Cost-based TE**: Set TE metric to financial cost per Mbps. CSPF optimizes for cost while IGP optimizes for performance.
- **Administrative steering**: Set high TE metric on links you want TE to avoid, without affecting IGP routing.

**Default**: If no TE metric is configured, CSPF uses the IGP metric. Most SP deployments start with IGP metric for CSPF and only introduce separate TE metrics when they have a specific optimization need.

### 5. Administrative Groups (Link Coloring)

Admin groups are a 32-bit bitmask assigned to each TE-enabled link. They provide a flexible mechanism for classifying links and constraining CSPF.

**Common coloring schemes:**

| Bit | Color | Meaning |
|-----|-------|---------|
| 0 | Red | Long-haul fiber (high latency) |
| 1 | Blue | Metro fiber (low latency) |
| 2 | Green | Protected path (SONET/SDH protection) |
| 3 | Yellow | Diverse fiber route |
| 4 | Orange | Customer-facing links |
| 5 | Purple | Inter-AS links |

A link can have multiple bits set (e.g., `0x00000003` = bits 0 and 1 = long-haul AND metro — perhaps a metro ring on long-haul fiber).

**Operational pattern**: Define a color taxonomy once, assign colors to links, then reference colors in LSP constraints. When new LSPs are created, they express intent ("use only low-latency metro links") rather than enumerating specific paths. If the topology changes, CSPF automatically finds a new path that matches the colors.

**Extended admin groups (RFC 7308)**: The original 32-bit limit was sometimes insufficient. RFC 7308 extends admin groups to arbitrary length, carried in new sub-TLVs.

### 6. Bandwidth Constraints and DS-TE

**Basic bandwidth model**: Each link has maximum reservable bandwidth, and RSVP-TE reservations consume from this pool. Unreserved bandwidth per priority is tracked and flooded.

**DS-TE (DiffServ-aware TE, RFC 4124)**: Extends the bandwidth model to support multiple traffic classes, each with its own bandwidth constraints.

**Two constraint models:**

**MAM (Maximum Allocation Model, RFC 4125):**
- Each Class Type (CT) gets a fixed maximum bandwidth allocation.
- CT0 can use up to BC0, CT1 up to BC1, etc.
- Bandwidth not used by one CT cannot be borrowed by another.
- Simple, predictable, but potentially wasteful.

**RDM (Russian Dolls Model, RFC 4127):**
- Bandwidth constraints are nested. BC0 encompasses all classes. BC1 encompasses CT1 and higher. And so on.
- Lower-priority classes can use bandwidth not consumed by higher-priority classes.
- More efficient utilization but more complex to reason about.

**DS-TE use case**: An SP wants to guarantee bandwidth for VoIP traffic (CT1) while allowing best-effort data (CT0) to use remaining capacity. With MAM, VoIP gets a fixed slice. With RDM, VoIP gets its guaranteed slice plus whatever data traffic isn't using.

### 7. Flooding Thresholds and Dampening

Every RSVP-TE reservation changes the unreserved bandwidth on a link. Flooding every change would overwhelm the IGP.

**Flooding threshold**: Only re-flood TE attributes when the bandwidth change exceeds a threshold (e.g., 10% of link capacity or an absolute value like 100 Mbps). Small reservations don't trigger re-flooding.

**Dampening**: Apply exponential backoff to TE re-flooding, similar to SPF throttling. Prevents flood storms during rapid reservation changes.

**Trade-off**: Aggressive flooding keeps the TED fresh but loads the IGP. Conservative flooding reduces IGP load but increases CSPF failures due to stale data. Most SPs use percentage-based thresholds (e.g., flood when unreserved bandwidth crosses 10%, 20%, 30%... boundaries).

## Protocol Interactions

**TE ↔ IGP**: The IGP distributes TE attributes and the TED is derived from the LSDB. TE bandwidth changes trigger IGP re-flooding. This bidirectional coupling means IGP stability directly affects TE stability.

**TE ↔ RSVP-TE**: CSPF computes the path; RSVP-TE signals it. If RSVP-TE signaling fails (admission control, preemption), CSPF must recompute.

**TE ↔ BGP**: BGP can resolve next-hops through TE tunnels. When a BGP next-hop is reachable via a TE tunnel, BGP traffic follows the engineered path. This is how VPN traffic gets traffic-engineered — BGP doesn't know about TE; it just resolves the next-hop.

**TE ↔ SR**: SR-TE replaces RSVP-TE signaling with segment lists. CSPF concepts remain the same, but path instantiation changes from per-hop signaled state to source-routed segment stacks.

## Edge Cases & Gotchas

**1. Stale TED and CSPF failure loop**: CSPF succeeds based on stale TED data, but RSVP-TE signaling fails (bandwidth already consumed). CSPF retries with same stale data, fails again. This continues until the IGP re-floods fresh TE data. Solution: implement random backoff on CSPF retry and ensure flooding thresholds aren't too conservative.

**2. Admin group bit exhaustion**: 32 bits seems like a lot until you start encoding region + type + protection + customer class. Extended admin groups (RFC 7308) exist but not all platforms support them. Plan your color taxonomy carefully.

**3. TE metric of 0**: Some implementations treat TE metric 0 as "use IGP metric." Others treat it as actual cost 0. This vendor inconsistency can cause unexpected CSPF behavior in multi-vendor networks.

**4. Bandwidth overbooking**: Maximum reservable bandwidth can be set higher than maximum bandwidth (deliberate overbooking). This is valid and common for best-effort TE. But if all overbooked LSPs actually push traffic simultaneously, congestion results. Overbooking requires traffic profiling and monitoring.

**5. IGP-TE flooding storms**: In a large network with many RSVP-TE LSPs, bandwidth changes cascade — one reservation changes unreserved bandwidth, triggers flooding, which may cause other head-ends to reoptimize, generating more reservations, more flooding. Proper flooding thresholds and dampening prevent this.
